# UCA 任务执行规范

本目录把 phase 方案转成可以直接按 Git 执行的任务包。目标不是“列一个待办清单”，而是让每个任务都具备以下信息：

- 做什么
- 改哪些范围
- 怎么验证
- 完成后要回写哪些信息
- 如何把上下文交给下一个任务

## 1. 标准执行流程

1. 从 [TASK_INDEX.md](TASK_INDEX.md) 选择当前唯一进行中的任务。
2. 新建任务分支：
   - 分支名格式：`task/<task-id>-<short-name>`
3. 在该分支完成实现、文档和测试。
4. 运行任务文件中的验证步骤；如果实际验证命令与任务文件不同，先更新任务文件再执行。
5. 回写该任务文件的“执行记录”：
   - 状态
   - 分支名
   - 实际新增内容
   - 验证结果
   - 已知遗留问题
   - 给下一个任务的交接内容
6. 提交 commit。
7. 发起 PR 或代码评审。
8. 验证通过后再合并到 `main`。
9. 合并完成后，更新 [TASK_INDEX.md](TASK_INDEX.md) 的状态和下一任务入口。

## 2. 强约束

- 同一时刻只允许一个任务处于 `in_progress`。
- 任务未更新“执行记录”前，不允许标记为 `done`。
- 任务未写清“交接给下一个任务”的内容，不允许合并到主干。
- 任何偏离原任务范围的新增工作，都必须先写回任务文件的“本次新增内容”。
- 验证失败时不能强行合并；要么补修复，要么把状态改为 `blocked` 并写明阻塞原因。

## 3. Git 工作流

当前目录下**尚未初始化 Git 仓库**，因此真正的 Git 流程从 [UCA-000](00_repo_bootstrap.md) 开始。

标准流程：

```bash
git checkout -b task/<task-id>-<short-name>
# 开发
# 验证
git add .
git commit -m "<task-id>: <summary>"
git push -u origin task/<task-id>-<short-name>
```

合并规则：

1. 先过本任务的验证项
2. 再更新任务文件
3. 再做 review / PR
4. 最后合并到 `main`

## 4. 完成后必须回写哪些字段

每个任务文件底部的“执行记录”至少要更新：

- `状态`
- `执行分支`
- `开始日期`
- `完成日期`
- `实际新增内容`
- `验证结果`
- `遗留问题`
- `交接给下一个任务`
