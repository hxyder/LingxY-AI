## Part B — 本轮迭代执行记录

### 2026-04-20（深夜）—— P0-0 战略转向：single-brain agent，不再前置 regex 分类

**触发反思**：用户质疑当前 trigger regex 都是中文/英文绑定的——"5 分钟后" vs "in 5 minutes" vs "etwa 5 Minuten später"——永远写不完，且每加一种语言都是新 regex。这是 Week 1 老毛病的变种。

**架构对齐主流开源**（LangGraph / CrewAI / AutoGPT 的共同模式）：一个"大脑"（LLM）看到 user command + 完整 tool belt + 资源清单 + workflow 目录，自主决策。没有上游的 regex 分类器抢理解权。

**本轮实际改动**：

- 🟢 **agent-loop 系统提示重写** ([agent-loop.mjs](src/service/executors/tool_using/agent-loop.mjs))：
  - 新增 `formatResourceContext(task)` — 给 LLM 一个"你现在手里有什么"清单：当前时间、附件路径、选中文本、已连接账户（含 capability）
  - Decision rules 从 13 条刚性条款改写成"Guidance (not a rigid checklist)"原则性段落，用自然语言描述场景而非硬编码 if-else
  - 明确教 LLM："future-time 请求 → 用 `create_scheduled_task`，别现在执行"、"quantifier → 先 list_files/glob_files 再逐个调用"、"ambiguous → return `{final: clarifying question}`"
  - 删除旧 rule #13（"time offset 已被上游 strip"）—— 不再欺骗 LLM，让它自己看命令做决定
- 🟢 **plan-executor 保持 Week 1-revised 最窄形态**（只处理 trigger=time_phrase + LLM=schedule 这一条路径），承认它只是**窄优化**而非通用方案
- 🟢 **没有**再加 `hasQuantifier` / `hasAmbiguousReference` 之类的前置 regex。Week 2 原计划里的这两项直接由 enriched agent-loop 的 guidance 覆盖

**为什么这是正解**：
- 新的用户表达（新语言、新措辞、新语气）不需要改 regex，LLM 天然泛化
- 当 LLM 看到 "打开桌面所有图片文件"，它会先 `list_files` 再批量 `open_file`，因为提示明确引导
- 当 LLM 看到 "明天下午1点在日历里新建任务"，它会用 `account_create_event` 而非 `create_scheduled_task`，因为资源块告诉它已连 Google + 有 calendar 能力，guidance 告诉它"时间作为参数 vs 作为调度" 自己判断
- 当用户说 "把这个文件发给 X" 但没附件，LLM 会看到资源块 `Attached files: (none)` 然后返回澄清

**保留的 regex**（按 archive/README 的判据合法）：
- `nl_to_cron.mjs` — deterministic 时间解析（"5 分钟后" → ISO）
- `fast-path-router.mjs` — tier-0 速度路径（open URL / launch app）
- `REFUSAL_PATTERNS` / truthfulness guard — 安全网
- `trigger.mjs hasTimePhrase` — 仅做是/否探测，决定是否让 plan-executor 跑一次 LLM；不分类

**待观察**：小模型（DeepSeek / 本地）是否稳定遵从 guidance。如果 LLM 在"future-time 该 schedule 还是 immediate"上反复出错，可以考虑把 plan-executor 的 understanding LLM 作为强化版（但这一层仍靠 LLM 决策，不是新的 regex 补丁）。

共同症状：框架**假设一条文本 = 一个意图 = 一个执行器**，用关键字绕过理解。

### 设计

```
用户输入 → 触发器 → [理解器 LLM, 可选] → TaskPlan → 执行器
             ↑
         纯代码，0 LLM
```

#### 触发器 (`src/service/core/intent/trigger.mjs`)

纯代码判断是否需要理解。触发条件（任一命中即进入理解器）：

1. **时间偏移**：正则匹配 "(N 分钟/小时/天|明天|今晚|之后|after|later|in N)" 且不是循环 ("每天/每周/every")
2. **数量词**：`/(所有|全部|every|each|all)\s*(的|files?|items?)/`
3. **多动词连接**：`/(然后|接着|之后|and\s+then|再)/`
4. **指代歧义**：`/(这个|那个|this|that)\s*(文件|图片|邮件|file|image|email)/` 且 context_packet 无对应资源
5. **显式问句**：命令以 `？|?` 结尾且非单问句
6. **命令中提到 "发送/上传/创建" + 目标域，但缺关键参数**（收件人、主题、路径）

不触发则跳过理解器，走当前 fast / context / agentic 路径（保留现有速度）。

#### 理解器 (`src/service/core/intent/understand.mjs`)

一次 LLM 调用，system prompt 简短，产出严格 JSON：

```json
{
  "summary": "用户想 5 分钟后收到一份美股简报邮件",
  "steps": [
    {
      "id": "s1",
      "intent": "send_email",
      "when": "at:+300s",                       // now | at:<iso> | at:+<seconds>
      "executor_hint": "tool_using",
      "inputs": {
        "to": ["user_inferred_from_context"],
        "subject": "",
        "body": "",
        "needs_data": ["us_stock_latest_news"]  // 决定了 exec 时先 search
      },
      "depends_on": []
    }
  ],
  "clarifications_needed": [
    {
      "question": "收件人邮箱是？",
      "applies_to_step": "s1",
      "inputs_field": "to",
      "blocking": true                          // true = 不回答不能执行
    }
  ],
  "confidence": "high"                          // high / low，low 时走原路径
}
```

#### 执行器 (`src/service/core/intent/plan-executor.mjs`)

- 有 blocking clarification → 用当前聊天卡 UI 问用户，等回答 → re-invoke 理解器（带用户回答）
- 每个 step 按 `depends_on` 拓扑排序
- `step.when === "now"` → 立即按 `executor_hint` 调度
- `step.when === "at:<iso>"` → 调 `create_scheduled_task` 建 one-shot（直接走后端，不让 LLM 再决策）
- `step.inputs.needs_data` → 在执行器里先运行对应 data-fetch，再填入

#### 非目标（避免重蹈 Task Runtime 覆辙）

- ❌ 不对每条文本都跑 plan（触发器把关）
- ❌ plan 不包含 verifier 循环（各执行器自己的成功判断复用）
- ❌ plan 不再加 "normalize" 和 "classify" 两个独立 LLM 调用
- ❌ plan 失败时 **降级**而不是阻塞：直接把原文本交给当前默认路径

#### 落地顺序与当前进度

原计划四步走（Week 2 加 quantifier/ambiguity 触发器 → Week 3 加澄清 UI → Week 4 撤补丁）被用户一针见血否了——"都是语言绑定的 regex，永远写不完"。改走 single-brain 路线：

1. **Stage 1 🟢 已完成**（2026-04-20 深夜）：agent-loop 拿到完整资源清单（时间、附件、账户、workflow 目录）+ 原则性 guidance，LLM 自主决策。Plan-executor 仅保留 time_phrase 窄优化（trigger 只做"有没有时间短语"探测，LLM 决定是 schedule 还是 immediate 还是 clarify）。
2. **Stage 2 🔴 清理 legacy regex** (P0-0d)：每次撤一个老分类器都先写回归证明 LLM 靠 resourceHint + guidance 能接管，再搬到 [archive/](src/service/core/intent/archive/) 并记一条日志。
3. **Stage 3 🟡 澄清 UI 回流** (P0-0c)：LLM 反问时，气泡里给个 mini-form 收集字段，回答 merge 进原命令重新 submit，减少对话轮次。
4. **Stage 4 🔴 观察 + 调优**：跑一段时间，看 DeepSeek/本地小模型是否稳定遵从 guidance。如果 future-time 场景反复出错，再考虑启用 understand.mjs 作为强化层（仍是 LLM，不是 regex）。
