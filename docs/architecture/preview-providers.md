# Preview providers (UCA-182)

This document explains the file-preview architecture introduced in
Phase 1 and refined through Phase 13. It is targeted at the next
person who needs to add a format, debug a failing render, or inspect
the metrics panel.

## 10-second tour

1. `/file/render-preview-html?path=…` hits a single dispatch handler in
   [src/service/core/http-server.mjs](../../src/service/core/http-server.mjs).
2. The handler calls `runtime.previewRegistry.render(filePath)`
   ([src/service/preview/registry.mjs](../../src/service/preview/registry.mjs)).
3. The registry resolves one provider (highest priority whose
   `canHandle(ctx)` returns truthy), hits the content-addressed cache
   on disk + LRU in memory, and if it misses calls `provider.render(ctx)`.
4. Provider returns `{ kind, html | pdfPath, cacheable, meta }`. The
   HTTP handler translates that into a 200 / 302 / 404 response.
5. Renderer side, [live-preview.js](../../src/desktop/renderer/live-preview.js)
   and [preview/client-registry.js](../../src/desktop/renderer/preview/client-registry.js)
   dispatch client-side handlers (pdf, text, csv, image, iframe-remote)
   that paint into the live-preview panel.

## Provider contract

Each provider is a plain object exporting the shape described in
[src/service/preview/provider.mjs](../../src/service/preview/provider.mjs):

```js
{
  id:            string,   // unique; used in cache key + metrics
  extensions:    string[], // lowercase with leading dot
  mimePrefixes?: string[], // optional; checked when ext is empty
  priority:      number,   // highest wins; sidecar=100, format=10, fallback=1
  version:       string,   // bump to invalidate cache of older output
  canHandle?(ctx) → boolean | Promise<boolean>,
  render(ctx)    → Promise<{ kind, html?, pdfPath?, cacheable, etag?, meta? }>
}
```

`kind` is one of:

| kind            | response                                         |
|-----------------|--------------------------------------------------|
| `"html"`        | served as `text/html; charset=utf-8`             |
| `"pdf-redirect"`| 302 to `/file/pdf?path=…`, which streams bytes   |
| `"native-open"` | 404 — client shows a "use external app" placeholder |

## Built-in providers (priority order)

| id                 | priority | extensions                                                      | notes |
|--------------------|----------|-----------------------------------------------------------------|-------|
| sidecar            | 100      | docx/xlsx/pptx/pdf/html (if `<name>-preview.html` exists)       | generated by `generate_document` |
| pdf                | 10       | .pdf                                                            | returns `pdf-redirect`; renderer uses pdfjs |
| markdown           | 10       | .md / .markdown / .mdown / .mkd                                 | marked + small regex sanitizer |
| docx               | 10       | .docx                                                           | mammoth + Word-like CSS (A4 paper) |
| xlsx               | 10       | .xlsx                                                           | exceljs with cell colors / widths / freeze panes |
| pptx               | 10       | .pptx                                                           | two tier: soffice → pdf, else coordinate layout |
| csv                | 10       | .csv / .tsv                                                     | RFC-4180 minimal parser → `<table>` |
| image              | 10       | .png / .jpg / .jpeg / .gif / .webp / .bmp / .svg                | inline data URL (≤8 MB) |
| html-passthrough   | 10       | .html / .htm                                                    | verbatim for full docs, wrapped for fragments |
| text               | 10       | .txt / .log / .json / source code extensions                    | styled `<pre>`, ~512 KB cap |

Add a new format by dropping a sibling in
[src/service/preview/providers/](../../src/service/preview/providers/)
that matches the contract, then importing it in
[providers/index.mjs](../../src/service/preview/providers/index.mjs).
No other files need to change.

## Cache

Providers that return `cacheable: true` go through
[src/service/preview/cache.mjs](../../src/service/preview/cache.mjs):

- **Key:** `sha256(filePath | mtimeMs | providerId | providerVersion)`
- **Disk:** `<runtime.paths.previewCacheDir>/<key>.html`
- **LRU:** 32 most-recent entries kept hot in memory
- **Invalidation:** file edited on disk (mtime change) OR provider
  bumps `version` field → automatic miss

The registry clears nothing on its own; manual clear via
`POST /preview/cache/clear` or the Settings → 预览 panel button.

## Concurrency + metrics

Registry wraps each render in a global `p-limit`-style queue with
concurrency 4. Metrics snapshot (available at `/preview/status`):

```
{
  renders: total calls,
  cacheHits: served from LRU or disk,
  byProvider: { [id]: { hits, cacheHits, renderMs, errors } }
}
```

Used by [Settings → 预览](../../src/desktop/renderer/console.html#L1749)
to show hit rate + per-provider latency.

## pptx: LibreOffice vs coordinate layout

pptx has two rendering tiers, picked at render-time:

1. **Tier 1 (soffice):** if `runtime.capabilities.libreoffice.present`
   is true, shell out to `soffice --headless --convert-to pdf` and
   return a `pdf-redirect`. Pixel-perfect output. Requires manual
   LibreOffice install (~700 MB) — `/preview/libreoffice/install`
   auto-invokes winget when available.
2. **Tier 2 (coordinate layout, default):** jszip + linkedom read the
   OOXML `<p:sp>` / `<p:pic>` elements with their `<a:off>` + `<a:ext>`
   EMU coordinates, convert to px (÷ 9525 at 96dpi), and emit
   absolutely-positioned `<div>`s inside a fixed-size slide frame.
   Font sizes from `<a:rPr sz>`, colors from `<a:solidFill>`, bullets
   from `<a:buChar>`. Close enough to the original layout that users
   rarely need Tier 1.

Both tiers honour `<p:sldSz>` so 16:9 and 4:3 decks render at their
correct aspect.

## Streaming-phase previews

During a `generate_document` / `write_file` call the UI shows the
file before it is done. See
[preview/streaming.js](../../src/desktop/renderer/preview/streaming.js):

- **write_file with text/md/json/code:** the `content` field is
  extracted from partial JSON and painted into a `<pre>` (or marked
  for `.md`) with 150ms debounce.
- **generate_document (binary output):** only the `title` and
  `sections[].heading / slides[].heading / sheets[].name` strings are
  pulled. They are laid out as a markdown outline with a banner
  "大纲预览 · 正在生成". No simulated layout — the layout only
  exists once the real renderer has run.

## Known limitations

- **docx:** mammoth ignores page headers/footers, columns, precise
  page breaks. Acceptable for preview; user can external-open.
- **xlsx:** no charts, no conditional formatting, no formula
  recalculation. 5000-row cap per sheet.
- **pptx Tier 2:** no animations, no transitions, no slide-master
  background fills, no line/connector shapes. Font sizes and colors
  from the inline run; no theme colour resolution.
- **pdfjs worker:** loaded from `node_modules/pdfjs-dist/legacy/build`
  via file:// URL; packaged builds still work because the electron-
  main IPC resolves `app.getAppPath()`.
