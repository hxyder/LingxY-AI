# 系统级智能悬浮执行器（Universal Context Agent）详细方案与路线图

> **【Claude 总体评注】** 本方案在产品定位、UX 思路、任务分级、状态闭环上写得非常完整，方向清晰，是一份"产品视角"的高质量方案。但作为"工程方案"，还缺三类东西：
> 1. **系统层落地细节**：Windows UIAutomation/Accessibility 抓取选区的真实可行性、跨进程通信、Native Messaging Host、Office Add-in 类型选择(VSTO vs JS Add-in)、PDF Reader 适配现实约束等没有具体讨论
> 2. **非功能性需求**：安全/隐私、数据脱敏、Token 成本/配额、离线兜底、自动更新、可观测性、性能 SLO、测试策略、多显示器/HiDPI、无障碍均缺失
> 3. **进程与数据模型**：Electron 主/渲染进程划分、本地服务到底是子进程还是独立服务、IPC 协议、本地存储选型(SQLite?)、崩溃恢复都没有定义
>
> 详见文末"22. 可行性分析与改进建议"。本文中绿色 `【Claude 注释】` 块均为后补的逐节点评。

## 1. 项目定位

### 1.1 一句话定义
一个运行在桌面系统之上的 **上下文感知型智能执行器**：当用户在文件、网页、Office、PDF、图片、链接、文本选区、剪贴板或当前窗口中产生明确上下文时，系统以轻量、快速、低打扰的方式提供操作入口；用户输入指令后，系统自动理解意图、抽取上下文、路由动作、执行任务，并在主控制台中统一呈现任务状态、执行过程、结果产物、失败原因与后续建议。

### 1.2 核心目标
1. **快**：识别快、弹出快、执行快、反馈快。
2. **灵活**：不能只有固定模板动作，必须支持自然语言命令与组合操作。
3. **不死板**：不能用户一选中文件就强行走固定流程，也不能一律变成大任务。
4. **低打扰**：存在感强，但不烦人，不乱弹，不遮挡，不抢焦点。
5. **可追踪**：所有命令、任务、执行状态、结果位置都可追踪。
6. **可扩展**：未来可接不同模型、CLI、插件、工具链。
7. **上下文统一**：不同来源内容必须抽象成统一上下文对象，保证动作层与执行层可复用。

### 1.3 面向的用户行为
用户并不是想”和 AI 聊天”，而是想：
- 对当前看到的内容立刻做事
- 不离开当前工作界面
- 不复制粘贴很多内容
- 快速触发分析、总结、生成、转换、写邮件、提取表格、生成报告等动作
- 在之后随时回到主程序查看最近做过什么、结果在哪里、是否执行成功

> **【Claude 注释】** 这一段定位非常准确——核心是”做事而不是聊天”。但还需要补充：
> - **竞品差异化定位**：与 Grammarly（写作辅助）、Raycast/Alfred（启动器）、Copilot（编辑器内）、Power Toys、Coze/Dify（无代码 Agent）相比，本品的独特价值是”系统级 + 上下文驱动 + 任务可追踪”。如果不写清楚和这些产品的关系，开发后期容易功能定位漂移。
> - **目标用户画像**：是科研工作者、白领、学生、还是 Pro Users？目标用户决定了 Office/PDF 优先级、模型成本承受度、隐私敏感度。建议补一个”目标用户画像”小节。

### 1.4 非目标（建议补充）
> **【Claude 注释】** 一个好的产品方案应该明确”不做什么”。建议在此处补一个”非目标”小节，例如：
> - 不做完整 IDE 内代码补全（那是 Copilot/Cursor 的领域）
> - 不做团队协作 Workspace（不是 Notion/Confluence）
> - 不做对话式 Chatbot（不是 ChatGPT 客户端）
> - 不做对所有 GUI 应用做无差别 OCR/截图监控（隐私雷区）

## 2. 产品原则

### 2.1 速度优先原则
系统整体设计必须遵循：
- 所有检测逻辑尽量事件驱动，而不是重轮询
- 轻动作必须尽可能本地快速完成
- 重动作必须先进入任务队列，再流式反馈
- 不能让用户等待长时间无反馈
- 不能识别半天才出现入口
- 不能点击后卡住没有状态

### 2.2 多入口并存原则
不能把产品能力完全押注在“像 Grammarly 一样跟随鼠标的小图标”上。必须同时支持：
- 跟随浮标
- 固定悬浮窗
- 右键菜单
- 快捷键唤起
- 浏览器扩展入口
- Office 内按钮/插件入口
- 拖拽到浮窗

这样即使部分应用不能精准检测选区，也能有兜底路径。

### 2.3 统一上下文原则
所有来源的内容必须转成统一结构，而不是让每个执行器单独理解输入。

### 2.4 任务分级原则
不是所有命令都要进入复杂 agent 工作流。
必须区分：
- 即时动作
- 中等动作
- 深度任务

### 2.5 显示优先原则
状态显示必须清楚：
- 当前识别到什么
- 将要执行什么
- 正在执行到哪一步
- 成功与失败怎么区分
- 结果在哪里
- 是否需要用户确认
- 是否支持撤销/重试/继续处理

> **【Claude 注释 - 关于"快"】** 2.1 提到"快"，但缺少 SLO 量化。建议明确：
>
> | 操作 | P50 目标 | P95 目标 | 失败兜底 |
> |---|---|---|---|
> | 浮窗冷启动可见 | 150ms | 350ms | >500ms 自动降级到固定窗 |
> | 热路径识别上下文 | 80ms | 200ms | 超时则显示"识别中" |
> | 轻执行器返回首字 | 800ms | 2s | 显示流式 loading |
> | 任务入队反馈 | 50ms | 150ms | 强制本地确认 |
>
> 没有 SLO，"快"就是空口号。Phase 2 的验收里建议直接 plug 这些数字。

### 2.6 安全与隐私优先原则（建议补充）
> **【Claude 注释】** 原方案完全没有提"隐私"。这是一个会读屏、读剪贴板、读文件的产品，在 Windows 上属于"高敏感行为"。建议补充：
> - **本地 vs 云**：默认所有上下文先脱敏后上传，敏感字段（手机号/邮箱/身份证/银行卡）正则脱敏
> - **静默清单**：用户可加入"应用白名单/黑名单"——例如银行客户端、Bitwarden、密码管理器自动禁用
> - **数据保留期**：剪贴板缓存默认 5 分钟自动清理，截图文件 24 小时清理
> - **审计日志**：每一次 LLM 调用都本地留痕——发了什么、收了什么、谁触发的
> - **Kill switch**：托盘图标一键全局禁用

## 3. 使用场景总览

### 3.1 文件场景
用户在资源管理器中选中文件或文件夹：
- 分析文件内容
- 生成摘要/报告
- 读取多个文件并比较
- 提取表格/图片/结构
- 生成邮件、PPT 大纲、任务清单
- 归档、重命名、分类

### 3.2 Office 场景
用户在 Word、Excel、PPT 中选中内容：
- 总结选区
- 改写内容
- 提取结论
- 生成图表
- 基于选区生成邮件
- 将选区发送到控制台做进一步分析

### 3.3 PDF 场景
用户在 PDF 阅读器中选中内容或指定文件：
- 抽取文本
- 摘要
- 提取章节结构
- 提取表格
- 生成阅读报告
- 转换为可编辑文档

### 3.4 网页场景
用户在网页中选中文本、链接、图片、表格、代码块：
- 总结网页
- 提取信息
- 保存为知识卡片
- 抓取内容生成报告
- 转成邮件/笔记
- 做竞品分析

### 3.5 图片/截图场景
用户选中图片、截图、剪贴板图像：
- OCR
- 内容理解
- 图表识别
- 表格识别
- 生成图片说明
- 标注重点

### 3.6 通用浮窗场景
用户主动打开悬浮窗：
- 分析当前剪贴板
- 读取最近上下文
- 输入命令处理最近选中的文件/文本
- 查看最近任务
- 打开报告输出目录
- 从主控制台继续处理

## 4. 总体架构

系统建议划分为七层：

1. **上下文感知层 Context Capture Layer**
2. **悬浮交互层 Overlay & Entry Layer**
3. **命令理解与意图路由层 Intent Router**
4. **任务编排层 Task Orchestration Layer**
5. **执行器层 Executors Layer**
6. **状态与产物中心 State / Artifact Center**
7. **主控制台 Console / Workspace Layer**

> **【Claude 注释 - 缺失的层】** 七层划分思路是对的，但缺少两个关键的横切关注点（cross-cutting concern）：
>
> - **8. 安全与权限层 Security & Permission Layer**：处理脱敏、白/黑名单、敏感动作授权、Kill switch、审计。这一层不能"嵌"在其他层里，必须独立，否则后期补丁式打补丁会出问题。
> - **9. 可观测性层 Observability Layer**：埋点、性能指标、错误上报、用户行为统计、远程诊断。MVP 阶段不上 SaaS 化埋点也要写日志收集到本地，否则线上问题完全没法定位。
>
> 另外**进程模型**未明确。建议补充：
>
> ```
> ┌─────────────────────────────────────────┐
> │  Tray / 主控制台 (Electron 主进程)        │
> │  ↕ IPC                                  │
> │  Renderer 进程 × N (浮窗/控制台/详情页)    │
> └──────────┬──────────────────────────────┘
>            │ Named Pipe / WebSocket
> ┌──────────▼──────────┐
> │  Local Service (Node子进程或独立 daemon)  │
> │  - Task Queue                            │
> │  - Intent Router                         │
> │  - Executor Registry                     │
> │  - SQLite + Artifact Store               │
> └──────────┬──────────┘
>            │ spawn / stdin-stdout
> ┌──────────▼──────────┐
> │  Native Helper (C# / C++ / Rust)         │
> │  - UIAutomation / Hook / 全局快捷键         │
> │  - 屏幕截图、窗口枚举                      │
> └─────────────────────┘
>            │
> ┌──────────▼──────────┐
> │  Browser Native Messaging Host           │
> │  Office Add-in Server                    │
> │  Kimi Code CLI Subprocess                │
> └─────────────────────┘
> ```
>
> 这样划分的好处：
> 1. Electron 崩溃不影响后台任务（任务在 Local Service 里）
> 2. Native Helper 用 C#/Rust 写更稳，不污染 Node 主进程
> 3. Local Service 可以单独打包成 Windows Service / launchd 自启动

## 5. 上下文感知层设计

### 5.1 职责
负责识别用户当前所在环境、所选内容、当前可用动作来源，并快速产出统一上下文对象。

### 5.2 支持的上下文来源
1. 文件系统选择
2. 文件右键菜单
3. 文件拖拽
4. 剪贴板文本/图片/文件
5. 当前活动窗口信息
6. 浏览器选区、链接、图片、网页 DOM 元素
7. Office 选区
8. PDF 选区或指定文件
9. 屏幕截图/视觉兜底
10. 固定浮窗的手动输入

### 5.3 上下文对象统一结构
建议定义统一的 `ContextPacket`：

```json
{
  "context_id": "ctx_20260406_xxx",
  "source_type": "file | file_group | text_selection | image | link | webpage | clipboard | office_selection | pdf_selection | window | screenshot",
  "source_app": "Excel",
  "window_title": "Budget.xlsx - Excel",
  "capture_mode": "event | manual | plugin | extension | visual_fallback",
  "text": "选中的文本内容",
  "html": "可选网页HTML片段",
  "url": "https://example.com",
  "file_paths": ["C:/docs/test.pdf"],
  "image_paths": ["C:/temp/capture.png"],
  "selection_metadata": {
    "sheet_name": "Sheet1",
    "range": "A1:D20",
    "slide_index": 3,
    "page_index": 8,
    "mime_type": "application/pdf"
  },
  "entity_hints": {
    "contains_table": true,
    "contains_link": false,
    "contains_image": false,
    "possible_language": "zh-CN"
  },
  "captured_at": "2026-04-06T13:30:00-04:00"
}
```

### 5.4 上下文识别速度要求
目标设计：
- 文件选中识别：100–250ms 以内完成入口可见
- 剪贴板触发：100ms 以内给出状态
- 浏览器选区浮标：100–200ms 以内出现
- Office 插件选区：200–400ms
- 视觉兜底：允许 1–3 秒，但必须先显示”识别中”状态

> **【Claude 注释 - 关于 ContextPacket 结构】** 当前结构作为传输模型 OK，但缺关键字段，建议补：
>
> ```jsonc
> {
>   “schema_version”: “1.0”,                   // 必须，方便后续演进
>   “trace_id”: “trace_xxx”,                   // 全链路追踪
>   “source_process_id”: 12345,                // 用来支持白/黑名单
>   “source_process_name”: “EXCEL.EXE”,        // 同上
>   “security_level”: “public|internal|sensitive|secret”,
>   “redaction_applied”: true,                 // 标识 text 是否已脱敏
>   “redaction_map”: {...},                    // 脱敏对应表（本地保留）
>   “content_hash”: “sha256:...”,              // 用于幂等去重
>   “size_bytes”: 1024,                        // 用于传输预算控制
>   “language_detected”: “zh-CN”,
>   “user_consent_token”: “consent_xxx”        // 涉及隐私时的用户授权凭据
> }
> ```
>
> **关于 100-250ms 目标的可行性**：
> - 文件右键菜单弹出本身要 80-150ms（Windows Shell Extension 冷启动），所以”右键菜单出现就能看到本应用图标”是可达的，但**菜单项里要立刻显示 ContextPacket 摘要**就极难，建议改为”先弹菜单，点击后再做分析”
> - 浏览器选区 100-200ms 是合理的，需要扩展端做去抖处理，并直接走内容脚本，不能走 Native Messaging 往返
> - Office 200-400ms 偏乐观，Office.js Add-in 第一次加载可能需要 1-3s，建议区分”冷启动”和”热路径”

### 5.5 检测策略
#### 文件系统
- 利用资源管理器右键、拖拽和选中文件事件
- 不建议强行监听所有路径变化来猜选中对象

#### 浏览器
- 用浏览器扩展获取选区、链接、DOM 目标和坐标
- 扩展与桌面主程序通信

#### Office
- 用 Office Add-ins 或宿主插件事件获取选区
- 不要依赖 OCR 去识别 Office 内容

#### PDF
- 优先文件级处理
- 选区级集成视 PDF 阅读器而定
- 不通用时走“复制选区到程序”或“发送文件到程序”

#### 视觉兜底
- 对无法读取选区的 app，支持截图识别
- 必须明确提示用户当前是视觉模式，可能不如原生选区精确

> **【Claude 注释 - 关于"通用选区抓取"的现实】** 这一节的"检测策略"在 Windows 上落地有以下硬约束，需要写进风险章节：
>
> 1. **UIAutomation 不是万能**：很多 Electron 应用（VSCode、Slack、Discord）的文本选区根本不暴露给 UIAutomation；Chromium 系应用要打开 `--force-renderer-accessibility` 才行。最稳的做法仍然是：**浏览器扩展处理网页选区，Office Add-in 处理 Office，剩余应用走"复制到剪贴板 + 全局快捷键"**。
> 2. **文件选区没有标准 API**：Explorer 选中文件的 API 不是事件而是 `IShellWindows`，需要轮询或者 hook，**建议放弃"自动检测 Explorer 选中"**，改为右键菜单 + 拖拽 + 全局快捷键。
> 3. **PDF 阅读器**：Acrobat 有 IAC，Foxit 有自己的 SDK，SumatraPDF 没有。**统一抓取不可能**，建议只支持"打开 PDF 文件 + 整文件作为上下文"，选区交给用户复制。
> 4. **视觉兜底**性能成本极高（每张 1080p 截图本地 OCR 都要 200-800ms），建议仅在用户主动触发时启用，而不是后台轮询。

## 6. 悬浮交互层设计

### 6.1 交互入口类型
#### 6.1.1 跟随浮标
- 靠近选区或鼠标附近显示
- 尺寸小，默认只显示图标
- 支持 hover 预览动作，点击后展开
- 不能遮挡主要内容

#### 6.1.2 固定悬浮窗
- 固定在屏幕边缘
- 可折叠/展开
- 用于兜底唤起、历史查看、任务总览
- 可拖拽位置

#### 6.1.3 右键菜单
- 文件右键
- 选中文本右键（浏览器/支持的应用）
- 图片右键
- 链接右键

#### 6.1.4 快捷键入口
- 全局快捷键打开命令框
- 快捷读取当前剪贴板/最近上下文

### 6.2 悬浮 UI 的状态
1. **未出现**：当前上下文不满足显示规则
2. **准备出现**：识别到上下文，判断是否显示
3. **轻浮标状态**：只显示小图标
4. **展开状态**：显示快捷动作+输入框
5. **处理中状态**：已提交任务
6. **结果状态**：显示成功/失败/继续操作

### 6.3 显示原则
- 不在每次鼠标移动时乱闪
- 只在“上下文稳定”后出现
- 显示需要去抖与节流
- 可设定“只在选中内容时出现”
- 可设定“悬停模式”作为高级选项而非默认开启

### 6.4 弹出内容建议
点击浮标展开后包含：
1. 当前识别到的对象摘要
2. 快捷动作区
3. 自定义命令输入框
4. 是否需要确认的开关
5. 最近任务入口
6. “发送到主控制台继续处理”按钮

### 6.5 快捷动作模板
#### 面向文件
- 分析文件
- 生成摘要
- 生成报告
- 提取表格
- 写邮件
- 比较文件

#### 面向文本
- 总结
- 改写
- 翻译
- 提取待办
- 解释

#### 面向链接/网页
- 抓取并总结
- 保存为笔记
- 生成汇报
- 提取关键信息

#### 面向图片
- OCR
- 图像理解
- 提取图表
- 生成描述

### 6.6 低打扰机制
- 同一对象短时间内不重复弹出
- 用户关闭一次后，对该对象本轮不再弹出
- 支持”敏感窗口禁用”
- 支持”全屏场景禁用”
- 支持”会议/演示模式静默”

> **【Claude 注释 - 缺失的 UX 边界】** 桌面悬浮窗的工程难点不在”出现”而在”出现之后的所有边界情况”。建议补充：
>
> 1. **多显示器**：浮窗必须跟随光标所在屏幕；不同 DPI 缩放（100%/125%/150%/200%）下大小要正确
> 2. **HiDPI 渲染**：必须 Per-Monitor DPI Aware（Win10+）
> 3. **窗口层级**：悬浮窗 z-order 必须高于普通窗口但低于系统通知 / UAC 弹窗 / 锁屏
> 4. **Focus stealing**：悬浮窗一旦抢焦点会破坏用户在原应用的输入法/选区——必须用 `WS_EX_NOACTIVATE` + `SetWindowPos(HWND_TOPMOST)` 模式，绝对不能 `SetForegroundWindow`
> 5. **输入法**：浮窗输入框被激活时不能让用户在原应用的 IME 状态丢失——这是一个常见痛点
> 6. **快捷键冲突**：全局快捷键必须可配置且检测冲突
> 7. **Windows Focus Assist (勿扰模式)**：要尊重系统级勿扰，与”会议模式”区分开
> 8. **屏幕共享中**：检测 `IsScreenCaptureInProgress`，自动隐藏敏感悬浮窗

## 7. 命令理解与意图路由层

### 7.1 核心职责
将“用户输入 + 上下文对象”转化为系统真正可执行的标准动作。

### 7.2 路由前必须判断的维度
#### 7.2.1 动作复杂度
- 轻动作：几秒内可完成
- 中动作：10–30秒内完成
- 重动作：需要多步处理、文件生成、CLI 执行

#### 7.2.2 输出类型
- 文本输出
- 邮件草稿
- Markdown 报告
- Word/Excel/PPT/PDF 产物
- 图表
- 结构化数据
- 文件改动

#### 7.2.3 上下文粒度
- 单段文本
- 单文件
- 多文件
- 当前窗口
- 网页链接
- 截图图像
- 表格区域

#### 7.2.4 风险与确认需求
- 只读动作
- 会写入文件的动作
- 会发送内容的动作
- 会调用外部程序的动作
- 会覆盖原内容的动作

### 7.3 意图类型建议
系统内部不要只靠自然语言直接调执行器，建议先归类成动作类型：
- `summarize`
- `analyze`
- `rewrite`
- `translate`
- `extract_table`
- `extract_entities`
- `generate_report`
- `generate_email`
- `compare_sources`
- `create_presentation_outline`
- `convert_format`
- `classify_files`
- `send_to_console`
- `launch_deep_workflow`

### 7.4 路由决策输出结构

```json
{
  "route_id": "route_xxx",
  "intent": "generate_report",
  "complexity": "heavy",
  "executor_type": "kimi_code_cli",
  "requires_confirmation": true,
  "estimated_mode": "queued_task",
  "output_schema": {
    "type": "markdown_report",
    "save_required": true
  },
  "reasoning_summary": "选中对象为 PDF 文件，用户要求分析完整内容并生成正式报告，因此进入重任务流程"
}
```

### 7.5 灵活性设计
为了不死板，路由层要支持：
- 快捷动作 + 自然语言补充
- 一次命令里组合多个动作
  - 例如："分析这个文件，并给我一封发给导师的邮件"
- 模糊命令智能澄清
  - 例如："帮我处理一下这个表"
- 上下文补全
  - 例如：用户只说“帮我写封邮件”，系统默认引用最近一个文件或当前选区，但必须提示引用了什么

### 7.6 问答与澄清机制
系统必须支持两类确认：
#### 轻确认
直接在浮窗一句话追问：
- “你是想总结当前选区，还是整份文件？”
- “输出成邮件草稿还是 Markdown？”

#### 重确认
当涉及重任务时：
- 显示任务摘要
- 显示输入来源
- 显示预计输出类型
- 让用户确认再执行

### 7.7 默认确认策略
- 只读轻动作：默认免确认
- 文件生成类：轻确认或可配置自动执行
- 修改原文件：必须确认
- 发送邮件/对外动作：必须确认
- 多文件批量处理：建议确认

> **【Claude 注释 - Intent Router 的实现路径】** 这一节定义了"做什么"，但没说"怎么做"。Intent 分类有三种实现选项：
>
> | 方案 | 延迟 | 准确率 | 成本 | 推荐 |
> |---|---|---|---|---|
> | 纯规则/正则 | <10ms | 60% | 0 | 作为第一道筛选 |
> | 本地小模型(BERT/MiniLM) | 30-100ms | 85% | 一次性下载 200MB | **推荐 MVP 用** |
> | 云端 LLM 分类(JSON mode) | 300-1500ms | 95% | 按 token 计费 | 仅作为兜底 |
>
> 建议采用 **三层级联**：
> 1. 命中 Quick Action 模板按钮 → 直接路由（0ms）
> 2. 命中关键词正则（"总结""翻译""写邮件"）→ 直接路由
> 3. 都不命中 → 调用 LLM 分类器返回 `{intent, confidence, slots}`，confidence < 0.6 时进入轻确认
>
> 另外**多动作组合**（"分析+写邮件"）的实现方式应明确：是 LLM 返回 `actions: [...]` 数组，还是后端拆成 DAG？建议是 LLM 返回结构化 `plan`，编排层把它转成 DAG 任务。

## 8. 任务编排层设计

### 8.1 职责
把路由后的动作变成可执行的任务对象，交给合适执行器，并管理状态、重试、取消、依赖与结果归档。

### 8.2 任务模型
建议统一任务对象：

```json
{
  "task_id": "TASK-20260406-0001",
  "created_at": "2026-04-06T13:40:00-04:00",
  "source_context_id": "ctx_xxx",
  "user_command": "分析这个文件并生成报告",
  "intent": "generate_report",
  "priority": "normal",
  "execution_mode": "instant | queued | background_streaming",
  "executor": "kimi_code_cli",
  "status": "queued",
  "steps": [],
  "artifacts": [],
  "logs": [],
  "requires_confirmation": true,
  "confirmed": true
}
```

### 8.3 任务分级
#### A 类：即时任务
- 直接返回结果
- 例如短文本总结、翻译
- 在浮窗内展示即可

#### B 类：半即时任务
- 进入小任务队列
- 10–30秒
- 例如网页抓取后总结、单文件结构提取

#### C 类：深度任务
- 进入正式任务队列
- 显示进度、步骤、日志、产物
- 例如生成完整报告、跨文件分析、批量处理

### 8.4 任务生命周期
建议统一状态流：
- `draft`
- `awaiting_confirmation`
- `queued`
- `starting`
- `running`
- `streaming`
- `success`
- `partial_success`
- `failed`
- `cancelled`
- `waiting_user_input`
- `needs_review`

### 8.5 多任务机制
必须支持多任务并存：
- 浮窗里显示当前任务数量
- 主控制台显示任务列表
- 任务可并发，但需有并发上限
- 同类型重任务可做排队
- 支持“最近任务继续处理”

### 8.6 并发策略
建议：
- 轻任务：可高并发
- 中任务：中等并发
- Kimi Code CLI 类深度任务：低并发 + 排队
- 对同一来源对象的重复命令可提示复用或重跑

### 8.7 任务取消与中断
必须支持：
- 未开始任务：直接取消
- 运行中任务：请求中断
- 中断后保留部分日志
- 明确显示”已取消”与”失败”不同

### 8.8 持久化与崩溃恢复（建议补充）
> **【Claude 注释】** 这是当前方案最大的工程缺口之一：**任务必须能在程序崩溃后恢复**。
>
> 建议机制：
> - 所有任务状态变更立刻 WAL 写 SQLite，而不是等结束才写
> - 启动时扫描 `running` / `streaming` 状态的任务，按规则处理：
>   - 时间戳 < 5min：标记 `interrupted`，给用户一键恢复入口
>   - 时间戳 > 5min：标记 `failed`，附”程序异常退出”原因
> - Kimi CLI 子进程必须有 PID 记录，重启时尝试 `taskkill` 并清理临时目录
>
> 没有这一机制的话，用户跑完一个 10 分钟的报告任务，控制台一关任务全丢，体验会非常糟糕。

### 8.9 任务持久化数据模型（建议补充）
> **【Claude 注释】** 建议明确本地 SQLite schema：
>
> ```sql
> CREATE TABLE tasks (
>   task_id TEXT PRIMARY KEY,
>   created_at INTEGER,
>   updated_at INTEGER,
>   status TEXT,
>   intent TEXT,
>   executor TEXT,
>   user_command TEXT,
>   context_packet_json TEXT,
>   priority INTEGER,
>   parent_task_id TEXT,    -- 支持任务派生
>   trace_id TEXT
> );
> CREATE INDEX idx_tasks_status_updated ON tasks(status, updated_at);
>
> CREATE TABLE task_events (   -- 事件源(Event Sourcing) 风格
>   event_id INTEGER PRIMARY KEY,
>   task_id TEXT,
>   ts INTEGER,
>   event_type TEXT,           -- step_started/log/artifact_created/...
>   payload_json TEXT
> );
>
> CREATE TABLE artifacts (
>   artifact_id TEXT PRIMARY KEY,
>   task_id TEXT,
>   path TEXT,                 -- 绝对路径
>   mime_type TEXT,
>   size_bytes INTEGER,
>   created_at INTEGER
> );
> ```
>
> 用 Event Sourcing 而不是只存当前状态有几个好处：(1) 时间线展示天然可用 (2) 重放调试容易 (3) 支持 partial_success。

## 9. 执行器层设计

### 9.1 执行器分类
#### 9.1.1 快速执行器 Fast Executor
适合：
- 总结
- 改写
- 翻译
- 提取要点
- 基础问答

#### 9.1.2 工具执行器 Tool Executor
适合：
- 文件解析
- OCR
- 表格抽取
- 网页抓取
- Office 文档读写
- 文件格式转换

#### 9.1.3 Kimi Code CLI 执行器
适合：
- 深度分析
- 报告生成
- 多步工作流
- 文件产物生成
- 跨工具调用
- 结构化处理

### 9.2 Kimi Code CLI 的正确职责
Kimi 不应负责：
- 系统级 UI 感知
- 鼠标跟踪
- 浮窗渲染
- 入口是否显示判断

Kimi 负责：
- 读取任务包
- 根据上下文执行多步任务
- 调用本地工具
- 生成产物
- 输出执行日志
- 返回结果路径与总结

### 9.3 任务包机制
建议所有深任务都生成标准任务包给 Kimi：

```json
{
  "task_id": "TASK-20260406-0020",
  "task_type": "report_generation",
  "user_command": "分析这个文件内容，生成一份详细报告，并给出关键风险总结",
  "context": {
    "source_type": "file",
    "file_paths": ["C:/docs/sample.pdf"],
    "metadata": {
      "mime_type": "application/pdf"
    }
  },
  "output_requirements": {
    "primary": "markdown_report",
    "save_required": true,
    "output_dir": "C:/ProgramData/UCA/outputs/2026-04-06/"
  },
  "rules": {
    "must_read_source": true,
    "must_save_result": true,
    "must_return_artifact_paths": true,
    "must_emit_progress": true
  }
}
```

### 9.4 执行反馈协议
执行器必须在运行中返回结构化事件：
- `accepted`
- `started`
- `step_started`
- `step_finished`
- `log`
- `warning`
- `needs_input`
- `artifact_created`
- `success`
- `failed`

这能保证 UI 做出清晰状态展示。

> **【Claude 注释 - Kimi CLI Bridge 工程细节】** 这一节抽象层次合适，但 Kimi CLI 进程管理的细节缺失，建议补充：
>
> 1. **传输协议**：建议 stdin 投任务包(JSON Lines)，stdout 流式输出事件(JSON Lines)，stderr 收集到调试日志
> 2. **进程模型**：每个深任务起一个独立子进程而不是常驻——隔离崩溃，但启动慢；或者池化常驻 N 个 worker
> 3. **超时与心跳**：执行器必须每 30s 发心跳，10 分钟无心跳视为僵死，强制 kill
> 4. **资源限制**：CPU/内存/磁盘配额（Windows Job Objects）防止失控
> 5. **认证**：Kimi CLI 是否需要 API Key？Key 存在哪？建议用 Windows Credential Manager / DPAPI
> 6. **不可用时降级**：如果 Kimi CLI 未安装或挂掉，必须降级到"基础执行器返回简版结果 + 在控制台显示安装/重启提示"
> 7. **多版本兼容**：CLI 版本变化的契约协商机制
>
> 另外建议把"执行器"做成插件化注册：
>
> ```typescript
> interface Executor {
>   id: string;
>   name: string;
>   capabilities: Intent[];      // 它能处理的意图
>   priority: number;            // 同一个意图多个执行器时的偏好
>   estimateCost(task: Task): Cost;
>   execute(task: Task): AsyncIterable<ExecutorEvent>;
>   cancel(taskId: string): Promise<void>;
> }
> ```
>
> 这样未来接入 Claude CLI / OpenAI / 本地 ollama 都不用动核心代码。

## 10. 状态显示与执行显示设计

### 10.1 为什么状态显示是核心功能
这个产品不是一次性聊天工具，而是“执行工具”。执行工具必须让用户看得见：
- 系统识别到了什么
- 接下来打算做什么
- 当前进行到哪一步
- 是否成功
- 如果失败为什么失败
- 成果文件在哪里

### 10.2 浮窗中的状态显示
浮窗应提供轻量但完整的状态：
- 已识别对象："已识别 1 个 PDF 文件"
- 已接收命令："正在准备生成报告"
- 当前状态：排队中 / 执行中 / 等待确认 / 已完成 / 失败
- 最近结果："报告已生成，点击打开"

### 10.3 主控制台中的任务列表
建议字段：
- 任务名称
- 来源对象
- 命令摘要
- 当前状态
- 执行器
- 开始时间
- 持续时间
- 最近更新
- 结果数量
- 是否需要处理

### 10.4 任务详情页
任务详情需要包含：
1. 命令原文
2. 解析后的意图
3. 来源上下文
4. 执行步骤时间线
5. 日志流
6. 产物列表
7. 失败信息
8. 重试/继续处理入口

### 10.5 状态颜色与语义建议
- 蓝色：准备中 / 排队中 / 处理中
- 绿色：成功
- 黄色：等待用户、部分成功、提醒
- 红色：失败
- 灰色：取消/过期

### 10.6 成功后的显示
成功时必须清楚显示：
- 成功摘要
- 结果文件路径
- 打开文件
- 打开目录
- 在主控制台中继续处理
- 基于结果再执行下一步

### 10.7 失败后的显示
失败时不能只显示“失败”。
必须分层展示：
1. 失败在哪一步
2. 原因类别
3. 原始错误摘要
4. 用户可采取的动作
5. 是否支持一键重试

建议失败分类：
- 上下文读取失败
- 文件权限失败
- 解析失败
- 外部工具不可用
- CLI 执行失败
- 模型调用失败
- 输出保存失败
- 用户中断
- 网络异常

### 10.8 部分成功状态
很多任务可能会“部分完成”。
例如：
- 报告生成成功，但表格提取失败
- 总结完成，但邮件草稿未生成

这时状态应为 `partial_success`，并明确哪些完成，哪些失败。

### 10.9 流式执行显示
对中重任务，建议流式显示：
- 步骤开始
- 步骤日志
- 产物创建事件
- 当前处理阶段

示例：
1. 读取文件
2. 解析正文
3. 提取章节
4. 生成摘要
5. 写入报告
6. 报告已保存

## 11. 问答、确认与人机交互完整流程

### 11.1 用户交互不应过于死板
系统不能只有“输入命令 → 执行”。
必须支持：
- 追问
- 确认
- 补充上下文
- 推荐动作
- 中途要求用户决定输出格式

### 11.2 交互流程模板
#### 流程 A：简单即时任务
1. 用户选中文本
2. 浮标出现
3. 用户点击“总结”
4. 系统直接执行
5. 浮窗显示结果
6. 可继续“改写/发邮件/发送到控制台”

#### 流程 B：需要确认的重任务
1. 用户选中文件
2. 输入“分析并生成报告”
3. 系统展示：
   - 输入来源：sample.pdf
   - 输出：Markdown 报告
   - 执行器：Kimi Code CLI

5. 任务入队
6. 浮窗显示开始执行
7. 主控制台显示完整过程
8. 成功后可打开报告

#### 流程 C：信息不足需追问
1. 用户选中 Excel 区域
2. 输入“帮我处理这个表”
3. 系统判断意图不清
4. 提供选项：
   - 总结数据
   - 生成图表
   - 检查异常值
   - 写成汇报
5. 用户选择后继续执行

#### 流程 D：执行中需补充输入
1. 用户要求“基于这个文件写邮件”
2. 系统执行到邮件阶段
3. 发现缺少收件人或语气
4. 提示用户补充
5. 用户输入后继续执行

### 11.3 确认界面应显示的信息
- 将处理什么对象
- 将执行什么动作
- 是否会生成文件
- 是否会修改原对象
- 结果将保存在哪里
- 是否会调用外部工具

### 11.4 智能推荐动作
识别上下文后，系统应动态推荐：
- 选中文本 → 总结 / 改写 / 翻译
- 选中 PDF → 生成报告 / 提取结构 / 写邮件
- 选中表格 → 总结数据 / 生成图表 / 导出分析
- 选中链接 → 抓取内容 / 保存为笔记 / 做摘要

## 12. 多任务显示与任务中心设计

### 12.1 为什么必须有任务中心
因为用户会在多个界面连续触发很多命令：
- 一个 PDF 生成报告
- 一个网页抓取摘要
- 一个 Excel 生成图表

如果没有任务中心，用户会不知道哪些已经完成、哪些失败、结果在哪。

### 12.2 任务中心页面结构
#### A. 顶部概览
- 正在运行数量
- 排队数量
- 成功数量
- 失败数量
- 最近产物数量

#### B. 中间任务列表
每行显示：
- 任务标题
- 来源图标
- 命令摘要
- 当前状态
- 进度简述
- 最后更新时间
- 打开详情

#### C. 右侧/下方任务详情
- 步骤流
- 日志

## 13. 成功/失败/异常的完整处理逻辑

### 13.1 成功处理
成功后应自动提供：
- 结果预览
- 打开文件
- 打开目录
- 复制路径
- 基于该结果继续生成新动作

### 13.2 失败处理
失败时不能直接终止体验。
必须提供：
- 明确失败原因
- 失败发生阶段
- 原始日志入口
- 修复建议
- 重新执行按钮
- 切换执行模式选项

### 13.3 重试机制
支持：
- 原参数重试
- 修改参数后重试
- 改用其它执行器重试
- 跳过已完成步骤继续重试

### 13.4 降级机制
系统必须有降级策略：
- 原生选区读取失败 → 建议剪贴板模式
- CLI 失败 → 尝试基础执行器输出简版结果
- 深任务失败 → 先给部分中间结果
- 网页抓取失败 → 提示仅基于选中片段处理

### 13.5 用户中断机制
用户可以：
- 取消任务
- 暂停查看日志
- 关闭浮窗但不中断任务
- 关闭主窗口后后台继续执行（可配置）

## 15. 快速、灵活、不死板的关键机制

### 15.1 快速的机制
1. 事件驱动检测，而非高频扫描
2. 快速路径优先：轻任务先走轻执行器
3. 缓存最近上下文
4. 常见动作本地预生成模板
5. 日志与结果流式回传
6. UI 先响应，再补执行细节

### 15.2 灵活的机制
1. 支持自然语言命令
2. 支持快捷动作组合
3. 支持多来源统一处理
4. 支持同一结果继续派生动作
5. 支持改用其它执行器
6. 支持任务模板与用户自定义动作

### 15.3 不死板的机制
1. 不强制所有命令进入大工作流
2. 不把所有请求都变成 Master Plan 或复杂任务
3. 根据用户意图自动分流到最短路径
4. 出现歧义时轻追问，不要机械拒绝
5. 支持“先给草稿，再深度处理”模式

#### 浏览器扩展

#### Office 适配层

### 16.3 数据存储建议

### 16.4 事件总线
建议全系统基于事件流：
- context_detected
- overlay_shown
- command_submitted
- task_created
- task_started
- step_updated
- artifact_created
- task_failed
- task_success

> **【Claude 注释 - 技术选型应再细化】** 16.1 直接押 Electron 是合理选择（生态成熟、UI 灵活、Node 生态），但应当对比：
>
> | 维度 | Electron | Tauri | WPF (.NET) | Qt |
> |---|---|---|---|---|
> | 包体积 | 80-150MB | 5-15MB | 40MB | 30MB |
> | 内存 | 200-500MB | 80-150MB | 100MB | 80MB |
> | UI 灵活度 | ★★★★★ | ★★★★ | ★★★ | ★★★ |
> | 系统集成（hook/UIA） | 弱 | 中（Rust 直调） | **强** | 中 |
> | 团队上手 | 易 | 较难 | Win 限定 | 较难 |
>
> **实际建议**：Electron 做主控制台和悬浮窗 UI，但**系统级抓取 + 全局 hook 必须用 C# (.NET) 或 Rust 写一个独立的 Native Helper**，然后通过 Named Pipe 与 Electron 通信。纯 Electron 抓不动 UIAutomation。
>
> **缺失的几个核心模块**应补充到 16.2：
> - **Crash Reporter** - 收集崩溃 dump
> - **Update Manager** - 自动更新（建议用 electron-updater + 自托管 release server）
> - **Telemetry Client** - 性能/错误埋点（可关闭）
> - **License/Auth Manager** - 如果做商业化
> - **Settings Sync Module** - 多设备配置同步（可选）
> - **Permission Broker** - 统一管理白/黑名单和敏感动作授权

### 16.5 跨平台策略（建议补充）
> **【Claude 注释】** 整个文档默认 Windows，但没明说。建议补充：
> - **MVP 阶段**：Windows 10/11 only
> - **Phase 4 后**：考虑 macOS（AppleScript/Accessibility API/Services 菜单）
> - **Linux**：低优先，Wayland 抓取选区难度极高
>
> 写明边界比含糊好。

### 16.6 数据存储 schema 与并发（建议补充）
> **【Claude 注释】** 推荐 **SQLite + WAL 模式**，主进程独占连接，其他进程通过 Local Service 走 IPC 访问。理由：
> - 单文件、零运维、跨平台
> - WAL 模式下读写并发友好
> - 任务量不大（每天几百到几千条），SQLite 足够快
>
> 不建议用 LevelDB/IndexedDB（需要 schema migration 时痛苦）。

## 19. 风险与解决建议

### 19.1 风险：过度追求全场景原生检测
解决：先做多入口并存，不把成功率压在“任意位置悬浮跟随”上。

### 19.2 风险：执行逻辑过重导致响应慢
解决：轻任务与重任务严格分流，UI 先响应，结果后补。

### 19.3 风险：Kimi CLI 直接接自然语言导致不稳定
解决：统一任务包，限制输入结构，返回标准事件。

### 19.4 风险：失败黑箱
解决：步骤化事件流 + 失败分类 + 重试入口。

### 19.5 风险：用户觉得打扰
解决：低打扰策略、显示规则、静默模式、敏感窗口禁用。

### 19.6 风险：UIAutomation/选区抓取不通用（建议补充）
> **【Claude 注释】** 详见 §5.5 注释。解决：放弃"全局通用选区检测"幻想，多入口并存（剪贴板/快捷键/拖拽/扩展/Add-in）。

### 19.7 风险：LLM Token 成本失控（建议补充）
> **【Claude 注释】** 一份完整 PDF 报告生成可能消耗 50K-200K tokens，单任务成本几分到几毛。如果用户每天跑几十个任务，月成本会达到几十到几百元。
> 解决：
> - 模型分级（轻动作走 Haiku/小模型，深任务走 Opus/Kimi）
> - 上下文截断与摘要前置
> - 用户设置月度预算上限
> - 命中近期相同任务时提示复用旧产物

### 19.8 风险：隐私合规（建议补充）
> **【Claude 注释】** 这是一个会读屏读剪贴板的程序，企业客户和教育机构会拒绝部署。
> 解决：
> - 默认本地脱敏后再上传
> - 提供"完全离线模式"（仅用本地 LLM）
> - 提供企业级"代理网关"配置，让数据走客户自己的 LLM 网关
> - 明确数据政策与日志可审计

### 19.9 风险：MVP 范围过大、节奏失控（建议补充）
> **【Claude 注释】** 详见 §17 注释。Phase 1 应再拆为 1a/1b/1c，每个 sub-phase 都能独立 demo。

## 20. 最终建议

这个产品应该被定义为：

**“一个系统级、上下文驱动、低打扰、高可追踪的智能执行入口。”**

它不是单纯悬浮按钮，也不是单纯聊天框，而是：
- 在用户当前工作位置附近提供入口
- 快速识别上下文
- 快速理解动作
- 快速执行或转入任务流
- 在浮窗与主控制台之间形成完整闭环

最关键的成功标准不是“会不会弹出来”，而是：
1. 弹得准不准
2. 动作快不快
3. 流程顺不顺
4. 失败清不清楚
5. 结果找不找得到
6. 能不能继续下一步

如果以上 6 点做好，这个产品就会非常强。

## 22. 可行性分析与改进建议（Claude 补充）

### 22.1 总体评估

| 维度 | 评分 | 说明 |
|---|---|---|
| **产品定位清晰度** | ★★★★★ | "上下文驱动的执行入口" 这条产品主线非常清楚 |
| **UX/交互设计完整度** | ★★★★☆ | 状态显示/确认/失败处理写得好；缺多显示器/HiDPI/IME 等边界情况 |
| **架构合理性** | ★★★★☆ | 七层划分思路对，但缺安全/可观测性两个横切层和明确进程模型 |
| **工程可落地性** | ★★★☆☆ | 上下文检测部分严重依赖 UIAutomation 等不稳定能力，且范围过大 |
| **非功能性需求覆盖** | ★★☆☆☆ | 隐私/成本/离线/可观测性/测试/更新机制几乎全空白 |

**结论**：方案作为"产品愿景与高层设计"是 8/10，作为"工程实施方案"是 5/10，需要在落地前补齐三块——**(1) 系统级抓取的现实约束 (2) 进程模型与持久化 (3) 安全/成本/可观测性等非功能性需求**。

1. 把"做事"和"聊天"区分开，避开了大多数 AI 产品的同质化陷阱
2. 任务分级（轻/中/重）和确认策略很务实
3. 状态显示与失败处理设计完整，这是大多数 AI 工具的薄弱点
4. 多入口并存的兜底策略（不押注于跟随浮标）很成熟
5. ContextPacket / Task 的统一抽象方向正确

### 22.3 主要改进点（按优先级）

2. **明确进程模型**：Electron + Local Service + Native Helper 三进程划分（参见 §4 注释）
3. **加入安全与权限层**：白/黑名单、脱敏、Kill switch、敏感动作授权
4. **加入持久化与崩溃恢复**：任务 WAL + Event Sourcing（参见 §8.8、§8.9 注释）
5. **拆分 Phase 1** 为 1a/1b/1c，每个 ≤ 6 周

#### P1 - 显著影响产品质量
6. **量化 SLO**：为每个交互定义 P50/P95 延迟目标
7. **Intent Router 三层级联**：规则 → 本地小模型 → LLM 兜底
8. **Kimi CLI Bridge 工程细节**：进程隔离、心跳、降级、配额（参见 §9.4 注释）
9. **多显示器/HiDPI/IME/Focus Stealing** 边界情况（参见 §6.6 注释）
10. **成本控制**：模型分级 + 月预算 + 命中复用

#### P2 - 中长期完善
11. 可观测性层（埋点/远程诊断/性能指标）
12. Auto-update 机制
13. 离线模式与企业代理网关
14. 测试策略（单元/集成/E2E/性能回归）
15. i18n / a11y

### 22.4 不建议做的事

1. **不要做"鼠标跟随浮标 across all apps"**：技术 ROI 最差，浏览器扩展 + Office Add-in 已经覆盖 80% 价值场景
2. **不要做后台屏幕监控/OCR 轮询**：隐私雷区且性能不可接受，改为用户主动截图
3. **不要做自己的 LLM 推理后端**：MVP 阶段直接走云 API，自己跑模型是巨大坑
4. **不要做团队协作**：和定位冲突，会拖死 MVP
5. **不要做插件市场**：在产品没站稳前不要开放扩展 API
6. **不要在 Phase 1 同时支持 macOS/Linux**：先把 Windows 做透

## 23. Codex 复核意见（2026-04-08）

### 23.1 总体结论

我对 Claude 的总体判断 **基本同意**，尤其同意以下四点：
1. 这份文档目前更像"高质量产品方案"，还不是可直接开工的工程蓝图
2. Windows 上的通用选区抓取必须收缩预期，不能把成功率押在 UIAutomation 上
3. 安全/隐私/成本/可观测性必须前置，而不是 MVP 后再补
4. Phase 1 必须拆小，否则工程节奏和验收都会失控

如果用一句话概括：**Claude 的方向判断是对的，当前真正还差的不是"再多想几个功能"，而是把少数关键假设变成可验证的工程决策。**

### 23.2 我补充的四个落地提醒

#### A. Explorer 右键入口不要把"托管 Shell Extension"当成 MVP 前提
Claude 说要补系统层落地细节，这一点我是同意的；而在 phase 细化后，我认为这里还应再保守一点：
- **MVP 首选**：注册表命令菜单 / `uca-cli.exe`
- **增强版再评估**：`IExplorerCommand` / 自定义 Shell Extension

原因不是功能做不到，而是 **in-proc Explorer 扩展一旦做复杂，调试、签名、兼容性和崩溃隔离成本都会迅速上升**。这类能力适合做优化项，不适合作为 Phase 1b 的前置阻塞。

#### B. Scheduler 需要明确"无人值守授权模型"
Claude 提出了安全层和确认流，这个方向正确；但当文档扩展出 Scheduler 之后，需要再补一句更具体的话：

> **不是所有可交互动作都适合进入定时任务。**

定时任务如果运行时还要等确认，就会卡死；如果不确认，又会绕过安全边界。建议在协议里增加：
- `unattended_safe`：允许无人值守自动执行
- `approval_required`：触发后生成待审批任务，不直接执行

这样 Scheduler 才不会和 Confirmation / Security Broker 自相矛盾。

#### C. 脱敏映射只放内存是对的，但要补"崩溃后的 fail-closed 语义"
我同意 Claude 把隐私前置，也同意默认不把脱敏映射持久化到磁盘；但这会引出一个恢复语义问题：
- service 崩溃或重启后，旧任务若还想继续完成，就**拿不回原文**

因此文档应明确：
- 默认策略：**任务直接失败并保留脱敏占位符**，不做跨重启还原
- 若未来必须支持跨重启恢复，再引入 **DPAPI 加密的短期密封存储**

换句话说，这里不是功能缺失，而是需要把**失败语义写清楚**。

#### D. Office localhost HTTPS 不能当作"已解决问题"
Claude 提醒了 Add-in 类型与现实约束，我完全同意；进一步说，phase 文档里把 `https://localhost:9413 + 自签根证书` 写得太像既定方案了。更稳妥的写法应是：
- 把它列为 **Phase 4 前置 spike / ADR**
- 若企业策略禁止安装自签根证书，Office 集成要有降级路径，而不是整期卡住

建议的降级路径：
- 先保证 **选区读取 + 发任务 + 控制台查看结果**
- 回写文档、深度嵌入式体验作为第二层验收

### 23.3 我对 Claude 建议的边界判断

有两点我会比 Claude 再收一点：

1. **不要在 Phase 0 试图一次性定死所有后期实现**
Phase 0 更适合定：
- 协议
- 进程边界
- 风险登记
- 必做 spike

而不是把 Office、OCR、Scheduler、Action Tools 的所有技术路线都提前锁死。

2. **不要把"可观测性/安全/成本"理解成要首期做完整平台**
这些能力必须前置，但前置的重点是：
- 有统一入口
- 有最小可审计链路
- 有明确禁止绕过的规则

不一定要在 Phase 1 就把所有报表、远程诊断、企业策略中心一次做满。

### 23.4 最终判断

我的结论是：
- **Claude 的大方向判断合理，我整体赞成**
- **phase 拆分后的路线也基本成立**
- 但在真正开工前，还应把以下 4 件事列成明确 ADR / spike：
  1. Explorer 菜单入口的 MVP 实现方式
  2. Scheduler 的无人值守授权模型
  3. 脱敏映射的崩溃恢复语义
  4. Office localhost HTTPS / 证书方案

这 4 个点补齐后，这套方案就会从"想得很完整"更接近"能稳稳落地"。
