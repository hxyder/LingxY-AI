# Linxi Agent 语义路由与执行架构思路

## 0. 文档定位

这份文档用于统一 Linxi 当前 Agent 框架的方向：用户请求先经过语义理解，形成结构化任务契约，再进入工具策略和执行器。它替代早期“BERT / 正则 / 三维标签 / A-B-C 模式 / SaaS 后端部署”的旧思路。

> **Codex comment:** 当前目标不是回到关键词或硬规则路由，而是做一个 DeepSeek 式的**语义任务分类器**：先按用户语义产出结构化 IntentRoute / TaskSpec，再进入对应执行路径。之前失败的根因不是“没有分类器”，而是多层规则、硬信号、语义判断和执行器互相抢裁判权。

> **Codex comment / recommendation:** `routingModel` 应该是可选配置。用户在 Settings 里单独配置了 routing model，就用它做 SemanticRouter；否则直接使用用户当前选择的 QA model。这样普通用户无需额外配置，高级用户可以用 DeepSeek / Kimi / 便宜快模型降低路由成本和延迟。

---

## 1. 总体执行链路

当前推荐链路是：

```text
用户输入 + 上下文
-> 硬信号提取
-> SemanticRouter / 语义任务分类器
-> IntentRoute
-> TaskSpec + DecisionTrace
-> EvidencePolicy / ToolPolicy / SuccessContract
-> Executor Resolver
-> tool_using / multi_modal / 可选 DAG
-> Action Tool Guard / Risk / Approval
-> Final Synthesis
-> Store / SSE / UI
```

核心原则：

1. `SemanticRouter` 判断任务场景。
2. **硬信号只做约束**、事实、安全边界，不判断语义场景。
3. `TaskSpec` 是唯一任务契约。
4. 执行器只根据 `TaskSpec / ToolPolicy / SuccessContract` 行动。
5. 工具结果必须经过最终合成，不能直接当作用户答案。

当前代码已经接近这条链路：

| 层 | 代码位置 | 状态 |
|---|---|---|
| 语义路由 | [src/service/core/intent/semantic-router.mjs](src/service/core/intent/semantic-router.mjs) | 已有 strict schema |
| 上下文预处理 | [src/service/core/intent/router-preflight.mjs](src/service/core/intent/router-preflight.mjs) | 已接入 |
| 任务契约 | [src/service/core/task-spec.mjs](src/service/core/task-spec.mjs) | 已有 |
| 工具策略 | [src/service/core/policy/tool-policy-resolver.mjs](src/service/core/policy/tool-policy-resolver.mjs) | 已有 |

## 2. 谁来判断任务场景

任务场景由 `SemanticRouter` 判断，不由关键词、正则、文件扩展名、URL 或执行器自己判断。

### 2.1 routingModel 解析顺序

推荐规则：

```text
Settings.routingModel 已配置
-> 使用 routingModel 做 SemanticRouter

Settings.routingModel 未配置或为 auto
-> 使用用户当前选择的 QA model 做 SemanticRouter

QA model 不支持结构化 JSON / tool schema
-> 进入保守 fallback
```

Settings 建议配置形态：

```json
{
  "ai": {
    "router": {
      "enabled": true,
      "routingModel": "auto",
      "providerId": null,
      "model": null,
      "timeoutMs": 10000,
      "cacheTtlMs": 300000
    }
  }
}
```

字段说明：

| 字段 | 含义 |
|---|---|
| `enabled` | 是否启用 SemanticRouter |
| `routingModel` | `auto` 表示复用 QA model；也可以填用户选择的专用路由模型 |
| `providerId` | 可选，指向 Settings 里已有 provider |
| `model` | 可选，指定 provider 下的模型名 |
| `timeoutMs` | 路由调用超时 |
| `cacheTtlMs` | 相同输入的短期缓存时间 |

> **Codex comment / recommendation:** 不要要求用户额外安装 DeepSeek 或本地模型。默认复用 QA model。用户愿意优化速度时，再在 Settings 里单独指定 `routingModel`。

### 2.2 能力探测

不能只按模型名判断是否可路由，必须探测 provider 是否支持结构化输出：

| Provider 类型 | 建议 |
|---|---|
| OpenAI compatible / DeepSeek / Kimi API | 可优先用于 SemanticRouter |
| Anthropic tool use | 可用于 SemanticRouter |
| code_cli | 不建议作为 SemanticRouter 默认 provider |
| Ollama / 本地模型 | 需要实测 JSON 稳定性，不稳定则 fallback |

Fallback 规则：

1. 不执行高风险动作。
2. 不伪装实时查询结果。
3. 不把分类失败默认变成搜索。
4. 用户明确“不联网”时永远禁止联网。
5. 用户给了本地文件时优先本地上下文。
6. 不确定时由 `tool_using` 询问澄清或请求授权。

## 3. 硬信号的职责

硬信号不能判断“天气 / 新闻 / 股价 / 写文档 / 搜索 / 邮件”这种语义场景。硬信号只做事实和约束。

应该保留的硬信号：

| 信号 | 用途 |
|---|---|
| 明确“不联网 / 不搜索” | 强制 `web=forbidden` |
| 明确“帮我查 / 搜一下” | 作为外部信息需求证据，但仍交给 SemanticRouter 合并 |
| 本地文件 / 上传文件 / 当前选择 | 说明 source_scope 是本地事实 |
| URL | 说明存在外部资源，不等于整个任务都必须搜索 |
| 图片附件 | 选择 `multi_modal` |
| 时间短语 | 区分立即执行、定时、提醒 |
| 删除、发送、写入、执行命令 | 风险和审批边界 |
| connector 账号状态 | 能力边界和授权状态 |

> **Codex comment / risk:** 让硬信号判断任务场景会重现旧问题：规则和模型理解冲突时，任务会走错工具、走错执行器，甚至违背用户边界。正确做法是硬信号 veto 或约束 SemanticRouter，而不是替代 SemanticRouter。

## 4. IntentRoute / TaskSpec 契约

SemanticRouter 输出的是结构化 IntentRoute，之后编译成 TaskSpec。

当前代码中的主要字段：

```json
{
  "source_scope": "none | selection | uploaded_files | current_context | local_project | browser_page | external_world",
  "web_policy": "forbidden | optional | required",
  "output_kind": "conversation | markdown | file | docx | pptx | xlsx | pdf | html | csv | dashboard",
  "artifact_required": false,
  "executor": "fast | tool_using | agentic | translate | multi_modal | kimi",
  "research_depth": "single_lookup | multi_source | deep_research | unknown",
  "primary_intent": "qa | writing | coding | debugging | architecture_design | research | file_analysis | data_analysis | artifact_generation | automation | computer_control | email_calendar_action | unknown",
  "domain": "general | agent_harness | dairy_science | finance | tax | career | software | design | academic_writing | other",
  "user_goal": "...",
  "expected_output": "direct_answer | step_by_step | code | markdown_doc | table | email_draft | ppt | image | plan | execution | artifact | summary | comparison | recommendation | analysis | action_items | raw_results",
  "needs_external_info": false,
  "needs_current_information": false,
  "needs_user_files": false,
  "needs_tool_use": false,
  "needed_capabilities": ["external_web_read", "file_read", "artifact_generation", "code_execution", "browser_control", "email_calendar_action", "desktop_action", "image_understanding", "image_generation", "none"],
  "source_mode": "no_external | provided_context | single_lookup | multi_source_research | deep_research | unknown",
  "complexity": "low | medium | high",
  "risk_level": "low | medium | high",
  "interpretation": "immediate | schedule | needs_clarification",
  "confidence": 0.0
}
```

注意：

1. `needed_capabilities` 必须使用当前 schema 里的枚举，不要凭空写 `email_send`、`file_write` 这类未定义能力。
2. 写入、发送、删除等风险应由 `risk_level`、工具 guard、approval gate 和具体 tool schema 共同约束。
3. `output_kind` 表示输出载体，`expected_output` 表示用户期待的答案形态，两者不要混用。
4. `executor` 是路由建议，不应越过 `Executor Resolver` 直接生效。

## 5. 执行器策略

当前执行器选择应保持简单：

```text
有图片附件
-> multi_modal

其他任务
-> tool_using
```

`fast`、`agentic`、`kimi`、`translate` 可以继续存在，但不应该回到旧的 A/B/C 硬映射表。

### 5.1 single-brain agent 是默认路径

桌面助手的大多数任务适合单脑 agent loop：

1. 读取 TaskSpec。
2. 看 ToolPolicy。
3. 决定是否调用工具。
4. 接收工具结果。
5. 继续调用或最终合成。

这样比多套执行器各自解释用户语义更稳。

### 5.2 DAG / 多 Agent 只做复杂任务加速器

DAG 不应由 `complexity=high + needed_capabilities >= 3` 这种粗规则触发。更合理的触发条件是：

1. 用户目标可拆成两个以上互不依赖的子目标。
2. 明确需要多源调研、批量文件处理或多个长耗时读取分支。
3. 子任务之间可以并发，不需要前一步结果才能确定下一步。
4. ToolPolicy 允许相关工具执行。
5. 风险和审批边界能按子任务独立处理。

不适合 DAG 的场景：

1. 强顺序推理任务。
2. 简单问答。
3. 只有一个工具调用的任务。
4. 高风险写入任务。
5. 需要先澄清用户意图的任务。

> **Codex comment / recommendation:** Linxi 应保持 “single-brain agent + 可选 DAG” 架构。DAG 是加速器，不是默认裁判。

## 6. 工具选择与工具描述

工具选择准确率的优先级高于引入新模型。

推荐工具 description 格式：

```text
TRIGGER when:
- 用户需要 ...
- TaskSpec.primary_intent 是 ...
- ToolPolicy 允许 ...

SKIP when:
- 用户只需要解释，不需要执行
- 用户明确禁止联网/写入/发送
- 参数缺失且不能安全默认

ARGS:
- query: ...
- source: ...
- limit: ...
```

优先升级项：

1. 重写工具 description，使其面向 LLM 决策，而不是面向开发者注释。
2. 按 `primary_intent` 动态注入少量 few-shot。
3. 用 `needed_capabilities` 和 ToolPolicy 过滤工具可见范围。
4. 保留现有 tool-call 参数校验。

### 6.1 工具向量 Top-K 的位置

工具向量 Top-K 可以保留为未来优化，但它不是任务分类器。

适用条件：

1. 工具数量超过约 50 个。
2. 全量工具描述已经明显拖慢首 token。
3. 已经有 phase timing 证明工具 prompt 是瓶颈。
4. Top-K 只负责缩小工具候选，不负责判断用户场景。

不建议现在立刻引入 `@xenova/transformers`。如果未来需要：

1. 模型应打包进安装包。
2. 不应要求用户运行时下载模型。
3. 冷启动和内存占用要进入性能指标。

## 7. 速度收益与性能指标

这条路线会让复杂任务更快、更稳，但不保证所有请求都立刻变快。

| 场景 | 预期变化 |
|---|---|
| 简单闲聊 / 普通 QA | 可能多一次路由开销 |
| 需要判断是否联网 | 通常更快，因为少走错路径 |
| 本地文件任务 | 通常更稳更快，因为不会误搜索 |
| 多工具任务 | 更快，主要来自少返工 |
| 批量 / 多源任务 | 后续可通过 DAG / 并行工具明显提速 |

关键优化：

1. `routingModel` 用低温度、短 JSON、低 token。
2. SemanticRouter 做 5 分钟短缓存。
3. routingModel 未配置时复用 QA model，避免强制用户多配模型。
4. 工具 prompt 按能力过滤，减少无关工具。
5. read-only 工具结果可做短 TTL 缓存。
6. 只读且幂等的工具可逐步并行。

### 7.1 phase_timings

不要承诺固定 `< 200ms`。真实延迟取决于模型、网络、provider、本地机器、工具数量。

推荐新增窄表或事件式指标：

```text
task_id
phase
started_at
ended_at
duration_ms
status
metadata_json
```

优先记录阶段：

1. submit_to_task_created
2. semantic_router
3. task_spec_compile
4. tool_policy_resolve
5. executor_resolve
6. first_model_token
7. first_tool_call
8. tool_execution
9. final_synthesis
10. task_done

> **Codex comment / recommendation:** 不建议做“一行一个 task、一列一个 phase”的宽表。阶段会继续演化，宽表很快僵化。窄表或 task_events 更适合当前项目。

### 7.2 工具并行

工具并行可以提速，但不是小改。

当前 `tool_using` 更接近“一轮一个工具调用”的契约。要并行，必须先限定：

1. 只读工具。
2. 幂等工具。
3. 不需要审批。
4. 不互相依赖。
5. 可独立失败并合并错误。
6. SSE 事件顺序可解释。
7. 最终答案仍经过 final synthesis。

推荐先做 read-only 并行，再考虑更广泛的工具并发。

## 8. 回答生成与最终合成

工具执行完成不等于任务完成。

```text
TOOL_DONE != TASK_DONE
```

最终回答必须由 `expected_output`、`output_kind`、工具结果和 SuccessContract 共同决定。

可以模板化的部分：

1. docx / xlsx / pptx / pdf 等 artifact 结构。
2. 引用块格式。
3. 执行确认格式。
4. 表格列顺序。
5. 工具结果摘要 metadata。

不应机械模板化的部分：

1. 普通对话回答。
2. 邮件草稿正文。
3. 推荐建议。
4. 分析结论。
5. 用户要的自然语言解释。

> **Codex comment / recommendation:** 模板可以约束结构，但不能替代最终合成。最终合成要回答用户真正的问题，而不是把工具结果拼出来。

## 9. 桌面部署与用户安装

Linxi 是 Windows 桌面 Electron Agent，不是 SaaS 后端服务。

推荐拓扑：

```text
Electron main
-> renderer: overlay / console / dock
-> local service runtime: HTTP + SSE on localhost
-> SQLite / runtime.json / outputs / logs
-> helper processes: Office add-in / browser extension native host / OCR / connectors
```

持久化位置：

| 数据 | 位置 |
|---|---|
| 配置 | `%APPDATA%\UCA\config\runtime.json` |
| SQLite | `%APPDATA%\UCA\data\uca.db` |
| 输出文件 | `%APPDATA%\UCA\outputs` |
| 日志 | `%APPDATA%\UCA\logs` |
| 插件 / MCP / skills | `%APPDATA%\UCA\data\integrations` |

不推荐引入：

| 候选 | 结论 | 原因 |
|---|---|---|
| FastAPI / Uvicorn | 不装 | 跨语言栈，和 Electron + Node 主线冲突 |
| Redis | 不装 | 单机桌面用进程内 Map / SQLite 足够 |
| Grafana | 不装 | 单机桌面过重 |
| OpenTelemetry 后端 | 暂不装 | 先用本地 stats / SQLite 指标 |
| Pydantic | 不装 | Python 栈；当前是 JS / JSON schema 风格 |
| `@xenova/transformers` | 暂不装 | 工具数没到阈值，且会增加安装包和冷启动成本 |

可以保留：

1. `/health`：本地扩展、Office add-in、tray 和诊断需要。
2. `/metrics`：当前已有本地 metrics endpoint，可供 Console 或调试使用。
3. 本地 logs 和 stats UI。

用户侧安装原则：

1. 普通用户只安装 NSIS / 桌面安装包。
2. 不要求用户单独安装 Redis、Python 服务、向量模型或监控系统。
3. 如果未来引入本地 embedding 模型，应随安装包打包。
4. routingModel 不要求额外安装，默认复用 QA model。

## 10. 合理、不合理与推荐升级路线

### 10.1 合理

1. 用 SemanticRouter 做主场景判断。
2. `routingModel` 可选，默认复用 QA model。
3. 硬信号只做约束和事实。
4. TaskSpec 作为唯一任务契约。
5. ToolPolicy / SuccessContract 做执行边界。
6. single-brain agent 作为默认路径。
7. DAG 只用于可并发的复杂任务。
8. 工具结果必须最终合成。
9. 桌面端用 SQLite / 本地配置 / 本地日志。

### 10.2 不合理

1. BERT / 正则 / 关键词作为第一层场景裁判。
2. `realtime / certainty / load` 三维标签直接映射 A/B/C。
3. 分类失败默认搜索。
4. 工具结果直接当答案返回。
5. tool parallel 不设风险边界就并发。
6. 把 transitive dependency 当项目可用依赖。
7. 为桌面 Agent 引入 FastAPI / Redis / Grafana。
8. 要求用户额外安装路由模型或 embedding 模型。

### 10.3 推荐实施顺序

| 优先级 | 事项 | 说明 |
|---|---|---|
| P0 | 明确文档和代码口径 | 删除旧三维标签 / A-B-C / SaaS 部署主叙述 |
| P1 | Settings 增加 `routingModel` | `auto` 默认 QA model；用户可选专用快模型 |
| P1 | provider 能力探测 | 不支持结构化输出时 fallback |
| P1 | phase_timings | 先量化慢在哪里 |
| P2 | 工具 description 重写 | `TRIGGER / SKIP / ARGS` |
| P2 | few-shot 按 intent 注入 | 提升工具选择准确率 |
| P2 | read-only tool 缓存 | 进程内 TTL，先不用 Redis |
| P3 | 安全工具并行 | 只读、幂等、无需审批 |
| P3 | DAG 触发策略 | 只处理可并发复杂任务 |
| P4 | 工具 Top-K embedding | 工具数和指标证明需要后再做 |

## 11. 对 Claude 看法的处理

Claude 的大方向和本文一致：反对正则回潮，支持 SemanticRouter / TaskSpec / ToolPolicy 主链路，也指出旧文档里 FastAPI / Redis / Grafana 与桌面 Agent 不匹配。

本文采纳 Claude 的部分：

1. 工具向量 Top-K 可作为未来 prompt 瘦身手段。
2. 工具 description 重写是高 ROI。
3. read-only 缓存值得做。
4. 流式覆盖率需要审计。
5. SaaS 部署清单应改为桌面本地拓扑。

本文修正 Claude 的部分：

1. `routingModel` 还不是已实现配置项，应标为待实现。
2. 工具并行不是小优化，必须先限定只读、幂等、无需审批。
3. DAG 不能只靠 `complexity=high + needed_capabilities >= 3` 触发。
4. `phase_timings` 不建议做宽表。
5. `/health` 和本地 `/metrics` 仍有价值，不应完全删掉。
6. `lru-cache` 如果只是 transitive dependency，不能直接当项目依赖使用。
7. 当前 tool-call validator 不是 Pydantic，也不要引入 Python 栈。
8. 文档里的能力名必须以 `semantic-router.mjs` 的枚举为准。

> **Codex comment / final:** 更新后的方向是：用户语义优先，硬信号约束，TaskSpec 统一契约，工具策略负责边界，执行器只执行，最终合成负责把工具结果变成用户真正要的答案。`routingModel` 是速度优化入口，但不是用户必须安装的新组件。

## 12. 视觉路径：从 executor 路由迁到 tool-backed specialist

**状态**：已确认方向，未实施。下一个 PR 的目标。

### 12.1 现状问题

当前 image task 走 `multi_modal` executor，路由由 `executor-resolver.mjs:hasImage → multi_modal` 强规则 + `image-submission.mjs` 的视觉意图正则 `VISION_ANALYSIS_RE` 联合决定。已知病症：

1. 用户说"这是个什么"等不在正则里的口语 → 漏判 → 强转 `tool_using` → 走 chat provider（DeepSeek）→ 用户配置的 Vision/Image (Qwen) 形同虚设。`task_5f43d7f0-9c0d-4114-a0fa-3768f7049d4c` 是真实样本。
2. 改正则永远收敛不了——是 [phase1_llm_first_plan.md 硬约束 #6](phase1_llm_first_plan.md) 反对的"regex per test case"反模式。
3. 多步任务（看图 → 总结 → 发邮件）需要 executor 间切换，当前架构不天然支持。

### 12.2 决策

- **图片走单脑 + 工具化**：保持 `tool_using` 单脑 orchestrator，把视觉能力做成 `vision_analyze` action tool。模型自己决定何时调。
- **不引入 `subagents/` 通用框架**（Phase 3 ≠ 现在做）。`vision_analyze` 是 tool-backed specialist call，可以理解为"subagent 的最小形态"，但代码上不预抽象成 `registry.mjs` / `runner.mjs` / `specs/`。等真出现第二、第三个形状相似的 specialist 才从重复里抽接口。原则同 [phase1_llm_first_plan.md 硬约束](phase1_llm_first_plan.md) 与 CLAUDE.md "不要在任务需要之外引入抽象"。
- **provider 选择必须复用 `resolveProviderForTask("vision")`**，不写死 Qwen。Settings → Vision/Image 选什么就用什么（GPT-4o / Gemini / Qwen-VL / Claude vision / 豆包 / GLM-4V / Pixtral）。
- **chat provider 自身有视觉时的处理交给 runtime，不交给模型自我判断**：basd on `provider.capabilities.vision`，能看图就直接以 image block 塞进 user turn，不能看图就只给路径并暴露 `vision_analyze`。

### 12.4 拒绝项（明确不做）

- ❌ 不做 `src/service/subagents/{registry,runner,specs}.mjs` 通用框架。理由：当前只有 vision 一个 specialist，premature abstraction；将来若需要 mini-loop（裁剪局部 / OCR / 跨图比较），在 `vision_analyze` 工具内部加 for 循环就行，不需要框架。
- ❌ 不让模型自报"这不归我管"由 runtime 重启换 provider。理由：浪费一整个 round-trip + 模型自我认知不稳定。
- ❌ 不做 multi-agent supervisor 模式。理由：桌面单用户场景过度设计。
- ❌ 不写更聪明的 `VISION_ANALYSIS_RE`。理由：回到 regex per test case 反模式（违反硬约束 #6）。
- ❌ 不写死任何 vision 模型名。Qwen / GPT-4o / Gemini / Claude / 豆包 / GLM-4V / Pixtral 都只是 `resolveProviderForTask("vision")` 的实现细节，不进框架代码。

### 12.5 升级窗口（什么时候才做 subagent 框架）

满足全部三条才动：

1. 出现第二个形状相似的 specialist（比如 `file_deep_analysis`、`code_review` 这类需要"自己有工具集 + 多轮 loop + 限制 budget"的）；
2. 该 specialist 与 `vision_analyze` 内部至少 60% 接口重复；
3. 重复带来的实际维护痛点（不是预测痛点）已被 commit history 证明。

满足后再从两个真实实例里抽 `subagents/runner.mjs`，不要凭空设计。

> **Codex / Claude joint position:** 同意"vision tool 是 subagent 最小形态"的概念定位，但拒绝把它做成 generic subagent 平台。当前 PR 仅做 §12.3 的六步，对外是工具，对内留干净边界（`callVisionProvider` 私有 helper），可升级但不预升级。
