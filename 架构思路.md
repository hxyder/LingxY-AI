# Agent 框架快速判断、选工具、执行与回答优化指南

## 概述
要让 Agent 框架在不同场景、不同问题上都能**快速判断、精准选工具、高效执行并生成回答**，核心是在架构上做好**“分流”**和**“加速”**，避免把所有问题都丢给一个大模型从头推理。以下从四个关键环节展开。

---

## 1. 快速判断：分层路由，缩短决策路径
不做“万能型”推理，先把问题分派到最合适的处理单元。

- **轻量级意图分类**  
  在进入大模型前，用轻量分类器（精调小模型、BERT、甚至关键词匹配+正则）做第一层路由。将问题直接映射到预定义的场景标签（如“售后”、“技术咨询”、“查天气”），立刻缩小候选工具范围。

- **语义路由 + 工具检索**  
  若场景不固定，可将所有工具描述生成向量索引。收到问题后先用嵌入检索出 Top-K（如 3~5）个相关工具，再把这些工具定义填入提示词。大模型仅从少量候选中选择，判断速度极快，token 消耗也低。

- **多级路由兜底**  
  简单问题 → 规则/小模型直达答案；  
  中等复杂问题 → 走专用场景链（如“查询订单”链）；  
  只有真正开放的复杂问题才进入全功能 Agent 循环。

---

## 2. 选工具准且快：精炼提示 + 闭环校验

- **精简函数描述**  
  每个工具只保留核心的 `name`、`description` 和最关键参数，并在描述中写明**“何时该用、何时不该用”**，让模型迅速辨别。

- **强力 Few-shot 示例**  
  为每种典型场景提供“用户问题→所选工具→参数”的短示例，直接注入系统提示，比长篇说明能成倍提高选择准确率。

- **参数模板与预填**  
  能从上下文自动提取的参数（如用户位置、语言偏好），在框架层直接注入，不让模型去猜，减少错误和纠正回合。

- **工具选择后即时校验**  
  使用 Pydantic 或 JSON Schema 严格校验参数；参数不全或格式错误立刻要求重填（可附带简明错误提示），避免无效执行。

---

## 3. 执行提速：并行、缓存与降级

- **并行调用**  
  无依赖的工具调用（如同时查天气和查路况）直接并发执行。现代 LLM 的 parallel tool calling 可返回多个调用，框架需支持批量执行，总耗时仅取决于最慢的那个。

- **结果缓存**  
  对相同参数的工具调用做短期缓存（内存/Redis）。热点问题（如“今日金价”）可缓存数分钟，直接跳过执行。

- **预取与后台预热**  
  对高频场景，可在用户请求前基于当前上下文（如定位）预加载可能用到的工具结果，做到毫秒级响应。

- **流式感知执行**  
  不必等所有工具全返回才开始回答。哪个工具先就绪，可先将部分结果转为流式片段发给用户（如“已查询到您的订单状态…”），大幅提升体感速度。

- **优雅降级**  
  工具超时或不可用时，立刻切换备用工具、返回缓存旧值或引导用户补充信息，避免“卡死”。

---

## 4. 回答生成：结构化与流式兼备

- **模板化合成**  
  对结果格式固定的场景（如订单详情），用预置模板拼装答案，只让 LLM 润色综述部分，显著降低生成延迟。

- **流式输出**  
  将思考过程和工具执行状态实时反馈（如“正在调用订单系统…”），最终答案流式逐步呈现，让用户始终感受到进度。

- **引用与可解释性**  
  明确告知用户答案来自哪个工具，增强可信度，也便于出错时定位。

---

## 统一架构设计

把以上能力组织成一套可扩展的框架：

- **场景→配置映射**  
  每种场景对应一组工具白名单、一个路由规则、一套系统提示和 Few-shot 示例，可按场景目录随时新增，支持热更新。

- **动态工具集**  
  运行时根据路由结果只挂载该场景需要的工具，模型不被无关工具干扰，判断更快更准。

- **多 Agent 协作**  
  超复杂任务用规划器拆成子任务，分发到擅长子领域的子 Agent 或技能链；主 Agent 只负责编排和汇总。

---

## 一句话总结
把**“快速”**交给路由、并行、缓存和流式；把**“准确”**交给场景化提示、小范围工具筛选、强校验和可降级执行。两者结合，Agent 框架就能在面对不同场景和问题时，做到既通用又敏捷。


智能 Agent 框架架构设计文档
1. 设计目标
构建一个能够处理任意自然语言请求的智能 Agent 框架，具备以下能力：

多语言支持：中文、英文及其他常见语言。

自动任务分析：通过轻量级 LLM 调用判断请求的三个核心属性（实时性、确定性、认知负荷）。

动态路由：根据属性组合自动选择最合适的执行模式（直通、单步工具、计划模式）。

工具编排：支持单工具快速响应和多工具并行/串行编排。

流式输出：所有最终答案均以流式方式返回，优先保证首字延迟。

可扩展性：新增工具或场景无需修改核心路由逻辑。

3. 核心模块详解
3.1 属性判断层
职责：对用户输入进行快速、轻量的三维属性标记。

输入：用户原始文本（任意语言，如中英文混合）。

输出：JSON 格式的三个标签。

json
{
  "realtime": "高 | 中 | 低",
  "certainty": "高 | 中 | 低",
  "load": "高 | 中 | 低"
}
实现方式：调用 DeepSeek 模型，使用专门设计的 system prompt，要求仅输出上述 JSON，temperature=0，max_tokens≈30。此调用与主对话分离，独立进行一次轻量级推理（预计延迟 < 200ms）。

属性定义：

属性	高	中	低
实时性	需要最新/实时变化信息（天气、新闻、价格、渠道）	近期但非秒级（本周电影、本月活动）	稳定常识、历史、原理（数学、历史年份）
确定性	唯一固定答案（1+1=2）	有共识但存在例外（健康建议）	依赖观点/预测/未公开信息（股价预测）
认知负荷	多子问题、比较、多步推理（含“和、以及、顺便、分别”）	单一问句，需简单推理	极短句、问候、单一事实询问
3.2 路由决策层
职责：根据三维标签映射到具体的执行模式。

决策表（硬规则）：

load	realtime	certainty	模式	说明
低	低	高	A	直通模式：无工具，直接生成答案
低	高	任意	B	单步工具模式（通常为搜索）
低	任意	低	B	单步工具模式（搜索或知识库）
中	任意	任意	C	计划模式（至少两步）
高	任意	任意	C	计划模式（多步推理）
降级路径：当属性判断失败或超时（>2s）时，默认采用模式 B（单步工具），保证系统可用性。

3.3 执行模式层
模式 A：直通模式
触发条件：低负荷 + 低实时 + 高确定性。

执行流程：

直接调用主模型（DeepSeek）进行流式生成。
不提供任何工具定义。
流式返回答案给用户。
典型场景：问候、常识问答、数学计算、历史事实。

模式 B：单步工具模式
触发条件：低负荷 + (高实时 或 低确定性)。

执行流程：

调用主模型，附带 tools 定义（主要为搜索工具，也可包含天气、计算器等）。
模型返回 tool_calls（通常只有一个）或直接文本。
如果返回 tool_calls：
执行对应工具（异步，超时控制）。
将工具结果作为上下文，再次调用主模型（流式）生成最终答案。
如果模型直接返回文本（表示不需要工具），则直接流式输出。
优化：工具执行结果可缓存（相同问题短时间内复用）。

典型场景：天气查询、地点问询、实时新闻、价格查询。

模式 C：计划模式
触发条件：中/高负荷（即多步或复合问题）。

核心思想：Plan-and-Execute（先规划，后执行，再生成）。

执行流程：

规划：调用主模型（特殊 prompt）输出 JSON 格式的执行计划，包含步骤、工具、参数及依赖关系。
json
{
  "plan": [
    { "step_id": 1, "tool": "weather_query", "params": {"city": "北京", "date": "2026-04-27"}, "depends_on": [] },
    { "step_id": 2, "tool": "weather_query", "params": {"city": "北京", "date": "2026-04-28"}, "depends_on": [] },
    { "step_id": 3, "tool": "calculator", "params": {"expression": "{{2.temperature}} - {{1.temperature}}"}, "depends_on": [1, 2] }
  ]
}
调度：解析计划，构建依赖图，识别可并行执行的步骤（depends_on 为空或已满足）。
执行：
使用异步并发执行所有就绪步骤。
支持参数中的 {{step_id.field}} 占位符替换。
每个工具设置超时（如3秒），超时则标记失败但不中断整体。
聚合：收集所有步骤结果，整理为结构化文本。
生成：将原始问题 + 步骤结果输入主模型，流式生成最终答案。
约束：

计划步骤上限（如5步），超出则拒绝或降级为模式B。

整体执行超时（如20秒）。

典型场景：复合问题（“A和B的区别，并搜索最新评论”）、需要多个工具组合的任务。

3.4 工具层
工具抽象：

每个工具具有：名称、描述、参数 JSON Schema、异步执行函数。

工具注册中心：管理可用工具，支持动态注册。

核心工具示例：

工具名	描述	输入参数	输出
web_search	实时网络搜索	query: str	文本片段
weather_query	查询某地天气	city: str, date: str	{temp, condition}
calculator	数学表达式计算	expression: str	数字
code_executor	安全执行代码	code: str, language: str	执行结果
工具执行特点：

异步非阻塞。

支持超时和重试。

结果标准化为 dict 或 str，便于后续注入 LLM。

3.5 输出层
协议：Server-Sent Events (SSE) 或 WebSocket。

首字优化：模式 A/B 在模型返回第一个 token 时立即转发；模式 C 可在规划阶段先发送“正在处理……”占位消息，避免用户等待无响应。

流式格式：每个 chunk 为一段文本，前端逐字显示。

中断支持：允许用户取消请求，停止后续工具调用和生成。

4. 数据流示例
示例 1：简单常识 → 模式 A
text
用户：什么是相对论？
→ 属性判断：{load:低, realtime:低, certainty:高}
→ 路由：模式 A
→ 直通调用主模型，流式输出答案。
示例 2：实时单步 → 模式 B
text
用户：美国哪里可以买到中国的白酒？
→ 属性判断：{load:低, realtime:高, certainty:低}
→ 路由：模式 B
→ 主模型调用（带搜索工具）→ 返回 search 调用
→ 执行搜索 → 返回结果
→ 再次调用主模型 → 流式输出：“在 Total Wine、亚马逊等有售...”
示例 3：复合多步 → 模式 C
text
用户：北京今天和明天的天气温差是多少？顺便查一下美国哪里买茅台。
→ 属性判断：{load:高, realtime:高, certainty:中}
→ 路由：模式 C
→ 规划阶段输出计划：
  步骤1: weather(北京,今天)
  步骤2: weather(北京,明天)
  步骤3: search(美国茅台购买渠道)
  步骤4: calculator(明天温度-今天温度) 依赖步骤1,2
→ 并行执行步骤1,2,3 → 步骤4等待前两者完成
→ 聚合结果 → 主模型生成最终答案：“今天20℃，明天28℃，温差8℃。购买渠道：...”
→ 流式输出。
5. 非功能性设计
5.1 超时与降级
组件	超时时间	降级动作
属性判断	2秒	默认模式 B（单步工具）
模式 A 主模型	5秒	返回固定兜底消息
模式 B 第一次调用	3秒	降级为模式 A 直接回答
模式 B 工具执行	3秒/工具	跳过该工具，返回部分结果
模式 C 整体	20秒	终止计划，返回已有的部分结果
模式 C 计划生成	5秒	降级为模式 B
5.2 流式首字目标
模式 A：< 500ms

模式 B：< 800ms（含一次工具调用）

模式 C：< 1.2s（先返回占位消息，后逐步输出）

5.3 可观测性
日志：记录每个请求的 request_id、属性判断结果、路由模式、工具调用细节、各阶段耗时。

指标：属性判断成功率、模式分布、工具调用成功率、平均首字延迟。

追踪：支持 OpenTelemetry 分布式追踪。

5.4 扩展性
新增工具：只需注册到工具中心，并在系统提示中更新工具列表，无需修改路由逻辑。

新增场景：无需修改，因三维标签已覆盖所有任务本质。

多语言：属性判断使用 DeepSeek 模型原生支持多语言，无需额外配置。

6. 部署与运维建议
API 服务：FastAPI + Uvicorn，异步 workers。

缓存：Redis 缓存属性判断结果（相同问题 5 分钟内复用）。

限流：基于用户/API Key 的令牌桶限流。

健康检查：提供 /health 端点，检查 DeepSeek API 连通性及工具服务状态。

配置管理：使用环境变量或配置中心管理超时、工具开关、模型名称等。

7. 总结
本框架通过 分离属性判断与主对话 + 三维标签路由 + 三种执行模式，实现了对任意自然语言请求的自动适配。它避免了硬编码场景分类，具备良好的泛化能力和扩展性，同时通过流式输出和并行执行确保了低延迟的用户体验。该设计可直接指导工程实现，也可作为未来更复杂 Agent 系统的基础架构
