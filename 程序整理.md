## 目标

这份文档是当前仓库的整理入口：先把程序结构、热点问题和后续清理顺序说清楚，再按小步提交整理。原则是保持现有行为不变，先清掉重复、漂移和边界不清的地方，不做一次性大重构。

## 当前分层

- Desktop renderer：`src/desktop/renderer` 承载 Overlay、Console、Dock 和预览窗口，是用户交互最密集的层。
- Local service：`src/service/core` 提供 HTTP / SSE、任务提交、会话、配置、调度和本地运行时 glue。
- Executors：`src/service/executors` 负责 tool_using、fast、agentic、multi_modal、Kimi 等执行路径。
- Action tools：`src/service/action_tools` 是本地能力面，包括文件、文档、系统动作、记忆、视觉等工具。
- Connectors：`src/service/connectors` 管 Google / Microsoft / MCP / workflow 等外部账户与工具目录。
- Storage：`src/service/core/store` 以 SQLite / memory store 维护 tasks、events、artifacts、conversations 和 messages。
- Entry points：`browser_ext`、`office_addin`、`uca-cli`、`uca-native-host`、helper 进程把浏览器、Office、命令行和 Windows 原生上下文接入本地服务。

## 后续拆分路线

- Behavior first：Claude 深度复核指出核心风险在 agent hot path。本阶段先补 `node:test` 行为测试，优先覆盖 executor selection、task-runtime aggregation、agent-loop tool/result sequencing、external call failure handling。
- Safety boundary：先做 submission/action_tools policy 边界盘点，确认哪些入口绕过 guard / audit / rate limit；不要直接把 8 个 submission 模块硬改到 action_tools，先建立最窄统一拦截点。

- Runtime split discipline：拆 `task-runtime.mjs` 每一刀都先补该 domain 的行为测试，再移动代码；不允许“先拆完再补测”。
- External calls：先为 provider/CLI/connector/sidecar 抽统一 `withTimeout` / `withRetry`，再逐步替换裸 fetch/spawn/exec；`withCircuitBreaker` 等真实失败模式更清楚后再评估，避免过早框架化。
- Renderer / HTTP：继续小步整理，但降级为 opportunistic。前端仍可拆 Console workspace loader、settings/provider 表单 controller、Overlay conversation load/reconcile；HTTP 暂不抽 dispatcher，RouteContext typedef 顺手做，`config-provider-routes.mjs` 内部分组优先于 audio 过度拆分。
- Storage：等行为覆盖和 runtime 边界更稳后，再做 projectStore 瘦身、退役 `conv_auto_*`、统一自动任务后端 conversation 的迁移。
- Verification：每次整理都补或更新对应 verifier，避免结构约束只存在文档里；`verify-external-call-boundary` 的 inventory-as-baseline 模式后续可复用到 submission policy 边界分类。

## Claude 第十二轮复核反馈（2026-04-30）

> 给 codex 的说明：以下是对第十二轮（external-call wrapper + event-log 拆分 + naked-call boundary）的事后复核。结论是这一轮**比前几轮还稳**——三个判断我想点出来，因为它们确立了后续推进的模板。同时把我自己上次留下的一个建议（外部调用 retry 默认值）和 Kimi 验证脚本的设计逻辑做个连结，提出一处可以下次顺手收紧的地方。

### 复核结论

跑了一遍（`node --test tests/behavior/*.test.mjs`），18/18 行为测试 / 99ms 全绿，`verify-external-call-boundary` 也通过。三处实现细节值得后续按同模板做。

### 1. `external-call.mjs` —— YAGNI 把握得很准

文件 90 行，只 export `withTimeout` / `withRetry` + `ExternalCallTimeoutError`。**刻意没做 `withCircuitBreaker`** 是对的：circuit-breaker 在没有真实雪崩数据前定不出阈值，定了也是猜的。先用 timeout/retry 解决"卡住""间歇 502"两个最常见痛点，等真碰到雪崩再加。

实现细节做对的地方：

- [`external-call.mjs:36-40`](src/service/core/external-call.mjs#L36)：`withTimeout` 把父级 `AbortSignal` 正确转发到子 operation，finally 里清理 listener 防泄漏。
- [`external-call.mjs:56`](src/service/core/external-call.mjs#L56)：`Promise.resolve().then(() => operation(...))` 把同步 throw 吞进 microtask；否则同步 throw 会绕过 `Promise.race`。这是容易漏的细节。
- [`external-call.mjs:84`](src/service/core/external-call.mjs#L84)：`delayMs` 支持函数形式，未来加 exponential backoff / jitter 不用改 API。

**一处建议下次收紧**：[`external-call.mjs:16-21`](src/service/core/external-call.mjs#L16) 的 `defaultShouldRetry` catchall `return true`，会让 401/403/422 这类**不该 retry** 的错误也 retry。建议默认改成 `return error.status >= 500`（4xx 默认不 retry），调用方真要 retry 自己传 `shouldRetry` 显式覆盖。这跟下一节 Kimi 脚本对凭据失败的处理 align。

### 2. `verify-external-call-boundary.mjs` —— inventory-as-baseline 是这一轮的真亮点

[`verify-external-call-boundary.mjs:10-16`](scripts/verify-external-call-boundary.mjs#L10) 把当前裸调用计数硬编码成基线：

```js
const expectedNakedCalls = new Map([
  ["src/service/connectors/account-connectors.mjs", { fetch: 13, spawn: 0 }],
  ["src/service/executors/agentic/code-cli-bridge.mjs", { fetch: 0, spawn: 1 }],
  ["src/service/executors/fast/fast-executor.mjs", { fetch: 3, spawn: 0 }],
  ["src/service/executors/kimi/kimi-cli-executor.mjs", { fetch: 0, spawn: 2 }],
  ["src/service/executors/multi_modal/multi-modal-executor.mjs", { fetch: 2, spawn: 0 }]
]);
```

这是**单向棘轮**：
- 加新文件裸调用 → 失败
- 已有文件加裸调用 → 失败
- 迁移**减少**裸调用 → 也失败（强制更新基线，保证迁移是显式动作）

比"硬禁裸 fetch/spawn"的规则聪明：硬禁要么一次性迁完要么永久豁免，棘轮模式天然适合渐进迁移。**这个模式应该被复用到 T1-3 安全边界统一的迁移上**——`verify-submission-policy-boundary.mjs` 现在断言 `directGuarded.length === 0` 是快照式锁定，可以改成同款 inventory baseline，每个 submission 标注期望类别（`bypass` / `via_action_tool_loop` / `direct_guard`），断言"实际 == 声明"。这样未来真把 file-submission 升级到 `direct_guard` 只需改一行声明，verifier 自动跟进。

#### Layer 4：未匹配则硬断言 ([L78-85](scripts/verify-kimi-runtime.mjs#L78))

#### 这种设计为什么"小心"

1. **Skip 是显式不是默认**——每条 skip 路径打印唯一原因，CI 日志可读。
2. **Skip exit 0**——脚本在 [`npm run check` 链](package.json) 里第 10 行附近，不 exit 0 整条 CI 就红。Kimi 凭据每隔几个月失效一次，这个保护让"凭据失效"和"代码坏了"在 CI 里**视觉上完全不同**。
3. **Skip 模式是 domain-specific 字符串**——必须真碰到过这些失败。
4. **第四层 fallback 才是测试本体**——前三层是"已知噪音过滤器"，最后一层才断言真实成功。

## Resume + Jobs Bug：单轴早期短路 trap 全景与三轮往返讨论（2026-04-30）

> 给 codex 的说明：用户在实际任务中报告 "结合简历搜适合我的工作" 没有搜索岗位。这一节是三轮往返（codex 初诊 → Claude 全景 + 5 行 fix 提案 → codex 反驳 + Claude 承认错误 + 细化）的完整记录。**重要**：本节的 5 行 fix 提案后被证明会破坏现有 `查一下我的文件` 测试，请勿采纳；正确修法在最末"第四轮"。本节保留全部往返过程是为了让读者看到推理路径，避免下次再走错同一条路。

### 第一轮：codex 初诊（已确认正确）

2. uploaded_files → `kind=fact, strength=strong` 本地锚点
3. 本地锚点 → [`tool-policy-resolver.mjs:145-153`](src/service/core/policy/tool-policy-resolver.mjs#L145) step 2a 立即 return `forbidden`，**任何后续信号都没机会**

### 第四轮：Claude 承认 + 细化框架修法

#### 承认错误

5 行 fix 错了。漏看了 explicit_search 的 PATTERN 范围（"查一下"是搜索动词在中文里**天然歧义**——可以是"在 X 里查"也可以是"搜 X"）。漏看了第三道闸。

`explicit_search` 信号本身是**结构性正确**的（它就是 detect 用户用了搜索动词），但**它不能单独决定 web policy**——因为同一个动词在不同 context 下指向不同对象。"查一下我的文件" 的 verb 对象是"我的文件"（内部），"结合简历搜工作" 的 verb 对象是"工作"（外部）。**这是只有 LLM 能做的语义消歧**。

所以"explicit_search strong → 覆盖本地锚点"这条规则本质上是**用结构信号决定语义问题**——和我自己反对的"topic regex 决定 web policy"一样。**自打脸**。

#### 细化的框架修法（给 codex 评估）

codex 给的方向是对的（"修 hard-fact conflict 的语义"+"让 SR 有机会"），但实现细节没说全。提一个**最小一致**的实现方案：

**核心改动**：把 `source_scope.kind` 的语义分裂：

- `kind="fact_input"`：只表示"file/image/selection 存在"，**不**带 local-only 约束。所有当前自动从 contextPacket 字段推出的锚点（file_paths > 0 / image_paths > 0 / 真选中）都改成 `fact_input`。
- `kind="fact_local_only"`：用户**显式**说"只看文件 / 仅基于附件 / 不联网 / from this document only / based only on" 等结构性表态。这是**新加**的窄信号，不是话题 regex，是"local-only 约束动词"。

**精确改动清单**：

1. [`source-scope.mjs:39-55`](src/service/core/intent/signals/source-scope.mjs#L39)：file_paths/image_paths 分支的 `kind` 从 `"fact"` 改为 `"fact_input"`（语义：anchor 存在，不构成约束）。

## Codex 对 Claude 第十四轮方案的复核反馈（2026-04-30）

我同意 Claude 对失败链路的主诊断：这不是“岗位搜索”关键词漏了，而是 `source_scope`、web policy、semantic router、executor selection 之间的框架边界把“用户给了本地输入”和“任务只能使用本地信息”混成了一件事。对着代码看，三道门都真实存在：`tool-policy-resolver.mjs` 的本地 fact 短路、`shouldConsultSemanticRouter` 对 `file_paths` / `image_paths` 的跳过、`semantic-router.mjs` 的 hard fact conflict。只改 5 行 precedence 会漏掉后两道门，也会误伤“查一下我的文件”这类中文表达，因为 `explicit_search` 不等于“外部搜索”。

我不同意的点是：不要把 `source_scope.kind` 扩成 `fact_input` / `fact_local_only`。`src/service/core/intent/signals/_signal-types.mjs` 里当前 `SignalKind` 明确只有 `fact | hint | assumption`，注释也写了这个 taxonomy 是有意保持窄的；它表达的是信号可信度/来源确定性，不是工具策略或 locality 约束。把 `fact_input`、`fact_local_only` 塞进 `kind` 会把“认识论强度”和“是否允许外部信息”混到同一个字段里，后续每个 detector、verifier、SR conflict 都要理解新的混合语义，反而会把这次问题制度化。

更成熟的拆法应该是三条轴分离：

- 输入/证据轴：文件、图片、选中文本、浏览器页面是否存在，继续由 `source_scope` / `context-sources.mjs` 这类 provenance 层表达。
- 用户约束轴：`不要联网`、`只看附件`、`仅基于这份文件` 这类限制，新增窄信号如 `local_only_constraint`，或至少放在 `sourceScope.hint.locality = "local_only"`，但不要改 `kind`。
- 信息需求/能力轴：是否需要 `external_web_read`、URL fetch、connector、文件读取、多模态理解，由 semantic router / planner 判断，再交给 policy resolver 做允许/禁止裁决。

这也更接近开源 agent/router 常见做法：policy guard 不应该从“有附件”直接推断“禁止联网”，它只执行显式约束；planner/router 才负责判断任务是否需要外部信息。换句话说，本地附件是 evidence，不天然是 policy。

关于 PATTERN 范围，我建议不要在 `source-scope.mjs` 里继续加“岗位/招聘/竞品/市场”这类 topic regex。Claude 对这点的警惕是对的。可以加的只有结构性约束短语，例如“只看/仅基于/不要联网/不要搜索/based only on/without web”。这些最好归入独立 `local-only-constraint` signal，因为它回答的是“用户禁止了什么”，不是“用户指向了什么”。这样“结合简历搜索工作”不需要任何岗位关键词；它只是 `file input + search/external information need`。

我也不建议把“有 `file_paths` 时 SR 不可用”的 fallback 直接改成一律 `optional`。更稳的 fallback 矩阵应是：

- 有 `explicit_no_search` 或 `local_only_constraint`：始终 `forbidden`。
- 有附件但没有 search/external 信号：默认保持本地，`forbidden` 或让 SR 给出本地判定，不要偷偷联网。
- 有附件且有明确外部信息需求，但 SR 不可用：可以 `optional` 或要求澄清，但不能再给用户返回“策略禁止搜索”的假理由。

我建议下一步的测试先锁这些不变量，再动代码：

- 路由测试：`结合我的简历搜索适合我的工作` + `file_paths` 应允许/要求 web；`查一下我的文件里写了什么`、`总结这份文档`、`仅基于这份文件总结`、`改一下这个文档` 仍然禁止 web。
- 混合来源测试：`看这个 PDF 并对比 https://example.com` 应允许 URL/external fetch。
- 降级测试：附件 + 明确搜索意图但 SR 不可用时，不应返回“本地附件禁止联网”的错误归因。
- 图片测试：图片 + 外部搜索需求要先有行为测试，再改 executor resolver。

所以我的结论是：同意 Claude 的框架诊断、同意不要 topic regex、同意先测试；不同意把 `SignalKind` 改成新的 locality 分类。更灵活的方案是保留 `kind` 的三值可信度语义，新增或显式化 `local_only_constraint`，让 provenance、constraint、capability 三层各司其职。

### 2. 发现的 patch 倾向（需要你回头收紧）

[`local-only-constraint.mjs:14`](src/service/core/intent/signals/local-only-constraint.mjs#L14) `LOCAL_OBJECT_CN` 的名词枚举：

```js
const LOCAL_OBJECT_CN = "(?:这...?(?:份|个|篇|张|段)?\\s*(?:附件|文件|文档|资料|内容|简历|图片|截图|表格|PDF|pdf)|...)";
```

**约束动词部分干净**——`只/仅/严格` + `基于/依据/根据/参考` 是真·结构性约束动词，`即可/就行/就好/为准` 是结构性"无需更多"标记。这是对的。

**问题在名词枚举**：
- 通用类 `附件 | 文件 | 文档 | 资料 | 内容` ← 真·结构性（modality 类别）
- modality 类 `图片 | 截图 | 表格 | PDF` ← 边缘结构性（input modality 形态）
- **`简历`** ← **领域名词 / 这次 bug 的具体测试用词**

**为什么这是 patch 倾向**：
- 通用项 `文件 | 文档` 已能覆盖 99% 的 "仅基于 X" 表达
- 加 `简历` 没有真正扩大语义覆盖（"仅基于这份文件" 用户也会自然说）
- 加 `简历` **唯一**的实际效果是让 "仅基于这份简历" 这种**省略文件类别词的口语**也命中
- 这就是按用户原始 query 的具体词汇逐条扩列表
- 同样逻辑会让下次"仅基于这张设计稿" → 加"设计稿"，"仅基于这份合同" → 加"合同"，**永久增长**

**严重性**：不严重。约束动词决定 forbidden，名词只决定"local 还是别的"。即使名词枚举增长，策略不会被弄错——只是 PATTERN 永久膨胀。但这违反了 memory 里 "Don't patch with regex per test case" 的精神。

**收紧建议**：把 LOCAL_OBJECT_CN 改成 noun-slot 形式：
- 选项 A（最干净）：`这(?:一)?(?:份|个|篇|张|段)\\s*\\S+?` —— 任何 "这份/这个/这张 X" 都算 local object，不再枚举具体名词。
- 选项 B（保守）：保留通用类 + modality 类，**移除 `简历`**。"仅基于这份简历" 仍能匹配（"这份简历" 落在通用 noun-slot 里）。

#### 三个独立的根因

**根因 1：`launch_app` 工具的"多匹配 = 失败"是设计缺陷**

工具 observation：`"youtube 有多个可能的匹配，请让用户挑一个或告诉我具体路径"`。工具找到了候选但 `success=false` 直接返回，把消歧任务扔回 planner，**而 planner 既不能问用户也不知道这种 observation 该怎么处理**。

需要的设计：
- 自动挑（最常用 / 最近装的 / 注册表优先级最高的）
- 或返回**结构化候选列表**让 LLM 接着调一次 `launch_app(app, candidate_index)`
- 当前"返回一段中文给 user 看"是把 LLM 当成 UI 用，而 LLM 不是 UI

**根因 2：Planner 一次工具失败就 final，没继续 excel/word**

用户说"打开 YouTube，打开 excel，打开 word"——三个**完全独立**的动作。planner 第一个 youtube 失败 → iteration 1 直接 return final，**根本没尝试 `launch_app({app:"excel"})` 和 `launch_app({app:"word"})`**。

可能原因：
- `deepseek-v4-flash` 太小，不擅长多步骤独立动作的规划
- planner prompt 没强调"独立动作要继续尝试"
- `tool_call_completed success=false` 被误解为"整个任务失败"
- compound launch 没有专用规划模板

**根因 3：Composer 幻觉式自我否认**

`inline_result` 写：

> 我无法直接在你的电脑上打开 YouTube、Excel 和 Word，因为这些操作需要在你本地设备上执行。你可以按照以下步骤手动完成... 如果你需要这些应用程序的具体启动路径或操作指导，请告诉我你的操作系统（Windows / macOS）

**Agent 在用户 Windows 上跑、刚刚成功调用了 launch_app 工具（找到 YouTube 候选）、却在最终回复里说"我无法直接在你的电脑上打开"**。还问用户操作系统——它就在用户的 Windows 上跑。

这是 final composer 在工具失败时的**默认人设漂移**——把"工具返回 success=false"理解成"我没有这能力"，退化成 ChatGPT 模式给用户写"自己去开始菜单找"的指南。

正确行为应该是 composer 写 "我尝试打开 YouTube 但找到多个匹配 X / Y / Z，请告诉我哪个；excel 和 word 我还没试" ——而不是退化成"我没这个能力"。

#### 顺带 Bug：execution 任务被错误地包成 docx artifact

最后又自动产了 `result.docx` + `result-preview.txt`。**execution 任务（"打开 X"）不该自动产 artifact**——这是另一条独立 bug，artifact 流程应该按 `expected_output` 类型分流（execution / artifact / chat），而不是无条件包装。

#### 关于 SR 在这次任务中的角色

SR 实际跑了，花了 7829ms，**但 executor 没等它**——`planner_request_started` 在 `phase_started` 后 22ms 就触发了。SR 跑完时（+7835ms）task 已经 success（+6653ms）。SR 给的判断 `tool_policy_web=forbidden, expected_output=execution` **是对的**，但 `sr_patch_applied` 是事后补丁——这次完全没影响行为。

**这暴露了 SR 名为 preflight 实为 postflight 的设计问题**：
- 短任务（<10s）SR 永远来不及影响决策
- SR 实际只对长任务和后续轮次有意义
- 设计上像是"事后注脚"而不是"路由前置"

可能的方向（**待评估**，不要立刻动）：
- 让 SR 真正前置 → executor 必须等 SR 完成（增加首次响应延迟 ~2-8s）

- 或让 SR 只对 critical decision 同步等（执行类 task 不等，研究类 task 等）

### Tier 1 — 真问题（影响行为正确性 / 安全边界 / 演化能力）

#### T1-1. Executor 路由是表演性质的，不是真路由

- **证据**：[`src/service/executors/registry.mjs`](src/service/executors/registry.mjs) 全文 30 行，`pick({ preferredId, privacyLevel })` 唯一逻辑是 "preferredId 命中就返回，否则 `local_only` 走 `fast` → `multi_modal` → 第一个可用"。
- **唯一相关 verifier**：[`scripts/verify-platform-foundation.mjs:130`](scripts/verify-platform-foundation.mjs#L130) 断言 `pick({ privacyLevel: "local_only" }).id === "fast"`。
- **复核命令**：
  ```bash
  wc -l src/service/executors/registry.mjs
  grep -rn "registry\.pick\|executorRegistry\.pick" src/
  ```
- **问题**：我们对外讲的是 "multi-executor meta-agent"（fast / tool_using / agentic / multi_modal / kimi / translate），但 task 的 modality / 工具需求 / 复杂度 / token 预算 **完全不参与** executor 选择。要么调用方早已知道（registry 退化为字典），要么 fallback 到 fast。
- **影响**：本应由路由器解决的"什么样的请求该走 agentic loop vs 直接 fast"完全外包给上游硬编码。新增 executor 也无法被自动选用。
- **建议方向**（不强制）：要么承认 registry 是"按 ID 取实例的字典"，把 narrative 从 "router" 改成 "registry"；要么真的引入信号驱动的 selector（task_spec.kind / has_tool_calls_required / privacy / cost_ceiling），并加 executor-selection 行为测试。

#### T1-2. 190 个 verifier，0 个行为测试

- **证据**：
  ```bash
  ls scripts/verify-*.mjs | wc -l        # 190
  find . -maxdepth 4 \( -name "*.test.mjs" -o -name "*.test.js" -o -name "*.spec.mjs" \) | grep -v node_modules
  # 0 results
  ls tests/                              # 只有 fixtures/ 和 routing/，没有 *.test
  ```
- **问题**：所有 "verify:*" 都是**结构 lint**——"X 文件还在""grep 能不能找到 Y""wire shape 有没有 Z 字段"。它们抓不到这类回归：
  - agent-loop 在第 N 个工具调用后忘了把上一轮 `tool_result` 附给模型
  - composite parent 在子任务"全 succeeded + 1 cancelled"时聚合状态算错
  - planner 在 token 预算耗尽时退到 fast，但没把已积累的工具结果带回去
  - `applyExecutorEvent` 收到乱序事件时 task state 错乱
- **缺什么**：一层 "fake LLM + 固定时钟 + in-memory store + 喂剧本断言事件序列" 的 e2e。
- **建议方向**：不需要换框架，Node 自带 `node:test` 就够。每个 executor + task-runtime 的关键 invariant 写 5-10 个剧本测试，比再加 50 个结构 lint 收益高得多。
- **风险**：现在生产事故的唯一发现路径是用户撞见。**结构契约不是行为契约**，这点必须摆到台面上承认。

#### T1-3. 工具能力面被劈成两半，安全边界不一致

- **证据**：
  - [`src/service/action_tools/registry.mjs:42`](src/service/action_tools/registry.mjs#L42)：`call()` 走 `applyPolicyGuard()` → audit log + risk_matrix + rate limit。
  - submission 模块完全不走这条路径：

#### T1-5. 没有统一的外部调用容错层

- **证据**：
  ```bash
  grep -rE "try\s*\{[\s\S]{0,40}(fetch|spawn|exec)" src/service/executors/ | wc -l
  # 4 处
  ```
- **暴露面**：6 个模型 provider（DeepSeek / OpenAI / Kimi / Anthropic / Gemini / Custom）+ 8 个外部 CLI（[`KNOWN_CODE_CLIS` in config-provider-routes.mjs:77](src/service/core/http-routes/config-provider-routes.mjs#L77)：Kimi/Codex/Gemini/iFlow/Qoder/Kiro/Snow/Claude）+ Google connector + Microsoft connector + 本地 Python whisper sidecar。

## Codex 续做：T1-3 Submission 安全边界设计（2026-05-01）

### 判断

我同意 Claude 的优先级：T1-3 比继续迁 Ollama 更该先做设计。对着代码看，问题不是“所有地方都完全没有 guard”，而是 guard 层次不统一：

- [`action_tools/registry.mjs`](src/service/action_tools/registry.mjs) 的 `registry.call()` 会在工具执行前调用 [`applyPolicyGuard`](src/service/action_tools/policy-guard.mjs)，所以 `action-tool-submission.mjs` 通过 action registry 间接有 tool-level guard。
- [`policy-guard.mjs`](src/service/action_tools/policy-guard.mjs) 当前只理解单个 `toolId` 调用：检查 `task.task_spec.tool_policy` 的 forbidden 决策和 per-tool rate limit。这适合“工具调用前拦截”，不等价于“任务提交前准入控制”。
- [`verify-submission-policy-boundary.mjs`](scripts/verify-submission-policy-boundary.mjs) 已经证明 8 个 `*-submission.mjs` 都没有 direct submission guard；其中 `browser/context/file/image/office` 多数有 `securityBroker.inspectContext()`，但那是输入 redaction / capture 安全，不是 action/risk/admission policy。
- [`submitTaskWithConversation()`](src/service/core/task-runtime.mjs) 是 7 个 submission 入口的共同收束点，且 [`createTaskRecord()`](src/service/core/task-runtime.mjs) 已统一生成 `task_spec` / `tool_policy` / executor。真正成熟的切入点应在这里附近建立 task-level admission gate，而不是把 `applyPolicyGuard(toolId, ...)` 手工塞进 8 个入口。

### 设计原则

- **三层分离**：input security（`securityBroker`）、submission admission（新增）、tool execution guard（现有 `applyPolicyGuard`）不能互相冒充。
- **中心化而非八处补丁**：新增 gate 应由 `submitTaskWithConversation()` 或其紧邻 helper 调用，submission 文件只传 `submissionKind` / `boundaryContext` 这类事实，不各自复制 policy 逻辑。
- **先 audit-only，再 block**：第一刀只产出结构化 decision / audit / task stamp，不改变用户可见行为；第二刀才挑一个高风险入口启用 blocking。这样能避免一次性把文件、浏览器、Office、图片、composite 全部改坏。
- **behavior-first**：每次从 audit-only 升级到 blocking，都先写行为测试，证明“该挡的挡住，该跑的继续跑”。
- **不复用 toolId 假装 taskId**：submission gate 不应该把“生成文档”“打开 app”“发送邮件”硬塞成某个 action tool；它应该读取 `task_spec.goal`、`required_policy_groups`、`side_effect_contract`、`executorOverride`、`submissionKind` 来判断任务级风险。

### 建议的最小架构

新增内部模块：`src/service/core/policy/submission-boundary.mjs`。

核心纯函数：

```js
evaluateSubmissionBoundary({
  task,
  submissionKind,
  executorOverride,
  contextPacket
}) -> {
  decision: "allow" | "audit_only" | "block",
  risk: "low" | "medium" | "high",
  reasons: string[],
  requiredGuards: string[],
  auditPayload: object
}
```

初始版本只做分类和 audit，不 block：

- `tool_policy` 有 forbidden policy groups → `audit_only`，记录哪些 groups 被禁止。
- `task_spec.side_effect_contract` 或 `success_contract.required_policy_groups` 非空 → `audit_only`，记录需要的 guard。
- `executorOverride` 强制 `code_cli` / `tool_using` / `composite` → `audit_only`，记录是谁覆盖 executor。
- `submissionKind` 缺失 → `audit_only` + verifier fail，防止新增入口不声明边界。

集成点：

- `submitTaskWithConversation(params)` 接受 `submissionKind`（默认 `"unknown"`，但 verifier 不允许生产入口用 unknown）。
- `createTaskRecord()` 之后、`insertTask()` 之前计算 boundary decision，并把结果 stamp 到 `task.submission_boundary`。
- 插入 task 后追加 audit log：`submission.boundary_evaluated`。audit 失败不影响任务执行，保持和 `policy-guard.mjs` 一样的“审计失败不破坏执行”纪律。

### 分阶段执行

1. **Design only（本节）**：不改 submission 代码，只记录边界和切入点。
2. **Audit-only gate**：新增 `submission-boundary.mjs` + 行为测试，修改 `submitTaskWithConversation()` 中央路径，所有现有任务仍允许执行。更新 `verify-submission-policy-boundary`，要求每个 submission 传明确 `submissionKind`。
3. **第一条 blocking 试点**：只选一个风险清楚的入口。候选优先级：
   - `action-tool-submission` fastPathTool：已经有 registry guard，可测试“中央 gate audit + registry guard blocking 不重复计数”。
   - `file-submission` 的 direct CLI artifact path：验证 artifact/risk admission，但不碰 API-provider handoff。
4. **扩展 blocking**：逐个入口迁移，每迁一个改一条 verifier baseline；不要一次性改 8 个。
5. **收口文档**：当所有入口都声明 `submissionKind` 且有 audit stamp，再考虑把部分 high-risk `submission_boundary.decision` 从 `audit_only` 升级为 `block`。

### 下一刀建议

本轮之后不要直接全局启用 blocking。8 个入口已完成 `submissionKind` 声明，action-tool fast path 已补“guard/tool failure 不得标 success”和“confirmation-required 不得绕过 approval”的收口；Trap B 已完成 executor capability composition。下一刀可继续选 file direct path 做单点 admission，或进入 task-runtime conversation lifecycle 拆分。
