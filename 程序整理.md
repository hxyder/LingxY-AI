# 程序整理

## 目标

这份文档是当前仓库的整理入口：先把程序结构、热点问题和后续清理顺序说清楚，再按小步提交整理。原则是保持现有行为不变，先清掉重复、漂移和边界不清的地方，不做一次性大重构。

## 当前分层

- Desktop renderer：`src/desktop/renderer` 承载 Overlay、Console、Dock 和预览窗口，是用户交互最密集的层。
- Local service：`src/service/core` 提供 HTTP / SSE、任务提交、会话、配置、调度和本地运行时 glue。
- Executors：`src/service/executors` 负责 tool_using、fast、agentic、multi_modal、Kimi 等执行路径。
- Action tools：`src/service/action_tools` 是本地能力面，包括文件、文档、系统动作、记忆、视觉等工具。
- Connectors：`src/service/connectors` 管 Google / Microsoft / MCP / workflow 等外部账户与工具目录。
- Storage：`src/service/core/store` 以 SQLite / memory store 维护 tasks、events、artifacts、conversations 和 messages。
- Entry points：`browser_ext`、`office_addin`、`uca-cli`、`uca-native-host`、helper 进程把浏览器、Office、命令行和 Windows 原生上下文接入本地服务。

## 已发现热点

- `src/desktop/renderer/console.js` 和 `src/desktop/renderer/overlay.js` 仍承担 DOM refs、状态、网络、渲染、事件绑定、会话缓存等多种职责，文件过大且有重复 helper。
- `src/service/core/http-server.mjs` 已拆出 task/task events、notes/conversation/project、config/providers、AI status/MCP runtime controls、scheduler/templates、audio/Echo/note transcription、preview/file、location/browser context/overlay handoff/page explain、Office static/setup、runtime admin 等 route 模块；server 文件目前主要保留 CORS、routeGroups、404/500 和 server start/stop，已经基本成为 HTTP app shell。
- `projectStore` 与 SQL `conversations` 的边界仍混杂：前端本地 store 已逐步退为 cache，但 `turns`、项目指针和后台任务结果仍在同一对象里。
- `conv_auto_*` 仍是前端合成的自动任务会话，和后端 conversation source of truth 不完全一致，是后续存储清理重点。
- 验证脚本和实现有少量漂移，整理时必须把 verifier 当作结构契约同步维护。

## 后续拆分路线

- Behavior first：Claude 深度复核指出核心风险在 agent hot path。本阶段先补 `node:test` 行为测试，优先覆盖 executor selection、task-runtime aggregation、agent-loop tool/result sequencing、external call failure handling。
- Safety boundary：先做 submission/action_tools policy 边界盘点，确认哪些入口绕过 guard / audit / rate limit；不要直接把 8 个 submission 模块硬改到 action_tools，先建立最窄统一拦截点。
- Agent runtime：行为测试站稳后，再拆 `task-runtime.mjs`：composite aggregation 已先拆出；后续继续按 conversation lifecycle、task lifecycle、event/log writer、active execution registry 分模块，外壳保留兼容 export。
- Runtime split discipline：拆 `task-runtime.mjs` 每一刀都先补该 domain 的行为测试，再移动代码；不允许“先拆完再补测”。
- External calls：先为 provider/CLI/connector/sidecar 抽统一 `withTimeout` / `withRetry`，再逐步替换裸 fetch/spawn/exec；`withCircuitBreaker` 等真实失败模式更清楚后再评估，避免过早框架化。
- Renderer / HTTP：继续小步整理，但降级为 opportunistic。前端仍可拆 Console workspace loader、settings/provider 表单 controller、Overlay conversation load/reconcile；HTTP 暂不抽 dispatcher，RouteContext typedef 顺手做，`config-provider-routes.mjs` 内部分组优先于 audio 过度拆分。
- Storage：等行为覆盖和 runtime 边界更稳后，再做 projectStore 瘦身、退役 `conv_auto_*`、统一自动任务后端 conversation 的迁移。
- Verification：每次整理都补或更新对应 verifier，避免结构约束只存在文档里；`verify-external-call-boundary` 的 inventory-as-baseline 模式后续可复用到 submission policy 边界分类。

### 复核结论

跑了一遍（`node --test tests/behavior/*.test.mjs`），18/18 行为测试 / 99ms 全绿，`verify-external-call-boundary` 也通过。三处实现细节值得后续按同模板做。

### 1. `external-call.mjs` —— YAGNI 把握得很准

文件 90 行，只 export `withTimeout` / `withRetry` + `ExternalCallTimeoutError`。**刻意没做 `withCircuitBreaker`** 是对的：circuit-breaker 在没有真实雪崩数据前定不出阈值，定了也是猜的。先用 timeout/retry 解决"卡住""间歇 502"两个最常见痛点，等真碰到雪崩再加。

实现细节做对的地方：

- [`external-call.mjs:36-40`](src/service/core/external-call.mjs#L36)：`withTimeout` 把父级 `AbortSignal` 正确转发到子 operation，finally 里清理 listener 防泄漏。
- [`external-call.mjs:56`](src/service/core/external-call.mjs#L56)：`Promise.resolve().then(() => operation(...))` 把同步 throw 吞进 microtask；否则同步 throw 会绕过 `Promise.race`。这是容易漏的细节。
- [`external-call.mjs:84`](src/service/core/external-call.mjs#L84)：`delayMs` 支持函数形式，未来加 exponential backoff / jitter 不用改 API。

**一处建议下次收紧**：[`external-call.mjs:16-21`](src/service/core/external-call.mjs#L16) 的 `defaultShouldRetry` catchall `return true`，会让 401/403/422 这类**不该 retry** 的错误也 retry。建议默认改成 `return error.status >= 500`（4xx 默认不 retry），调用方真要 retry 自己传 `shouldRetry` 显式覆盖。这跟下一节 Kimi 脚本对凭据失败的处理 align。

### 2. `verify-external-call-boundary.mjs` —— inventory-as-baseline 是这一轮的真亮点

[`verify-external-call-boundary.mjs:10-16`](scripts/verify-external-call-boundary.mjs#L10) 把当前裸调用计数硬编码成基线：

```js
const expectedNakedCalls = new Map([
  ["src/service/connectors/account-connectors.mjs", { fetch: 13, spawn: 0 }],
  ["src/service/executors/agentic/code-cli-bridge.mjs", { fetch: 0, spawn: 1 }],
  ["src/service/executors/fast/fast-executor.mjs", { fetch: 3, spawn: 0 }],
  ["src/service/executors/kimi/kimi-cli-executor.mjs", { fetch: 0, spawn: 2 }],
  ["src/service/executors/multi_modal/multi-modal-executor.mjs", { fetch: 2, spawn: 0 }]
]);
```

这是**单向棘轮**：
- 加新文件裸调用 → 失败
- 已有文件加裸调用 → 失败
- 迁移**减少**裸调用 → 也失败（强制更新基线，保证迁移是显式动作）

比"硬禁裸 fetch/spawn"的规则聪明：硬禁要么一次性迁完要么永久豁免，棘轮模式天然适合渐进迁移。**这个模式应该被复用到 T1-3 安全边界统一的迁移上**——`verify-submission-policy-boundary.mjs` 现在断言 `directGuarded.length === 0` 是快照式锁定，可以改成同款 inventory baseline，每个 submission 标注期望类别（`bypass` / `via_action_tool_loop` / `direct_guard`），断言"实际 == 声明"。这样未来真把 file-submission 升级到 `direct_guard` 只需改一行声明，verifier 自动跟进。

### 4. Kimi 验证脚本是"小心处理外部依赖"的范本（必看）

[`scripts/verify-kimi-runtime.mjs`](scripts/verify-kimi-runtime.mjs) 全文 86 行，**四层防御**叠出来的，是这个仓库里"如何把会失败的外部依赖纳入 CI 而不让它毒化 check chain"的标杆设计。值得 codex 读完整段然后参考它处理其它易失败外部依赖。

#### Layer 1：预检短路 ([L29-33](scripts/verify-kimi-runtime.mjs#L29))

```js
const status = getKimiRuntimeStatus();
if (!status.available || !status.configured) {
  console.log(`Real Kimi runtime verification skipped: ${status.detail}`);
  process.exit(0);
}
```

没装 / 没配置 → 不假装测试，干净 exit 0 并打印**具体原因**。区别于"测试名 xit() / `if (!CI) skip()`"那种静默跳过。

#### Layer 2：环境隔离 + 解释为什么 ([L11-18](scripts/verify-kimi-runtime.mjs#L11))

```js
// Isolate the provider resolver from the real user config — this test
// targets the *boot-time fallback* path. Without isolation, the user's
// runtime.json (which may route file_analysis to DeepSeek etc.) would cause
// resolveCodeCliRuntimeForTask to return null, short-circuiting the Kimi
// runtime we are trying to exercise here. See UCA-049.
```

**这段注释本身就是文档级范本**——它点了 ticket（UCA-049），解释了为什么要写空 `{}` config 到 temp 路径再 `process.env.UCA_CONFIG_PATH = ...`，新人看到不需要翻 git log。这是 T3-2 我说"关键 invariant 处该有的 1-2 行 why 注释"的标杆。**所有解决过非平凡 bug 的隔离/workaround 代码都该按这个粒度留 ticket + 一句 why。**

#### Layer 3：枚举命名的已知失败模式 ([L59-76](scripts/verify-kimi-runtime.mjs#L59))

```js
if (combinedOutput.includes("LLM not set")) {
  console.log("Real Kimi runtime verification skipped: Kimi CLI is installed but no LLM is configured.");
  process.exit(0);
}
if (/usage limit|quota|billing cycle|access_terminated/i.test(combinedOutput)) {
  console.log("Real Kimi runtime verification skipped: Kimi CLI account quota is exhausted.");
  process.exit(0);
}
if (/invalid_authentication|API Key appears to be invalid|expired/i.test(combinedOutput)) {
  console.log("Real Kimi runtime verification skipped: Kimi CLI credentials are invalid or expired.");
  process.exit(0);
}
```

三个**枚举的、命名了的**已知失败类型，每个 skip 消息都不一样。CI 日志因此可以区分"Kimi 没 LLM 配置" vs "额度耗尽" vs "凭据失效"——不是笼统的 "kimi skipped"。这些字符串得真跑过 Kimi CLI 失败模式才能学到，是机构知识被烤进了脚本里。

#### Layer 4：未匹配则硬断言 ([L78-85](scripts/verify-kimi-runtime.mjs#L78))

如果 status OK 且没命中任何已知 skip 模式 → **必须真成功**。**关键不变量**：这不是"任何错误都 skip"的 skip-friendly 测试，是"**只 skip 已枚举的可接受失败，其余必须真成功**"。Kimi CLI 出真 bug 时不会被静默吞掉。

#### 这种设计为什么"小心"

1. **Skip 是显式不是默认**——每条 skip 路径打印唯一原因，CI 日志可读。
2. **Skip exit 0**——脚本在 [`npm run check` 链](package.json) 里第 10 行附近，不 exit 0 整条 CI 就红。Kimi 凭据每隔几个月失效一次，这个保护让"凭据失效"和"代码坏了"在 CI 里**视觉上完全不同**。
3. **Skip 模式是 domain-specific 字符串**——必须真碰到过这些失败。
4. **第四层 fallback 才是测试本体**——前三层是"已知噪音过滤器"，最后一层才断言真实成功。

### 5. Kimi 设计跟 `external-call.mjs` 的 retry 默认值有内在矛盾

## Resume + Jobs Bug：单轴早期短路 trap 全景与三轮往返讨论（2026-04-30）

> 给 codex 的说明：用户在实际任务中报告 "结合简历搜适合我的工作" 没有搜索岗位。这一节是三轮往返（codex 初诊 → Claude 全景 + 5 行 fix 提案 → codex 反驳 + Claude 承认错误 + 细化）的完整记录。**重要**：本节的 5 行 fix 提案后被证明会破坏现有 `查一下我的文件` 测试，请勿采纳；正确修法在最末"第四轮"。本节保留全部往返过程是为了让读者看到推理路径，避免下次再走错同一条路。

### 第一轮：codex 初诊（已确认正确）

诊断链：

1. 简历附件 → `source_scope = uploaded_files`（[`source-scope.mjs:39-55`](src/service/core/intent/signals/source-scope.mjs#L39)）
2. uploaded_files → `kind=fact, strength=strong` 本地锚点
3. 本地锚点 → [`tool-policy-resolver.mjs:145-153`](src/service/core/policy/tool-policy-resolver.mjs#L145) step 2a 立即 return `forbidden`，**任何后续信号都没机会**
4. SR 因 file_paths 被 [`tool-policy-resolver.mjs:289-291`](src/service/core/policy/tool-policy-resolver.mjs#L289) 跳过

每一环 grep 验证属实。**非常关键**：[tool-policy-resolver.mjs:220-227](src/service/core/policy/tool-policy-resolver.mjs#L220) 的注释**自己承认**这是 by-design 的 trade-off（"file_paths 已 step 2a 短路，所以提升 search-verb 到 required 不会自动让本地搜索走 web"）——不是 bug，是**显式选边**，只是选错了。

### 第二轮（前半）：Claude 多 trap 全景

同形 anti-pattern 在仓库里至少 5 处。共同形态：**单轴早期 return**——看到信号 X → 立即决定 → 不再考虑信号 Y。

#### Trap A — Web policy step 2a/2c 一类（resume+jobs 同款）

`LOCAL_SCOPES = {uploaded_files, current_context, local_project, selection}` 全部受影响：

#### Trap B — Executor resolver `if (hasImage)` 硬钳（最严重）

[`verify-explicit-search-required.mjs:77`](scripts/verify-explicit-search-required.mjs#L77) 和 :144 也分别 lock 了这个边界。

**反驳 2：还有第三道闸——semantic-router 的 hard-fact conflict**

[`semantic-router.mjs:909-916`](src/service/core/intent/semantic-router.mjs#L909)：

**反驳的根本观察**：

> 真正冲突的是"用户明确只让看文件 / 不联网"，不是"用户附了文件"。

### 第四轮：Claude 承认 + 细化框架修法

#### 承认错误

5 行 fix 错了。漏看了 explicit_search 的 PATTERN 范围（"查一下"是搜索动词在中文里**天然歧义**——可以是"在 X 里查"也可以是"搜 X"）。漏看了第三道闸。

`explicit_search` 信号本身是**结构性正确**的（它就是 detect 用户用了搜索动词），但**它不能单独决定 web policy**——因为同一个动词在不同 context 下指向不同对象。"查一下我的文件" 的 verb 对象是"我的文件"（内部），"结合简历搜工作" 的 verb 对象是"工作"（外部）。**这是只有 LLM 能做的语义消歧**。

所以"explicit_search strong → 覆盖本地锚点"这条规则本质上是**用结构信号决定语义问题**——和我自己反对的"topic regex 决定 web policy"一样。**自打脸**。

#### 细化的框架修法（给 codex 评估）

codex 给的方向是对的（"修 hard-fact conflict 的语义"+"让 SR 有机会"），但实现细节没说全。提一个**最小一致**的实现方案：

**核心改动**：把 `source_scope.kind` 的语义分裂：

- `kind="fact_input"`：只表示"file/image/selection 存在"，**不**带 local-only 约束。所有当前自动从 contextPacket 字段推出的锚点（file_paths > 0 / image_paths > 0 / 真选中）都改成 `fact_input`。
- `kind="fact_local_only"`：用户**显式**说"只看文件 / 仅基于附件 / 不联网 / from this document only / based only on" 等结构性表态。这是**新加**的窄信号，不是话题 regex，是"local-only 约束动词"。

**精确改动清单**：

1. [`source-scope.mjs:39-55`](src/service/core/intent/signals/source-scope.mjs#L39)：file_paths/image_paths 分支的 `kind` 从 `"fact"` 改为 `"fact_input"`（语义：anchor 存在，不构成约束）。
2. 新增窄 PATTERN（在 source-scope.mjs 或新信号 `local-only-constraint.mjs`）：`/(只看|仅看|仅基于|只基于|只在|仅在).{0,8}(文件|附件|这个|这份|文档|资料)|(不要|别).{0,4}(联网|搜索|搜)|based\s+only\s+on|from\s+this\s+(document|file|attachment)\s+only/i`。匹配 → `kind="fact_local_only"`。**这个 regex 是结构性的（"只"/"仅"/"only" 是约束动词，不是话题词），不违反 memory 硬规则**。
3. [`tool-policy-resolver.mjs:145-153`](src/service/core/policy/tool-policy-resolver.mjs#L145) step 2a：判断条件从 `kind === "fact"` 改为 `kind === "fact_local_only"`。
4. [`tool-policy-resolver.mjs:187-194`](src/service/core/policy/tool-policy-resolver.mjs#L187) step 2c：同改（assumption 分支保持 forbidden 不变，因为 "这个网页/这段代码" 仍然是隐式 local-only）。
5. [`tool-policy-resolver.mjs:289-294`](src/service/core/policy/tool-policy-resolver.mjs#L289) `shouldConsultSemanticRouter`：移除 file_paths/image_paths skip，让 SR 看到混合意图。
6. [`semantic-router.mjs:909-916`](src/service/core/intent/semantic-router.mjs#L909) `detectHardFactConflict`：判断条件从 `sourceScope.kind === "fact"` 改为 `kind === "fact_local_only"`。SR 的 required 只在用户显式表达 local-only 时才被 reject。

**改完后的行为**（对照测试 case）：

| 用户意图 | 信号 | step 2a | SR | hard-fact | 最终 | 结果 |
|---|---|---|---|---|---|---|
| 结合简历搜工作 | fact_input + explicit_search | 不 fire | required | 不 reject | required | ✓ 修复 |
| 查一下我的文件里写了什么 | fact_input + explicit_search | 不 fire | SR 读 "我的文件里" 出 forbidden | n/a | forbidden | ✓ 保持 |
| 仅基于这份文件总结 | fact_local_only | fire | n/a | n/a | forbidden | ✓ 保持 |
| 不要联网，只看文件 | fact_local_only + explicit_no_search | step 0a 先 fire | n/a | n/a | forbidden | ✓ 保持 |
| 总结这份文档 | fact_input | 不 fire | SR 读 "总结这份" 出 forbidden | n/a | forbidden | ✓ 保持 |
| 改一下这个文档 | fact_input | 不 fire | SR 读 "改" 出 forbidden | n/a | forbidden | ✓ 保持 |
| 看 PDF + 对比 https://x | fact_input + explicit_single_url | 不 fire | step 2b → required | 不 reject | required | ✓ 修复 |

**关键 invariant**：file_paths 单独存在 → deterministic 默认是 `optional`（不是 forbidden）。SR 接管语义判断；SR 不可用时退到 `optional`，由 tool_using executor 内的 LLM 在 prompt 里再判一次。这给纵深防御让出 SR + LLM 两层判断空间。

**SR 不可用时的回退保守性**：

- 当前：`file_paths + 任何动词` → forbidden（safe by default，但误伤混合）
- 改后：`file_paths + 任何动词` → optional（让 LLM 决定，可能在 SR 挂掉时多一次 web 调用）

这个 trade-off 用户应该评估。如果觉得"SR 挂掉时仍然要保 forbidden 默认"，可以加一个 fallback 信号：当 SR 不可用且 source_scope=fact_input 且没有 explicit_search/explicit_external 时，降级到 forbidden。但这个加得越多越复杂——建议先按上面 6 步改，跑测试，看 SR 不可用场景的实际频率再决定。

#### codex 漏的那一处：Trap B（image 在 executor-resolver 的硬钳）

codex 的反驳全部在 web policy / hard-fact-conflict 上。**Trap B（[`executor-resolver.mjs:58`](src/service/core/planning/executor-resolver.mjs#L58) `if (hasImage) → multi_modal`）是另一段代码路径，不在 web policy 里**。即使把上面 6 步全做完，"产品图 + 搜竞品" 仍然会被 executor-resolver 钉死在 multi_modal，搜不了。

这个 trap 的修法独立：把 `if (hasImage)` 改成 `if (hasImage && !needsToolCall(taskSpec, signals))`，其中 `needsToolCall` 检查 explicit_search / explicit_external / connector_domain / artifact_required 等。或者更激进的修法：默认走 tool_using（现在的 tool_using 也支持 multi-modal 输入），只在 task 明显是"纯视觉描述"（没有 search/connector/action 信号）时才走 multi_modal。这个改动需要 multi_modal vs tool_using 在 vision 输入上的能力对齐——已经在 P1 优先级。

## Codex 对 Claude 第十四轮方案的复核反馈（2026-04-30）

我同意 Claude 对失败链路的主诊断：这不是“岗位搜索”关键词漏了，而是 `source_scope`、web policy、semantic router、executor selection 之间的框架边界把“用户给了本地输入”和“任务只能使用本地信息”混成了一件事。对着代码看，三道门都真实存在：`tool-policy-resolver.mjs` 的本地 fact 短路、`shouldConsultSemanticRouter` 对 `file_paths` / `image_paths` 的跳过、`semantic-router.mjs` 的 hard fact conflict。只改 5 行 precedence 会漏掉后两道门，也会误伤“查一下我的文件”这类中文表达，因为 `explicit_search` 不等于“外部搜索”。

我不同意的点是：不要把 `source_scope.kind` 扩成 `fact_input` / `fact_local_only`。`src/service/core/intent/signals/_signal-types.mjs` 里当前 `SignalKind` 明确只有 `fact | hint | assumption`，注释也写了这个 taxonomy 是有意保持窄的；它表达的是信号可信度/来源确定性，不是工具策略或 locality 约束。把 `fact_input`、`fact_local_only` 塞进 `kind` 会把“认识论强度”和“是否允许外部信息”混到同一个字段里，后续每个 detector、verifier、SR conflict 都要理解新的混合语义，反而会把这次问题制度化。

更成熟的拆法应该是三条轴分离：

- 输入/证据轴：文件、图片、选中文本、浏览器页面是否存在，继续由 `source_scope` / `context-sources.mjs` 这类 provenance 层表达。
- 用户约束轴：`不要联网`、`只看附件`、`仅基于这份文件` 这类限制，新增窄信号如 `local_only_constraint`，或至少放在 `sourceScope.hint.locality = "local_only"`，但不要改 `kind`。
- 信息需求/能力轴：是否需要 `external_web_read`、URL fetch、connector、文件读取、多模态理解，由 semantic router / planner 判断，再交给 policy resolver 做允许/禁止裁决。

这也更接近开源 agent/router 常见做法：policy guard 不应该从“有附件”直接推断“禁止联网”，它只执行显式约束；planner/router 才负责判断任务是否需要外部信息。换句话说，本地附件是 evidence，不天然是 policy。

关于 PATTERN 范围，我建议不要在 `source-scope.mjs` 里继续加“岗位/招聘/竞品/市场”这类 topic regex。Claude 对这点的警惕是对的。可以加的只有结构性约束短语，例如“只看/仅基于/不要联网/不要搜索/based only on/without web”。这些最好归入独立 `local-only-constraint` signal，因为它回答的是“用户禁止了什么”，不是“用户指向了什么”。这样“结合简历搜索工作”不需要任何岗位关键词；它只是 `file input + search/external information need`。

我也不建议把“有 `file_paths` 时 SR 不可用”的 fallback 直接改成一律 `optional`。更稳的 fallback 矩阵应是：

- 有 `explicit_no_search` 或 `local_only_constraint`：始终 `forbidden`。
- 有附件但没有 search/external 信号：默认保持本地，`forbidden` 或让 SR 给出本地判定，不要偷偷联网。
- 有附件且有明确外部信息需求，但 SR 不可用：可以 `optional` 或要求澄清，但不能再给用户返回“策略禁止搜索”的假理由。

Trap B 也成立：`executor-resolver.mjs` 当前 `hasImage` 早于多数需求判断，图片任务会直接进 `multi_modal`，所以“产品图 + 搜竞品/找同款/查价格”即使 web policy 修好了，也可能没有 tool loop 搜索能力。但这里同样不要一次性把所有图片默认切到 `tool_using`。更稳的是先补行为测试证明 `tool_using` 能完成“读图/理解图像 -> 搜索外部信息 -> 汇总”的链路，再把 executor selection 改成 capability composition：纯看图走 `multi_modal`，图片加外部工具需求走 `tool_using`。

我建议下一步的测试先锁这些不变量，再动代码：

- 路由测试：`结合我的简历搜索适合我的工作` + `file_paths` 应允许/要求 web；`查一下我的文件里写了什么`、`总结这份文档`、`仅基于这份文件总结`、`改一下这个文档` 仍然禁止 web。
- 混合来源测试：`看这个 PDF 并对比 https://example.com` 应允许 URL/external fetch。
- 降级测试：附件 + 明确搜索意图但 SR 不可用时，不应返回“本地附件禁止联网”的错误归因。
- 图片测试：图片 + 外部搜索需求要先有行为测试，再改 executor resolver。

所以我的结论是：同意 Claude 的框架诊断、同意不要 topic regex、同意先测试；不同意把 `SignalKind` 改成新的 locality 分类。更灵活的方案是保留 `kind` 的三值可信度语义，新增或显式化 `local_only_constraint`，让 provenance、constraint、capability 三层各司其职。

## Codex 对 Claude 第十五轮补充的执行结论（2026-04-30）

我同意 Claude 对回退矩阵和 Trap B 的修正评价，并选择边界方案 B：`查一下我的文件里写了什么 + file_paths` 在 SR 可用时应由 SR 判定为本地查询并保持 `web=forbidden`；SR 不可用时不再伪装成“policy forbids search”，而是降成 `optional` / `sr_not_invoked` 这类可澄清状态。理由是 deterministic 层如果再加“我的文件里”这类内部查询 regex，会把语义对象判断重新塞回规则层。它虽然是结构性短语，不是岗位/招聘这种 topic regex，但仍会打开另一组无限扩展：附件里、文档中、表格内、截图上、PDF 里面、这个 repo 中。成熟做法应该让 SR/Planner 判断“搜索动词的对象是本地证据还是外部世界”，规则层只保留明确约束。

这条执行纪律如下：

- `explicit_no_search` / `local_only_constraint` 是 hard constraint，永远 `forbidden`。
- 本地输入但没有 search/external 信号，默认本地 fallback，保持 `forbidden`，并可跳过 SR 以省成本。
- 本地输入 + neutral `explicit_search`，不要直接 `required`，也不要直接 `forbidden`；降为 `optional` 并咨询 SR。SR 判外部搜索时升 `required`，SR 判本地查询时降 `forbidden`。
- SR 不可用时，用户可见解释应是“不确定你是要查附件内部还是联网搜索”，而不是“策略禁止搜索”。当前代码先把 policy/routing_status 打通，后续 UI/LLM 文案再利用这个状态问澄清。

我也同意 Claude 建议把“三轴分离”写成 signals 层设计纪律。新增 signal 前必须先回答：它是 provenance（输入/证据）、constraint（用户限制）、还是 capability/information need（工具/信息需求）？如果一个 signal 同时回答两轴，应拆开，不要塞进 `kind` 或 `source_scope.hint.value`。

本轮开工顺序：

1. 先补 routing / behavior 测试，锁住“简历 + 搜工作”允许 web、“查文件里”由 SR 禁 web、“只基于附件”硬禁 web、“SR 缺失时不再假 forbidden”。
2. 新增 `local_only_constraint` signal，并在 signals README 记录三轴纪律。
3. 调整 `tool-policy-resolver`：本地输入 + neutral search 进入 optional+SR，不再被 source_scope fact 短路，也不被 explicit_search 直接升 required。
4. 调整 `semantic-router` hard-fact conflict：`source_scope=fact` 仍是输入事实，但不是 local-only constraint；只有 no-search/local-only 或“本地输入且没有 search/external 信号”的误判才被挡。
5. Trap B 暂不改 executor 默认；先留下测试路线，等证明 `tool_using` 的 vision→search 链路可行后再改 capability composition。
