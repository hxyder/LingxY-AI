# Linxi 面向日常用户的桌面生产力工具功能补充方案

## 1. 核心判断

如果 Linxi 面向的是日常用户，而不是开发者、研究者或重度自动化用户，那么它不能只强调：

- 多模型
- MCP
- Skills
- Agent 调度
- 工具调用
- 自动化工作流
- 本地配置
- API key

这些能力对高级用户很有价值，但对普通日常用户来说，门槛仍然偏高。

日常用户真正关心的是：

```text
我现在有点乱 → 你帮我整理
我不知道怎么做 → 你给我选项
我不想重复操作 → 你自动帮我做
我怕弄错 → 你先让我确认
我忘了什么 → 你提醒我
我找不到文件 → 你帮我找
我看不懂内容 → 你解释
我电脑很乱 → 你帮我收拾
```

所以，面向日常用户的 Linxi 应该从：

```text
强大的桌面 AI 工具箱
```

升级为：

```text
每天都能帮用户处理电脑杂事、信息、文件、日程、学习和生活决策的桌面 AI 小管家
```

---

## 2. 日常用户版 Linxi 的产品目标

### 2.1 产品定位

建议定位为：

> Linxi 是一个常驻桌面的 AI 生活与工作助手，能帮用户找文件、整理电脑、理解截图、总结网页、写邮件、创建提醒、处理账单和生成文档。

更生活化的表达：

> 你不用记命令，也不用配置复杂工作流。你只需要告诉 Linxi 你想做什么，它会帮你找到资料、整理内容、生成结果，并在关键操作前让你确认。

---

### 2.2 产品关键词

```text
简单
省心
可控
安全
每天可用
结果可见
不乱动电脑
不偷偷发送
不强迫配置
```

---

### 2.3 日常用户最重要的体验

对于普通用户，Linxi 应该做到：

| 用户心理 | Linxi 应提供的体验 |
|---|---|
| 我不知道怎么问 | 提供快捷按钮和场景入口 |
| 我不想配置 | 默认简单模式，按需引导权限 |
| 我怕 AI 乱操作 | 所有高风险动作先确认 |
| 我怕隐私泄露 | 清楚显示正在读取什么、发送什么 |
| 我想快点完成 | 右键菜单、截图入口、快捷键 |
| 我不想学工作流 | 自然语言即可创建任务和提醒 |
| 我想找东西 | 自然语言搜索本地文件和内容 |
| 我想少做重复操作 | 自动识别重复流程并保存为技能 |

## 3. 桌面整理助手

### 3.1 用户痛点

普通用户的桌面、下载文件夹、微信文件、邮件附件经常非常混乱。  
他们经常不知道哪些文件有用、哪些重复、哪些可以整理。

典型请求：

```text
帮我整理一下桌面
把下载文件夹里没用的东西分分类
找出重复文件
把最近一周的 PDF 放到一个文件夹
把截图按日期整理
把课程文件和账单文件分开
```

---

### 3.2 功能设计

| 功能 | 说明 |
|---|---|
| 自动分类 | 按 PDF、图片、压缩包、安装包、文档、表格、截图分类 |
| 智能命名 | 根据文件内容生成更清楚的文件名 |
| 重复文件检测 | 找出重复图片、重复 PDF、重复下载 |
| 临时文件清理 | 找出可能可以删除的缓存/安装包，但不自动删除 |
| 大文件提醒 | 找出占空间最大的文件 |
| 桌面一键收纳 | 把桌面文件按类型或日期整理 |
| 文件夹规则 | 以后下载的新文件自动归类 |
| 微信/邮件附件整理 | 自动识别并归档常见附件 |

---

### 3.3 操作原则

不能自动删除。  
所有移动、删除、重命名操作都必须先展示预览。

示例：

```text
我找到了：

- 37 个截图
- 12 个 PDF
- 5 个安装包
- 6 个重复文件
- 3 个超过 500 MB 的大文件

建议：

1. 截图 → Pictures/Screenshots/2026-05
2. PDF → Documents/PDF
3. 安装包 → Downloads/Installers
4. 重复文件 6 个，可删除 4 个

是否执行？
```

### 4.1 用户痛点

普通用户经常记不住文件名，只记得模糊信息：

```text

我记得有个关于 insurance 的 PDF，帮我找一下
找一下最近打开过的 Excel

```

---

### 4.2 功能设计

| 查询方式 | 示例 |
|---|---|
| 按内容找 | 找写了 lease renewal 的文件 |
| 按时间找 | 找我上周下载的 PDF |
| 按来源找 | 找微信保存的那个文件 |
| 按类型找 | 找最近的 Excel 表格 |
| 按图片内容找 | 找那张有车牌的照片 |

### 4.3 推荐体验

```text

   最近修改：2025-07-14

2. Apartment_Inquiry_Email.docx
```

## 5. 截图和剪贴板助手

### 5.2 功能设计

| 功能 | 场景 |


## 8. 提醒和待办助手

### 8.1 用户痛点

用户经常从邮件、网页、聊天、PDF 里看到一些需要之后处理的事情，但容易忘记。

典型请求：

```text
明天下午提醒我交房租
下周一提醒我给教授发邮件
每个月 1 号提醒我检查信用卡账单
3 天后提醒我 follow up 这个邮件
看到 Amazon 包裹送达后提醒我
```

---

### 8.2 功能设计

| 功能 | 说明 |
|---|---|
| 自然语言提醒 | 不需要写 cron |
| 从邮件提取待办 | 自动识别 deadline、reply、meeting |

| 重复提醒 | 每天、每周、每月、每年 |
| 任务状态 | todo / done / snoozed / expired |
| 跟进提醒 | 对邮件和消息设置 follow-up |

---

### 8.3 交互示例


## 12. Life Admin Vault：生活资料库

### 12.1 用户痛点

用户电脑里有很多重要生活文件：

- 租房合同

- 医疗记录

用户经常找不到，或者看不懂。

---

### 12.2 功能设计

| 分类 | 内容 |
|---|---|
| Housing | lease、rent、utilities |
| Health | insurance、doctor notes、prescriptions |
| Car | insurance、registration、maintenance |
| School | transcripts、forms、deadlines |
| Finance | bills、tax forms、receipts |
| Identity | passport、visa、I-20、I-94 |
| Work | offer letter、paystub、HR docs |

---

### 12.3 用户可问

```text
我的租房合同里 pet policy 是什么？
```

---

### 12.4 安全要求

Life Admin Vault 必须具备：

- 本地加密

- 不自动上传
- 读取前提示

- 可查看访问记录

# Part VII. 最终总结

面向日常用户，Linxi 不应该只做一个强大的 agent 工具箱。

它更应该成为：

```text
一个不用学、不用配、不用理解 agent，就能帮用户处理电脑杂事的小助手。
```

当前 Linxi 已经具备很多高级能力，例如：

- 工具调用
- 文档生成
- 定时任务
- MCP
- Skills
- 多模型
- 邮件集成
- 浏览器和 Office 集成

但面向日常用户，还应该重点补齐：

```text
找文件
整理文件
截图理解
智能右键
邮件草稿
提醒待办
网页比较
账单收据
生活资料库
简单模式
下一步建议
隐私透明提示
```

最终产品方向应该是：

> 入口简单，场景明确，结果可控，隐私透明，每天都能用。

# Part VIII. Claude 评估回复（2026-05-03）

> 我对照了 LingxY 现有代码（src/desktop / src/service / src/desktop/renderer / scripts/verify-* 共 967 个 tracked 文件），把这份提案逐条核对。整体**同意核心洞察、反对全盘照做**。下面分四个部分：已有能力清单、真缺什么、不建议做的、与 v1.0 上线节奏的协同。

---

## 38. 整体判断

**同意的核心洞察**：

- **当前 LingxY UX 偏向技术用户**：Console 12+ tabs 平铺、首次启动 5 步配置（权限/剪贴板/LLM/连接器/MCP），普通用户进来确实门槛偏高。
- **场景化分类比能力分类更接地气**："找文件 / 整理电脑 / 总结网页 / 写邮件 / 创建提醒"这种用户心理驱动的分组对消费者更友好。
- **副作用必经审批**是对的——这条 LingxY 已经做到，但 UI 强度可以更高。

**反对的部分**：

- **18 个功能模块全做是 12-24 个月工程**，不能作为 v1.0 P0。
- **很多"提案应该有"的功能 LingxY 已经有底层能力**，缺的只是 UX 包装。盲目重做会浪费已有工程投入。
- **Life Admin Vault / 购物比较 / 学习助手 / 收据整理 / 电脑问题修复等垂直功能不应内建**——应该走 Skills 路径，让生态扩展，不让核心产品什么都做但都不精。

---

## 39. 已有底层能力清单（避免重造）

下表是提案功能对照现有代码的诚实结果。**已有底层 + 缺 UX 包装**的项不应被当作"完全没有"重做。

### 39.1 提案功能 vs 现有实现

| 提案功能 | 现状 | 缺什么 |
|---------|------|-------|
| **#5 截图解释 / OCR** | `take_screenshot` 工具 + Vision 模型（Anthropic / OpenAI / Doubao）+ Windows OCR + Tesseract fallback 都已实现 | 缺"截图后自动弹建议卡片"的 UX 触发 |
| **#5 截图翻译** | `translate_text` 工具 + screenshot 组合可用 | 缺右键菜单入口 |
| **#7 邮件草稿 draft-confirm-send** | Gmail/Outlook connector workflow 已有 `gmail.draft-confirm-send.json` / `outlook.draft-confirm-send.json`，**永不自动发送**已实现 | 缺模板库（apartment / medical / professor 等场景） |
| **#7 多版本草稿** | agentic 一次能输出多版本，调一次模型即可 | 缺 UI 模板按钮 |
| **#8 自然语言提醒** | `nl_to_cron.mjs` 解析器 + schedule 系统已有，"每天 9 点提醒我"已可用 | 缺"从邮件自动提取 deadline"的触发 |
| **#8 Calendar 同步** | Google + Microsoft Calendar connector 已有（含 OAuth） | 缺"邮件→日历"快捷流程 |
| **#8 桌面 popup 提醒** | Popup card 系统已有（snooze / done / reschedule 等动作未全实现） | 缺 snooze 等交互 |
| **#9 总结/翻译网页** | `web_search_fetch` 工具 + 浏览器扩展同步当前 URL，"分析此页面"按钮在 Overlay 顶部 | 缺自动 prompt chips |
| **#14 学习助手 quiz/flashcards** | document_generator 可输出，agentic 能解析 PDF | 缺学习场景模板 |
| **#16 自然语言启动应用** | `launch_app` 工具，src/service/action_tools/tools/index.mjs:28-62 有微信/Zoom/Chrome/钉钉/Word/Excel/Outlook/VSCode 等 appName 映射 | **已完整可用** |
| **#16 打开最近文件** | `find_recent_files` / `glob_files` / `list_files` 工具 | 缺语义检索（"找体检报告"这种） |
| **#18 笔记快速记录** | Console → Notes tab、`/notes` / `/notes/upsert` / `/notes/append-chip` 已有 | 缺自动分类 + 笔记转任务 |
| **#24 隐私模式** | `privacy_settings` 模块已有：redaction rules / kill switch / presenter mode / data retention / app exclusion list 框架 | 缺面向用户的"隐私指示灯"实时 UI |
| **#24 Kill switch** | 已有 | 缺一键可见入口 |
| **#24 Audit log** | `audit-log.mjs` 已记录所有 tool call | 缺面向用户的访问记录视图 |
| **副作用审批** | popup card + connector workflow draft-confirm-send 模式已实现，发邮件 / 删文件 / 安装 MCP 包都走审批 | 已完整 |
| **首次启动引导** | `first_run_wizard/view-model.mjs` 5 步引导（权限/剪贴板/LLM/连接器/MCP） | 但**是配置驱动而不是任务驱动**（这是提案 #20 真正缺的） |
| **多窗口形态** | Console / Overlay / Dock / Popup card / Preview 5 种已有 | 提案没否认这点，只是说首页 IA 该重组 |
| **多 provider** | Anthropic / OpenAI / Gemini / DeepSeek / Kimi / Moonshot / xAI / Mistral / Ollama 都有 | 已完整 |
| **MCP / Skills** | 三层扩展系统都有 | 已完整，但 MCP 一键安装（社区 npm）刚做完 |

**结论**：提案里大约 **40-50% 的功能 LingxY 有底层能力**。提案的痛点不是"没功能"，是**"功能藏得深，普通用户找不到/不知道能这么用"**。这意味着主要工作应该是 **UX 包装**而不是"从零实现"。

| **#22 下一步建议卡片** | 中 | 小 | AI 输出后加结构化 follow-up |

## 41. 不建议做的（应该走 Skills 路径）

下面这些做成"项目核心功能"会偏离 LingxY 通用 AI 助手定位。建议作为**官方/社区 Skills 包**，让用户按需安装：

| 提案功能 | 为什么不进核心 | 应该怎么做 |
|---------|--------------|----------|
| **#10 购物/价格比较** | 垂直度高、涉及电商爬虫合规问题、不同地域规则差异大 | 不做，让用户用通用 web_search 工具临时实现 |
| **#11 生活决策助手** | 过于宽泛，靠 agentic prompt 即可，不需要专用结构 | 不做 |
| **#12 Life Admin Vault** | 大功能模块（housing/health/car/finance/identity），等于做半个 Notion | 拆成 housing-vault.skill / health-vault.skill 等独立 Skill 包，用户按需装 |
| **#13 收据账单整理** | 通用 Vision OCR 已能做，专用模板不应内建 | 做成 receipt-organizer.skill |
| **#14 学习助手 quiz/flashcards** | 学生场景独立、Anki 等已有专用工具 | 做成 study-helper.skill |
| **#15 电脑问题助手** | **风险面太广**：改注册表 / 改环境变量 / 关闭安全软件 / 改网络设置 / 删系统文件——AI 出错代价高 | **明确不做**。普通用户不该用 AI 改这些东西 |
| **#17 稍后阅读专用模块** | notes + projects 已能承载 | 不做专用，提供 save-as-note 入口即可 |

**走 Skills 路径的好处**：
- 用户按需安装，不用的不污染界面（直接对应 #19 简单模式诉求）
- 社区可贡献（生态扩展，不是单一产品包揽）
- LingxY 核心保持精简，质量可控
- 失败的实验可以直接卸载，不留痕迹

## 42. 关键反驳（按宪法 R1 默认反驳）

### 反驳一：#19 简单模式不能作为 P0 优先项

提案把简单模式列为 P0（"日常用户必须降低门槛"），但这是**大工程**（重设计 12+ tabs 的 Console IA）。建议改为：

- **P0+ 先做"看得见的低工程项"**：右键菜单（#6）+ Prompt chips（#21）+ 隐私指示灯（#24）。这些 ~5 天工作量，让普通用户立刻有 ROI。
- **P1 简单模式**：作为下一阶段重设计项目，因为它涉及 Console 信息架构变更，要从"功能视角"切换到"任务视角"，是 2-3 周工程。

如果一上来就重设计 IA，会拖慢 v1.0 上线节奏。

### 反驳二：Life Admin Vault 不应该内建

提案花了大段描述 housing/health/car/finance/identity 的分类索引、本地加密、敏感文件锁定、访问记录等。这是**独立产品级别**的投入（类似 1Password / Bitwarden 的范畴）。

LingxY 应该提供**通用底层能力**（Skills + Notes + Projects + privacy_settings 已有），让用户/社区自己组合"我的住房 vault""我的医疗 vault"，而不是内建一套生活分类预设。

否则 LingxY 会变成"什么都做但都不精"的产品。

### 反驳三：#15 电脑问题助手必须明确不做

提案虽然加了"系统修改前确认"，但"修改注册表 / 改环境变量 / 关闭安全软件 / 改网络设置 / 执行陌生脚本"这些操作的**风险面太广**，AI 出错代价高，用户也无法判断 AI 给的建议是否安全。

加上桌面 AI 工具是相对新的产品类别，**用户对"AI 改我系统"的心理接受度低**。一旦出过事故，对产品声誉是毁灭性打击。

建议明确不做，写在 README 的"不做的事"列表里。

### 反驳四：#4 自然语言找文件涉及隐私敏感

提案描述的"按内容找文件"需要建立**本地文件全文索引**（涉及读取所有文件的内容并向量化）。这有两个隐藏成本：

1. **隐私敏感**：用户的所有文档（合同、医疗、财务）会被索引到本地数据库。即使不上传，索引文件本身就是泄漏风险。
2. **存储和性能**：百万级文件的索引和增量更新非常吃磁盘和 CPU。

建议**作为可选 opt-in 功能**而不是默认开启，且必须有：
- 文件夹白名单（用户明确指定哪些目录索引）
- 索引文件加密
- 一键删除索引
- 索引访问记录

简言之：可以做，但不能做成"默认开启的全盘扫描"。

### 反驳五：剪贴板自动建议（#5）需要权衡用户烦扰

"复制内容后自动推荐操作"听起来很方便，但实际是**OS 级别的剪贴板监听**——会弹很多卡片打扰用户。

参考竞品：macOS 的 Universal Clipboard、Windows 的 Cloud Clipboard 都没做"复制后自动建议操作"，因为太吵。

建议改为：**用户主动唤起 Overlay 时**（按快捷键），自动检测剪贴板内容并显示 prompt chips（这是 #21 的延伸），而不是被动监听弹卡片。

## 44. 总结

**这份提案的最大价值**：把"用户心理需求"作为分类轴，识别出 LingxY 当前 UX 偏向技术用户的问题。这个洞察对。

**这份提案的最大风险**：野心太大，18 个功能模块若全做会让 LingxY 变成"什么都做但都不精"的产品，背离通用 AI 助手定位。

**我的建议**：

1. **采纳核心洞察**：用户心理驱动 + 简单模式 + 右键菜单 + prompt chips + 隐私指示灯
2. **不重造已有轮子**：截图/翻译/邮件草稿/自然语言提醒/Calendar/启动应用等底层都在，做 UX 包装即可
3. **垂直功能走 Skills 路径**：Life Admin Vault / 学习助手 / 收据整理等做成 Skills 让生态扩展
4. **明确不做的事**：电脑问题修复 / 购物比较 / 默认全盘索引 / 被动剪贴板监听
5. **节奏**：v1.0 上线先补质量基础（A1）+ 三个高 ROI 入口（A2）= ~10 天工作量；公开后再做 IA 重设计（B）

**最终方向（同意提案的最后一句）**：

> 入口简单，场景明确，结果可控，隐私透明，每天都能用。

但这是**目标**而不是**v1.0 范围**。v1.0 的目标是"让第一批用户愿意试用且不会因为基础质量问题流失"，简单模式 / 任务驱动 onboarding 等 IA 重设计应该作为第二阶段，先验证质量再优化体验。

# Part IX. Codex 评估回复（2026-05-03）

我同意 Claude 的主判断：这份提案真正有价值的不是“再堆 18 个功能”，而是把 Linxi 的产品叙事从“能力清单”转成“日常场景入口”。但我对 Claude 的落地排序和“已有能力”判断有几处不同意见，下面只写差异点。

## 45. 我同意的部分

- **普通用户入口要场景化**：Ask / Files / Web / Create / Automate / Privacy 这类任务入口，比现在 Console 里按工程模块分 tab 更好理解。
- **垂直功能应优先走 Skills**：Life Admin Vault、学习助手、收据整理这类场景不应塞进核心产品；核心应提供文件、截图、笔记、导出、隐私、审批、模板、skills runtime。
- **默认全盘文件索引不能做**：自然语言找文件可以做，但必须 opt-in、目录白名单、可删除索引、最好加密，不能默认扫描用户电脑。
- **被动剪贴板弹卡要谨慎**：更成熟的体验是用户唤起 Overlay 时读取当前剪贴板并给 prompt chips，而不是每次复制都打扰。

## 46. 不同意或需要校正的地方

### 46.1 Secret Store 不应放到公开后 B 批次

Claude 在批次 B 里写 “Secret store（API key 加密到 OS keychain）”，我不同意这个排序。对一个要求用户输入 LLM API key 的桌面工具，provider key 的存储边界属于 v1.0 信任底座，应该在 auto-update 之前或并行处理。

当前代码已经推进第一阶段：

- `src/service/security/secret-store.mjs` 新增 provider secret ref。
- `runtime.json` 不再保存 provider `apiKey` 明文，只保存 `apiKeyRef`。
- `/config/providers` 不再向 Console 回传 `apiKey`，只回传 `apiKeyConfigured`。
- `resolveProviderForTask()` 会同步水合 `apiKeyRef`，保持 executor 不需要大规模 async 改造。

但我也不同意把这一步描述成“已加密到 OS keychain”。当前是 file-backed local secret store，解决的是“配置文件明文 key”问题；OS keychain 需要后续 resolver async/cache 设计，不能在发布前硬塞。

### 46.2 `launch_app` 不能说“已完整可用”

Claude 表里写“自然语言启动应用已完整可用”，这过于乐观。代码里确实有 `src/service/action_tools/tools/index.mjs` 的 `launch_app` 工具和常见 app 映射，但真实任务已经暴露过框架问题：

- app 多候选时，工具返回 `success=false` + 中文 observation，而不是结构化候选列表。
- planner 看到第一个工具失败后可能直接 final，没继续执行用户要求的其他独立动作。
- final composer 可能退化成“我无法直接在你的电脑上打开”，与本地 agent 能力自相矛盾。

所以结论应改成：**底层工具存在，但还不是普通用户意义上的“完整可用”**。成熟修复不是给 YouTube/Excel/Word 写补丁，而是做三件框架事：

- `launch_app` 的多匹配返回结构化 candidates。
- planner 对复合独立动作要有 continue-on-independent-failure 纪律。
- final composer 在工具失败时不得自我否认本地能力，只能如实报告“已尝试/哪些失败/需要用户选择什么”。

### 46.3 副作用审批不能简单写“已完整”

连接器 draft-confirm-send、安全 broker、local mutation guard 这些边界已经很强，但从日常用户视角还不能说“完整”：

- 用户仍会遇到“定时任务运行后还要审批”这种体验问题，需要默认审批策略、任务确认卡和运行时提示更清楚。
- `Run now`、scheduled fire、connector send、file move/delete、MCP install 这些都是不同副作用域，用户应该看到统一的 approval vocabulary。
- 审批不是只有“有没有 guard”，还包括“为什么要我批、批了会发生什么、失败后能否撤销”。

因此我会把“副作用审批已完整”改成：**后端 guard 基线已完成，普通用户可理解的审批 UX 仍需收口**。

### 46.4 右键菜单 ROI 高，但不一定是 v1.0 阻塞

Windows shell extension 对普通用户非常有价值，但它会引入安装器、注册表、卸载清理、Explorer 重启、权限、杀软误报等发布复杂度。Linxi 现在正准备 GitHub 上线，我会把它排为 v1.0+ 高 ROI，而不是阻塞 v1.0 的 P0。

更低风险的入口顺序我倾向：

1. Prompt chips：复用 Overlay/Console/当前上下文，不碰系统扩展。
2. 隐私指示灯：复用现有 `privacy_settings` / audit log，增强信任。
3. 任务驱动 onboarding：让用户 5 分钟内完成第一件事。
4. 右键菜单：等 installer/release workflow 稳定后做。

### 46.5 “电脑问题助手”不应全盘不做，但必须限制为只读诊断

Claude 说 #15 必须明确不做，我同意“不让 AI 改系统”，但不同意完全放弃这个用户需求。普通用户确实经常需要解释错误弹窗、网络断了、软件打不开、磁盘满了。

建议边界：

- 可以做：截图解释、错误日志解释、只读系统信息收集、生成手动步骤、打开设置页。
- 不做：改注册表、删系统文件、关闭安全软件、执行陌生脚本、自动改网络/环境变量。

也就是说核心产品可以提供**只读诊断助手**，但不提供“AI 自动修电脑”。

## 47. 我建议合并后的产品路线

### v1.0 信任底座

1. Provider Secret Store 第一阶段（已启动）：key 不进 runtime config。
2. 数据导出 bundle：notes / conversations / tasks / schedules / artifact manifest，不含 secrets。
3. 本地诊断包：最近 logs/events/config redacted，用户手动附到 issue。
4. Soft delete / Undo：统一 deleted_at / restore_until。
5. Release workflow：installer、checksums、release notes、license artifact。

### v1.0+ 低风险日常入口

1. Prompt chips：基于当前窗口、选中内容、剪贴板、截图的快捷动作。
2. 隐私指示灯：显示正在读取什么、准备发送什么、谁批准了什么。
3. 任务驱动 onboarding：不是“先配完所有东西”，而是“你想先完成什么”。
4. 创建/定时任务确认卡：明确一次性/重复、是否发送、是否需要审批。

### 公开后扩展

1. 简单模式 / 高级模式：重做 Console IA。
2. Windows 右键菜单：作为系统级入口。
3. 本地文件语义搜索：opt-in、目录白名单、可删除索引。
4. Skills 市场化：Life Admin / Study / Receipt / Housing 等走官方或社区 skills。

## 48. 总结

Claude 的提案取舍大方向合理，但有三处需要收紧：Secret Store 要前置；`launch_app` 和副作用审批不能按“底层有代码”就算产品完整；右键菜单很好但不该压过发布信任底座。

我的判断是：**Linxi 现在最缺的不是更多能力，而是“用户敢用、知道怎么用、出了问题能恢复”的产品框架**。这和程序整理里的 v1.0 顺序一致：先信任底座，再做日常入口，再做大 IA。
