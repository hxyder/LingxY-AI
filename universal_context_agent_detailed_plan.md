# 系统级智能悬浮执行器（Universal Context Agent）详细方案与路线图

## 1. 项目定位

### 1.1 一句话定义
一个运行在桌面系统之上的 **上下文感知型智能执行器**：当用户在文件、网页、Office、PDF、图片、链接、文本选区、剪贴板或当前窗口中产生明确上下文时，系统以轻量、快速、低打扰的方式提供操作入口；用户输入指令后，系统自动理解意图、抽取上下文、路由动作、执行任务，并在主控制台中统一呈现任务状态、执行过程、结果产物、失败原因与后续建议。

### 1.2 核心目标
1. **快**：识别快、弹出快、执行快、反馈快。
2. **灵活**：不能只有固定模板动作，必须支持自然语言命令与组合操作。
3. **不死板**：不能用户一选中文件就强行走固定流程，也不能一律变成大任务。
4. **低打扰**：存在感强，但不烦人，不乱弹，不遮挡，不抢焦点。
5. **可追踪**：所有命令、任务、执行状态、结果位置都可追踪。
6. **可扩展**：未来可接不同模型、CLI、插件、工具链。
7. **上下文统一**：不同来源内容必须抽象成统一上下文对象，保证动作层与执行层可复用。

### 1.3 面向的用户行为
用户并不是想“和 AI 聊天”，而是想：
- 对当前看到的内容立刻做事
- 不离开当前工作界面
- 不复制粘贴很多内容
- 快速触发分析、总结、生成、转换、写邮件、提取表格、生成报告等动作
- 在之后随时回到主程序查看最近做过什么、结果在哪里、是否执行成功

## 2. 产品原则

### 2.1 速度优先原则
系统整体设计必须遵循：
- 所有检测逻辑尽量事件驱动，而不是重轮询
- 轻动作必须尽可能本地快速完成
- 重动作必须先进入任务队列，再流式反馈
- 不能让用户等待长时间无反馈
- 不能识别半天才出现入口
- 不能点击后卡住没有状态

### 2.2 多入口并存原则
不能把产品能力完全押注在“像 Grammarly 一样跟随鼠标的小图标”上。必须同时支持：
- 跟随浮标
- 固定悬浮窗
- 右键菜单
- 快捷键唤起
- 浏览器扩展入口
- Office 内按钮/插件入口
- 拖拽到浮窗

这样即使部分应用不能精准检测选区，也能有兜底路径。

### 2.3 统一上下文原则
所有来源的内容必须转成统一结构，而不是让每个执行器单独理解输入。

### 2.4 任务分级原则
不是所有命令都要进入复杂 agent 工作流。
必须区分：
- 即时动作
- 中等动作
- 深度任务

### 2.5 显示优先原则
状态显示必须清楚：
- 当前识别到什么
- 将要执行什么
- 正在执行到哪一步
- 成功与失败怎么区分
- 结果在哪里
- 是否需要用户确认
- 是否支持撤销/重试/继续处理

## 3. 使用场景总览

### 3.1 文件场景
用户在资源管理器中选中文件或文件夹：
- 分析文件内容
- 生成摘要/报告
- 读取多个文件并比较
- 提取表格/图片/结构
- 生成邮件、PPT 大纲、任务清单
- 归档、重命名、分类

### 3.2 Office 场景
用户在 Word、Excel、PPT 中选中内容：
- 总结选区
- 改写内容
- 提取结论
- 生成图表
- 基于选区生成邮件
- 将选区发送到控制台做进一步分析

### 3.3 PDF 场景
用户在 PDF 阅读器中选中内容或指定文件：
- 抽取文本
- 摘要
- 提取章节结构
- 提取表格
- 生成阅读报告
- 转换为可编辑文档

### 3.4 网页场景
用户在网页中选中文本、链接、图片、表格、代码块：
- 总结网页
- 提取信息
- 保存为知识卡片
- 抓取内容生成报告
- 转成邮件/笔记
- 做竞品分析

### 3.5 图片/截图场景
用户选中图片、截图、剪贴板图像：
- OCR
- 内容理解
- 图表识别
- 表格识别
- 生成图片说明
- 标注重点

### 3.6 通用浮窗场景
用户主动打开悬浮窗：
- 分析当前剪贴板
- 读取最近上下文
- 输入命令处理最近选中的文件/文本
- 查看最近任务
- 打开报告输出目录
- 从主控制台继续处理

---

## 4. 总体架构

系统建议划分为七层：

1. **上下文感知层 Context Capture Layer**
2. **悬浮交互层 Overlay & Entry Layer**
3. **命令理解与意图路由层 Intent Router**
4. **任务编排层 Task Orchestration Layer**
5. **执行器层 Executors Layer**
6. **状态与产物中心 State / Artifact Center**
7. **主控制台 Console / Workspace Layer**

---

## 5. 上下文感知层设计

### 5.1 职责
负责识别用户当前所在环境、所选内容、当前可用动作来源，并快速产出统一上下文对象。

### 5.2 支持的上下文来源
1. 文件系统选择
2. 文件右键菜单
3. 文件拖拽
4. 剪贴板文本/图片/文件
5. 当前活动窗口信息
6. 浏览器选区、链接、图片、网页 DOM 元素
7. Office 选区
8. PDF 选区或指定文件
9. 屏幕截图/视觉兜底
10. 固定浮窗的手动输入

### 5.3 上下文对象统一结构
建议定义统一的 `ContextPacket`：

```json
{
  "context_id": "ctx_20260406_xxx",
  "source_type": "file | file_group | text_selection | image | link | webpage | clipboard | office_selection | pdf_selection | window | screenshot",
  "source_app": "Excel",
  "window_title": "Budget.xlsx - Excel",
  "capture_mode": "event | manual | plugin | extension | visual_fallback",
  "text": "选中的文本内容",
  "html": "可选网页HTML片段",
  "url": "https://example.com",
  "file_paths": ["C:/docs/test.pdf"],
  "image_paths": ["C:/temp/capture.png"],
  "selection_metadata": {
    "sheet_name": "Sheet1",
    "range": "A1:D20",
    "slide_index": 3,
    "page_index": 8,
    "mime_type": "application/pdf"
  },
  "entity_hints": {
    "contains_table": true,
    "contains_link": false,
    "contains_image": false,
    "possible_language": "zh-CN"
  },
  "captured_at": "2026-04-06T13:30:00-04:00"
}
```

### 5.4 上下文识别速度要求
目标设计：
- 文件选中识别：100–250ms 以内完成入口可见
- 剪贴板触发：100ms 以内给出状态
- 浏览器选区浮标：100–200ms 以内出现
- Office 插件选区：200–400ms
- 视觉兜底：允许 1–3 秒，但必须先显示“识别中”状态

### 5.5 检测策略
#### 文件系统
- 利用资源管理器右键、拖拽和选中文件事件
- 不建议强行监听所有路径变化来猜选中对象

#### 浏览器
- 用浏览器扩展获取选区、链接、DOM 目标和坐标
- 扩展与桌面主程序通信

#### Office
- 用 Office Add-ins 或宿主插件事件获取选区
- 不要依赖 OCR 去识别 Office 内容

#### PDF
- 优先文件级处理
- 选区级集成视 PDF 阅读器而定
- 不通用时走“复制选区到程序”或“发送文件到程序”

#### 视觉兜底
- 对无法读取选区的 app，支持截图识别
- 必须明确提示用户当前是视觉模式，可能不如原生选区精确

## 6. 悬浮交互层设计

### 6.1 交互入口类型
#### 6.1.1 跟随浮标
- 靠近选区或鼠标附近显示
- 尺寸小，默认只显示图标
- 支持 hover 预览动作，点击后展开
- 不能遮挡主要内容

#### 6.1.2 固定悬浮窗
- 固定在屏幕边缘
- 可折叠/展开
- 用于兜底唤起、历史查看、任务总览
- 可拖拽位置

#### 6.1.3 右键菜单
- 文件右键
- 选中文本右键（浏览器/支持的应用）
- 图片右键
- 链接右键

#### 6.1.4 快捷键入口
- 全局快捷键打开命令框
- 快捷读取当前剪贴板/最近上下文

### 6.2 悬浮 UI 的状态
1. **未出现**：当前上下文不满足显示规则
2. **准备出现**：识别到上下文，判断是否显示
3. **轻浮标状态**：只显示小图标
4. **展开状态**：显示快捷动作+输入框
5. **处理中状态**：已提交任务
6. **结果状态**：显示成功/失败/继续操作

### 6.3 显示原则
- 不在每次鼠标移动时乱闪
- 只在“上下文稳定”后出现
- 显示需要去抖与节流
- 可设定“只在选中内容时出现”
- 可设定“悬停模式”作为高级选项而非默认开启

### 6.4 弹出内容建议
点击浮标展开后包含：
1. 当前识别到的对象摘要
2. 快捷动作区
3. 自定义命令输入框
4. 是否需要确认的开关
5. 最近任务入口
6. “发送到主控制台继续处理”按钮

### 6.5 快捷动作模板
#### 面向文件
- 分析文件
- 生成摘要
- 生成报告
- 提取表格
- 写邮件
- 比较文件

#### 面向文本
- 总结
- 改写
- 翻译
- 提取待办
- 解释

#### 面向链接/网页
- 抓取并总结
- 保存为笔记
- 生成汇报
- 提取关键信息

#### 面向图片
- OCR
- 图像理解
- 提取图表
- 生成描述

### 6.6 低打扰机制
- 同一对象短时间内不重复弹出
- 用户关闭一次后，对该对象本轮不再弹出
- 支持“敏感窗口禁用”
- 支持“全屏场景禁用”
- 支持“会议/演示模式静默”

## 7. 命令理解与意图路由层

### 7.1 核心职责
将“用户输入 + 上下文对象”转化为系统真正可执行的标准动作。

### 7.2 路由前必须判断的维度
#### 7.2.1 动作复杂度
- 轻动作：几秒内可完成
- 中动作：10–30秒内完成
- 重动作：需要多步处理、文件生成、CLI 执行

#### 7.2.2 输出类型
- 文本输出
- 邮件草稿
- Markdown 报告
- Word/Excel/PPT/PDF 产物
- 图表
- 结构化数据
- 文件改动

#### 7.2.3 上下文粒度
- 单段文本
- 单文件
- 多文件
- 当前窗口
- 网页链接
- 截图图像
- 表格区域

#### 7.2.4 风险与确认需求
- 只读动作
- 会写入文件的动作
- 会发送内容的动作
- 会调用外部程序的动作
- 会覆盖原内容的动作

### 7.3 意图类型建议
系统内部不要只靠自然语言直接调执行器，建议先归类成动作类型：
- `summarize`
- `analyze`
- `rewrite`
- `translate`
- `extract_table`
- `extract_entities`
- `generate_report`
- `generate_email`
- `compare_sources`
- `create_presentation_outline`
- `convert_format`
- `classify_files`
- `send_to_console`
- `launch_deep_workflow`

### 7.4 路由决策输出结构

```json
{
  "route_id": "route_xxx",
  "intent": "generate_report",
  "complexity": "heavy",
  "executor_type": "kimi_code_cli",
  "requires_confirmation": true,
  "estimated_mode": "queued_task",
  "output_schema": {
    "type": "markdown_report",
    "save_required": true
  },
  "reasoning_summary": "选中对象为 PDF 文件，用户要求分析完整内容并生成正式报告，因此进入重任务流程"
}
```

### 7.5 灵活性设计
为了不死板，路由层要支持：
- 快捷动作 + 自然语言补充
- 一次命令里组合多个动作
  - 例如："分析这个文件，并给我一封发给导师的邮件"
- 模糊命令智能澄清
  - 例如："帮我处理一下这个表"
- 上下文补全
  - 例如：用户只说“帮我写封邮件”，系统默认引用最近一个文件或当前选区，但必须提示引用了什么

### 7.6 问答与澄清机制
系统必须支持两类确认：
#### 轻确认
直接在浮窗一句话追问：
- “你是想总结当前选区，还是整份文件？”
- “输出成邮件草稿还是 Markdown？”

#### 重确认
当涉及重任务时：
- 显示任务摘要
- 显示输入来源
- 显示预计输出类型
- 让用户确认再执行

### 7.7 默认确认策略
- 只读轻动作：默认免确认
- 文件生成类：轻确认或可配置自动执行
- 修改原文件：必须确认
- 发送邮件/对外动作：必须确认
- 多文件批量处理：建议确认

## 8. 任务编排层设计

### 8.1 职责
把路由后的动作变成可执行的任务对象，交给合适执行器，并管理状态、重试、取消、依赖与结果归档。

### 8.2 任务模型
建议统一任务对象：

```json
{
  "task_id": "TASK-20260406-0001",
  "created_at": "2026-04-06T13:40:00-04:00",
  "source_context_id": "ctx_xxx",
  "user_command": "分析这个文件并生成报告",
  "intent": "generate_report",
  "priority": "normal",
  "execution_mode": "instant | queued | background_streaming",
  "executor": "kimi_code_cli",
  "status": "queued",
  "steps": [],
  "artifacts": [],
  "logs": [],
  "requires_confirmation": true,
  "confirmed": true
}
```

### 8.3 任务分级
#### A 类：即时任务
- 直接返回结果
- 例如短文本总结、翻译
- 在浮窗内展示即可

#### B 类：半即时任务
- 进入小任务队列
- 10–30秒
- 例如网页抓取后总结、单文件结构提取

#### C 类：深度任务
- 进入正式任务队列
- 显示进度、步骤、日志、产物
- 例如生成完整报告、跨文件分析、批量处理

### 8.4 任务生命周期
建议统一状态流：
- `draft`
- `awaiting_confirmation`
- `queued`
- `starting`
- `running`
- `streaming`
- `success`
- `partial_success`
- `failed`
- `cancelled`
- `waiting_user_input`
- `needs_review`

### 8.5 多任务机制
必须支持多任务并存：
- 浮窗里显示当前任务数量
- 主控制台显示任务列表
- 任务可并发，但需有并发上限
- 同类型重任务可做排队
- 支持“最近任务继续处理”

### 8.6 并发策略
建议：
- 轻任务：可高并发
- 中任务：中等并发
- Kimi Code CLI 类深度任务：低并发 + 排队
- 对同一来源对象的重复命令可提示复用或重跑

### 8.7 任务取消与中断
必须支持：
- 未开始任务：直接取消
- 运行中任务：请求中断
- 中断后保留部分日志
- 明确显示“已取消”与“失败”不同

## 9. 执行器层设计

### 9.1 执行器分类
#### 9.1.1 快速执行器 Fast Executor
适合：
- 总结
- 改写
- 翻译
- 提取要点
- 基础问答

#### 9.1.2 工具执行器 Tool Executor
适合：
- 文件解析
- OCR
- 表格抽取
- 网页抓取
- Office 文档读写
- 文件格式转换

#### 9.1.3 Kimi Code CLI 执行器
适合：
- 深度分析
- 报告生成
- 多步工作流
- 文件产物生成
- 跨工具调用
- 结构化处理

### 9.2 Kimi Code CLI 的正确职责
Kimi 不应负责：
- 系统级 UI 感知
- 鼠标跟踪
- 浮窗渲染
- 入口是否显示判断

Kimi 负责：
- 读取任务包
- 根据上下文执行多步任务
- 调用本地工具
- 生成产物
- 输出执行日志
- 返回结果路径与总结

### 9.3 任务包机制
建议所有深任务都生成标准任务包给 Kimi：

```json
{
  "task_id": "TASK-20260406-0020",
  "task_type": "report_generation",
  "user_command": "分析这个文件内容，生成一份详细报告，并给出关键风险总结",
  "context": {
    "source_type": "file",
    "file_paths": ["C:/docs/sample.pdf"],
    "metadata": {
      "mime_type": "application/pdf"
    }
  },
  "output_requirements": {
    "primary": "markdown_report",
    "save_required": true,
    "output_dir": "C:/ProgramData/UCA/outputs/2026-04-06/"
  },
  "rules": {
    "must_read_source": true,
    "must_save_result": true,
    "must_return_artifact_paths": true,
    "must_emit_progress": true
  }
}
```

### 9.4 执行反馈协议
执行器必须在运行中返回结构化事件：
- `accepted`
- `started`
- `step_started`
- `step_finished`
- `log`
- `warning`
- `needs_input`
- `artifact_created`
- `success`
- `failed`

这能保证 UI 做出清晰状态展示。

## 10. 状态显示与执行显示设计

### 10.1 为什么状态显示是核心功能
这个产品不是一次性聊天工具，而是“执行工具”。执行工具必须让用户看得见：
- 系统识别到了什么
- 接下来打算做什么
- 当前进行到哪一步
- 是否成功
- 如果失败为什么失败
- 成果文件在哪里

### 10.2 浮窗中的状态显示
浮窗应提供轻量但完整的状态：
- 已识别对象："已识别 1 个 PDF 文件"
- 已接收命令："正在准备生成报告"
- 当前状态：排队中 / 执行中 / 等待确认 / 已完成 / 失败
- 最近结果："报告已生成，点击打开"

### 10.3 主控制台中的任务列表
建议字段：
- 任务名称
- 来源对象
- 命令摘要
- 当前状态
- 执行器
- 开始时间
- 持续时间
- 最近更新
- 结果数量
- 是否需要处理

### 10.4 任务详情页
任务详情需要包含：
1. 命令原文
2. 解析后的意图
3. 来源上下文
4. 执行步骤时间线
5. 日志流
6. 产物列表
7. 失败信息
8. 重试/继续处理入口

### 10.5 状态颜色与语义建议
- 蓝色：准备中 / 排队中 / 处理中
- 绿色：成功
- 黄色：等待用户、部分成功、提醒
- 红色：失败
- 灰色：取消/过期

### 10.6 成功后的显示
成功时必须清楚显示：
- 成功摘要
- 结果文件路径
- 打开文件
- 打开目录
- 在主控制台中继续处理
- 基于结果再执行下一步

### 10.7 失败后的显示
失败时不能只显示“失败”。
必须分层展示：
1. 失败在哪一步
2. 原因类别
3. 原始错误摘要
4. 用户可采取的动作
5. 是否支持一键重试

建议失败分类：
- 上下文读取失败
- 文件权限失败
- 解析失败
- 外部工具不可用
- CLI 执行失败
- 模型调用失败
- 输出保存失败
- 用户中断
- 网络异常

### 10.8 部分成功状态
很多任务可能会“部分完成”。
例如：
- 报告生成成功，但表格提取失败
- 总结完成，但邮件草稿未生成

这时状态应为 `partial_success`，并明确哪些完成，哪些失败。

### 10.9 流式执行显示
对中重任务，建议流式显示：
- 步骤开始
- 步骤日志
- 产物创建事件
- 当前处理阶段

示例：
1. 读取文件
2. 解析正文
3. 提取章节
4. 生成摘要
5. 写入报告
6. 报告已保存

## 11. 问答、确认与人机交互完整流程

### 11.1 用户交互不应过于死板
系统不能只有“输入命令 → 执行”。
必须支持：
- 追问
- 确认
- 补充上下文
- 推荐动作
- 中途要求用户决定输出格式

### 11.2 交互流程模板
#### 流程 A：简单即时任务
1. 用户选中文本
2. 浮标出现
3. 用户点击“总结”
4. 系统直接执行
5. 浮窗显示结果
6. 可继续“改写/发邮件/发送到控制台”

#### 流程 B：需要确认的重任务
1. 用户选中文件
2. 输入“分析并生成报告”
3. 系统展示：
   - 输入来源：sample.pdf
   - 输出：Markdown 报告
   - 执行器：Kimi Code CLI
4. 用户点击确认
5. 任务入队
6. 浮窗显示开始执行
7. 主控制台显示完整过程
8. 成功后可打开报告

#### 流程 C：信息不足需追问
1. 用户选中 Excel 区域
2. 输入“帮我处理这个表”
3. 系统判断意图不清
4. 提供选项：
   - 总结数据
   - 生成图表
   - 检查异常值
   - 写成汇报
5. 用户选择后继续执行

#### 流程 D：执行中需补充输入
1. 用户要求“基于这个文件写邮件”
2. 系统执行到邮件阶段
3. 发现缺少收件人或语气
4. 提示用户补充
5. 用户输入后继续执行

### 11.3 确认界面应显示的信息
- 将处理什么对象
- 将执行什么动作
- 是否会生成文件
- 是否会修改原对象
- 结果将保存在哪里
- 是否会调用外部工具

### 11.4 智能推荐动作
识别上下文后，系统应动态推荐：
- 选中文本 → 总结 / 改写 / 翻译
- 选中 PDF → 生成报告 / 提取结构 / 写邮件
- 选中表格 → 总结数据 / 生成图表 / 导出分析
- 选中链接 → 抓取内容 / 保存为笔记 / 做摘要

## 12. 多任务显示与任务中心设计

### 12.1 为什么必须有任务中心
因为用户会在多个界面连续触发很多命令：
- 一个 PDF 生成报告
- 一个网页抓取摘要
- 一个 Excel 生成图表

如果没有任务中心，用户会不知道哪些已经完成、哪些失败、结果在哪。

### 12.2 任务中心页面结构
#### A. 顶部概览
- 正在运行数量
- 排队数量
- 成功数量
- 失败数量
- 最近产物数量

#### B. 中间任务列表
每行显示：
- 任务标题
- 来源图标
- 命令摘要
- 当前状态
- 进度简述
- 最后更新时间
- 打开详情

#### C. 右侧/下方任务详情
- 步骤流
- 日志
- 产物
- 失败原因
- 重试
- 继续处理

### 12.3 任务聚合
支持按以下方式筛选：
- 今日任务
- 按来源应用
- 按状态
- 按输出类型
- 按执行器

### 12.4 最近结果中心
单独提供“最近产物”：
- 报告
- 邮件草稿
- 导出的表格
- 图片识别结果
- 生成文档

## 13. 成功/失败/异常的完整处理逻辑

### 13.1 成功处理
成功后应自动提供：
- 结果预览
- 打开文件
- 打开目录
- 复制路径
- 基于该结果继续生成新动作

### 13.2 失败处理
失败时不能直接终止体验。
必须提供：
- 明确失败原因
- 失败发生阶段
- 原始日志入口
- 修复建议
- 重新执行按钮
- 切换执行模式选项

### 13.3 重试机制
支持：
- 原参数重试
- 修改参数后重试
- 改用其它执行器重试
- 跳过已完成步骤继续重试

### 13.4 降级机制
系统必须有降级策略：
- 原生选区读取失败 → 建议剪贴板模式
- CLI 失败 → 尝试基础执行器输出简版结果
- 深任务失败 → 先给部分中间结果
- 网页抓取失败 → 提示仅基于选中片段处理

### 13.5 用户中断机制
用户可以：
- 取消任务
- 暂停查看日志
- 关闭浮窗但不中断任务
- 关闭主窗口后后台继续执行（可配置）

## 14. 主控制台与悬浮程序的关系

### 14.1 角色划分
#### 悬浮程序
负责：
- 快速感知上下文
- 快速提供入口
- 发起命令
- 展示轻状态
- 作为低打扰操作层

#### 主控制台
负责：
- 任务中心
- 深度日志
- 产物管理
- 多任务并发视图
- 高级配置
- 重试与工作流扩展

### 14.2 流程协同
- 轻任务在浮窗内完成
- 中重任务在浮窗发起，在主控制台展开
- 用户可以从浮窗随时“在控制台打开”
- 控制台可反向显示“来自哪个浮窗上下文的任务”

## 15. 快速、灵活、不死板的关键机制

### 15.1 快速的机制
1. 事件驱动检测，而非高频扫描
2. 快速路径优先：轻任务先走轻执行器
3. 缓存最近上下文
4. 常见动作本地预生成模板
5. 日志与结果流式回传
6. UI 先响应，再补执行细节

### 15.2 灵活的机制
1. 支持自然语言命令
2. 支持快捷动作组合
3. 支持多来源统一处理
4. 支持同一结果继续派生动作
5. 支持改用其它执行器
6. 支持任务模板与用户自定义动作

### 15.3 不死板的机制
1. 不强制所有命令进入大工作流
2. 不把所有请求都变成 Master Plan 或复杂任务
3. 根据用户意图自动分流到最短路径
4. 出现歧义时轻追问，不要机械拒绝
5. 支持“先给草稿，再深度处理”模式

## 19. 风险与解决建议

### 19.1 风险：过度追求全场景原生检测
解决：先做多入口并存，不把成功率压在“任意位置悬浮跟随”上。

### 19.2 风险：执行逻辑过重导致响应慢
解决：轻任务与重任务严格分流，UI 先响应，结果后补。

### 19.3 风险：Kimi CLI 直接接自然语言导致不稳定
解决：统一任务包，限制输入结构，返回标准事件。

### 19.4 风险：失败黑箱
解决：步骤化事件流 + 失败分类 + 重试入口。

### 19.5 风险：用户觉得打扰
解决：低打扰策略、显示规则、静默模式、敏感窗口禁用。

## 20. 最终建议

这个产品应该被定义为：

**“一个系统级、上下文驱动、低打扰、高可追踪的智能执行入口。”**

它不是单纯悬浮按钮，也不是单纯聊天框，而是：
- 在用户当前工作位置附近提供入口
- 快速识别上下文
- 快速理解动作
- 快速执行或转入任务流
- 在浮窗与主控制台之间形成完整闭环

最关键的成功标准不是“会不会弹出来”，而是：
1. 弹得准不准
2. 动作快不快
3. 流程顺不顺
4. 失败清不清楚
5. 结果找不找得到
6. 能不能继续下一步

如果以上 6 点做好，这个产品就会非常强。
